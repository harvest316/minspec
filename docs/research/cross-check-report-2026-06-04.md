## Inter-spec themes

This section is the **requirements seed** for inter-spec cross-check (MinSpec Slice 2 AI checks). Each theme notes whether a **deterministic Slice-1 check** could catch it or whether it needs the **LLM reviewer (Slice 2)**.

1. **Dangling cross-artifact references (DR/SPEC/file).** DR-355 (5 sites, 2 specs), the SPEC-001 research-doc path, SPEC-014 stale `approve.ts:70-100` line citations, DR-001 cross-repo SPEC-100/101/102. **→ Slice 1 (deterministic).** A reference-resolver can: (a) assert every `DR-NNN` cited resolves to a `docs/decisions/DR-NNN.md` that exists; (b) resolve relative file links; (c) for `file:line` citations, optionally verify the line still contains the referenced symbol. Cross-repo refs need a declared external-namespace convention (e.g. `SPEC-100@scroogellm`) so the checker can *exempt* rather than flag them. This is the highest-ROI deterministic check.

2. **Status-claim lies (frontmatter vs code/counts).** SPEC-016 (`implementing`, no package), SPEC-011 (476/15 → 478/16 + prose contradiction), SPEC-015 (91 → 96 tests; AC names wrong file set), SPEC-004 & SPEC-015 design.md lag, DR-023/DR-029 unimplemented-claim. **→ Mostly Slice 1, partly Slice 2.** Deterministic: frontmatter-vs-prose status mismatch within one file; numeric "N lines / N tests" comments vs `wc -l` / `it(` counts; `design.md status` vs `requirements.md status` within a spec dir. **Needs Slice 2 (LLM):** "this spec says `implementing` but the *feature* has no code" requires mapping a spec to its implementing symbols — the DR-003 artifact-existence≠feature-existence rule. A deterministic proxy (does `packages/<expected>` exist?) catches the egregious cases (SPEC-016); subtle "implemented but wrong" needs the reviewer.

3. **The phantom `cross-checks` phase (propagation gap).** DR-029 introduced it; it is missing from `config.ts` Phase enum + PHASES + T3/T4 phaseMappings, the SPEC-015 lane map, and is referenced unguarded by SPEC-013/016. **→ Slice 1 for the leaf checks, Slice 2 for the cross-artifact invariant.** Deterministic: assert the Phase enum, PHASES array, the lane map, and tier phaseMappings agree on one canonical phase set. **Slice 2:** detecting that a *DR* introduced a methodology change and tracing which specs/code must propagate it is a reasoning task. This single theme spans 6 findings and one untracked DR-029 follow-up — the strongest argument that inter-spec coverage needs to exist at all.

4. **Validator asymmetry (the recurring class).** SPEC-017 `superseded-by` required-when-superseded but unenforced; SPEC-006/013/012 rules in `validateSpec()` with undocumented precedence. **→ Slice 1 (deterministic), once the symmetric primitive (#137) lands.** A `requiredWhen(condition)` validator combinator catches present-but-missing conditional fields. Rule-ordering/aggregation across specs is a DR-level documentation gap, not a check — surface it as a "multiple specs mutate one shared function with no documented order" Slice-2 advisory.

5. **Ownership / boundary ambiguity (composition without a contract).** SPEC-012↔SPEC-010 predicate ownership; SPEC-013↔SPEC-010 L1-checker home (shared vs minspec); SPEC-011→SPEC-016 documented reuse. **→ Slice 2 (LLM reviewer).** "Spec A says it composes B and must not re-implement B's predicates" is a semantic claim about code structure — deterministic tooling can verify a *function isn't duplicated by name*, but "these two predicate *sets* don't overlap in meaning" needs the reviewer. The reuse cases (SPEC-011→016) are healthy; the reviewer's job is distinguishing documented reuse from silent duplication.

6. **Core invariant vs subsystem behavior (scoped contradictions).** SPEC-001 Invariant 5 "no gate blocks" vs SPEC-019 FR-12 "T3-T4 blocked pending approval." **→ Slice 2 (LLM reviewer).** This is a *false* contradiction once tier/system scope is applied (Tier-0 core never blocks authoring; Tier-1 agent-execute may block dispatch). A deterministic check would mis-fire; the reviewer must reason about whether two "blocking" claims operate on the same object. Good negative-example seed: the reviewer must *not* flag scoped-distinct invariants.

7. **Untracked follow-ups (DR-023 materialization leaks).** DR-029 cross-checks propagation (no issue), EPIC-007 Slice 2-3 spec (conditional), DR-023 retrofit record, 8-of-30 DRs lacking a Follow-ups section. **→ Slice 1 (deterministic).** A DR-frontmatter rule asserting a `## Follow-ups (tracked)` section exists and that each item links a `SPEC-NNN` or `#issue` (vs prose-only) is exactly the DR-023 forward rule, mechanizable as a soft validator — which DR-023 itself claims exists but does not.
