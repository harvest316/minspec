# DR-023: Materialize DR follow-ups (no orphan consequences)

## Context

A DR is a **decision record**, not a work-tracker. Its *Consequences*, *new work
surfaced*, and *sequenced refactor* lists are **inert prose** — nothing converts
them into tracked issues or specs. The decided work then depends on a human/agent
*remembering* to act on it.

This leaks worst for **cross-repo or non-code follow-ups**, which never enter the
SDD spec flow. The concrete trigger: DR-022's **positioning** consequence (site +
marketplace copy) would have evaporated — it got tracked only because it was noticed
by hand and filed as [#86](https://github.com/harvest316/minspec/issues/86).

The existing Traceability Convention links **issue → DR** (backward: "this issue
references its DR"). It has **no forward rule** that a DR must *spawn* the trackable
work it decides. That asymmetry is the gap.

## Decision

**Every DR must materialize its surfaced work as tracked artifacts — a `SPEC-NNN`
or a GitHub issue `#` — at DR-creation time. Prose-only follow-ups are not allowed
to stand.**

1. **`## Follow-ups (tracked)` section.** Every DR gains one. Each unit of work the
   DR surfaces is a line linking either a spec id or an issue number. "None" is a
   valid, explicit answer (forces the author to confirm there is no orphan work).
2. **Forward materialization at creation.** Any follow-up not already covered by a
   spec is filed as a GitHub issue when the DR is written. The **Create ADR** flow
   prompts for follow-ups and offers to file the issues.
3. **Soft validator rule.** A DR whose follow-up items lack a spec/issue ref →
   **warning** (advisory, never block — only DR-012 approval gates; mirrors the soft
   `epic:` rule, DR-013 §4).
4. **Not every consequence is work.** The rule covers *actionable* follow-ups
   (things someone must build/change), not every descriptive consequence. Judgement
   stays with the author; the section is where it is recorded.

This makes the traceability chain bidirectional: **DR → issue/spec** (forward,
materialization) as well as **issue → DR** (backward, rationale).

## Consequences

**Positive:**

- **ADR template** gains the `Follow-ups (tracked)` section (Create ADR command), so
  every DR author is forced to confirm orphan work either exists (and is filed) or is
  explicitly "None".
- **Validator** gains a soft rule for DRs with un-materialized follow-ups — rides on
  the SPEC-013 validator work, no new validator infra needed.
- **Existing DRs** are not retrofitted en masse — offered, not forced, which avoids a
  churn wave across the whole `docs/decisions/` register.
- **DR-022 retrofitted** as the first instance (its positioning follow-up → #86),
  proving the rule on the exact case that triggered it.
- Traceability chain becomes bidirectional — `DR → issue/spec` (forward) now matches
  the existing `issue → DR` (backward) edge, closing the asymmetry named in Context.

**Negative:**

- Slightly more ceremony at DR creation (the author must file the issues for actionable
  follow-ups) — accepted because the cost is paid once, at decision time, instead of
  lost as orphan prose (the failure mode DR-022's #86 nearly hit).
- The soft validator rule (Decision §3) only checks that the `Follow-ups (tracked)`
  section is *present*, not that its contents are honest — "None" box-ticking (R1)
  remains possible and is an accepted residual leaning on human DR review.

## Alternatives considered

- **Rely on human/agent discipline** (status quo). Rejected — that *is* the leak;
  the positioning work nearly vanished under it.
- **Auto-create an issue for every Consequence bullet.** Rejected — noise; most
  consequences are descriptive, not actionable. Author selects actionable items.
- **Hard-block DRs without materialized follow-ups.** Rejected — only DR-012 blocks;
  this stays advisory like every other MinSpec check.
- **Use a project board / milestone instead of issues.** Rejected — heavier, a
  second source of truth; issues reuse the existing label/dispatch machinery.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | **Box-ticking** — author writes "None" to dodge the section. | Med · Med | Soft-warn + DR review (the human approving the DR challenges an empty Follow-ups on a DR that obviously surfaced work). Not tooling-solvable alone; accepted residual. |
| R2 | **Issue spam** — over-filing turns every consequence into a stale issue. | Med · Med | Decision §4: only *actionable* follow-ups; author curates. The validator checks presence of the section, not a minimum count. |
| R3 | **Stale links** — a filed issue is closed/renumbered, leaving a dangling ref. | Low · Low | Same dangling-ref handling as other cross-artifact edges (corruption detection, cf. SPEC-012 FR-15); soft-warn, not block. |

## Costly to Refactor

Cheap to reverse — **undoable in well under a day** (passes the ADR filter as a
low-blast-radius process rule). The decision is a convention plus a *soft* validator
rule (Decision §3, never blocks), so dropping it means deleting the `Follow-ups
(tracked)` template line from the Create ADR flow and the advisory check riding on the
SPEC-013 validator. The one lingering artifact is the section itself, already written
into existing DRs (e.g. this DR's `#87`, DR-022's `#86`); those `Follow-ups` blocks
would become harmless dead prose rather than needing a migration — they were never
load-bearing data. No schema, no stored state, no cross-repo contract is anchored to
this rule, so blast radius is limited to the ADR template and one validator warning.
