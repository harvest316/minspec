# DR-019: Deterministic next-task resolution (DAG, not LLM)

## Context

A session asked for "a prioritised list of docs / specs / epics / DRs I need to
approve", then reframed: not a list — **the single next task** the human dev must
do, with an optional expansion to sense the pipeline. The follow-up question was
the crux: **can priority be reliably assessed by programmatic means (a DAG)
instead of an LLM?**

Three approval/status mechanisms feed "what must I decide next":

- **specs** — DR-012 content-hash approval (approve before implement),
- **epics** — `status: proposed → active` promotion,
- **ADRs** — `status: proposed → accepted`.

Plus [SPEC-010](../../specs/minspec/SPEC-010-signpost-correctness/requirements.md)'s
within-feature phase signpost, whose Open Question #1 already asked how to order
*globally* when several items pend at once.

The signpost is EPIC-002's core promise: MinSpec is *always opinionated about
what happens next*, and **never wrong**. An LLM-derived next-task cannot meet
"never wrong" — it is non-reproducible, untestable, and would make the product's
central pointer a probabilistic guess.

## Decision

**Next-task priority is computed deterministically — a pure function over a
dependency DAG of structural state — and never by an LLM.**

### 1. All inputs are structural

Epic status + `order`, spec status/tier/phase + approval-hash, ADR status, and
the membership edges (`spec.epic`, `adr.epic`) plus the fixed phase DAG
(`specify→clarify→plan→tasks→implement`) and the DR-012 approve→implement gate.
Nothing requires semantic understanding.

### 2. Total order = (severity class, epic.order, artifact-id)

Severity classes, highest first: **gate-violation** (downstream work past an
unsatisfied gate) ▸ **blocked-ready** (gate whose clearance unblocks the next
phase under an active epic) ▸ **promote-parent** (proposed epic with waiting
members) ▸ **pending** (remaining proposed/unapproved). The **next task** is the
minimum of this total order. Fully topological + deterministic; unit-testable.

### 3. Subjective weight lives in explicit fields, not inference

The one thing a DAG cannot derive — relative importance between *independent*
branches (e.g. "ScroogeLLM is future") — MUST be lifted into explicit frontmatter
(`epic.order`), edited by the human. Prose in CLAUDE.md is invisible to the engine
until lifted. The engine computes structure; the human sets weight; **the LLM
infers neither at resolve time.**

**Deferral is a link, not a boolean.** "Not now" is expressed as a dependency on
the artifact that unblocks it (`depends_on`, decision §7), never a standalone
`deferred: true` flag — a boolean rots (set once, never unset, item stays hidden
after its real blocker clears). A link auto-clears, is explainable ("hidden
because blocked on EPIC-003"), and reuses one mechanism. A deferred node is thus
just a node with an un-cleared dependency, which the DAG already floors below its
blocker — no separate deferred-priority rule.

### 4. Two queues stay separate

The next task is the **human's** decision queue (approve / promote / accept /
author). The **agent/LLM dispatch** queue (DR-015/017) is a separate substrate
and never appears as a human next task.

### 5. Coherence is a deterministic precondition

A child must not outrank its parent's status (spec `implementing` under a
`proposed` epic; ADR `accepted` under a `proposed` epic). Such breaches are
detected structurally and surface as the top gate-violation next task — the same
class that caught, by hand, the two live inconsistencies in the triggering
session (stale epic INDEX; SPEC-004 implementing under proposed EPIC-004).

### 6. Tier 0, one engine

A single pure function in `packages/shared` (no `vscode`, no network; DR-004,
DR-014), consumed by the status-bar signpost, the explorer rollup, and CI alike —
so every surface agrees on the one next task.

### 7. Cross-cutting dependencies are machine-readable edges

The SDD tree (phase chain, epic membership) is only the *implicit* dependency
structure. Real dependencies cut across it — "this spec is blocked by that ADR",
"this DR depends on those DRs", "this spec supersedes that one" — and today live
only in prose (`Triggered by:`, `Resolves:`), invisible to the engine. They become
explicit frontmatter edges (`depends_on: [ID, …]`, `supersedes:`). The union of
implicit + explicit edges is the dependency DAG the resolver ranks over. The graph
**MUST be acyclic** — a cycle is corruption, and acyclicity is itself a free
correctness check.

### 8. Corruption: structural-only, deterministic-repair-first

Correctness is **reliably assessable for structure** (malformed frontmatter,
dangling refs, status-incoherence, dependency cycles) — and **not** for semantics
(well-formed but wrong content has no programmatic oracle). On detected structural
corruption the response is a ladder, never a silent fix: deterministic programmatic
repair offered first (confirm-before-write); LLM-escalation offered only when no
deterministic repair applies (confirm-before-write, bounded per DR-355). Composes
SPEC-005 (offer-never-silent); the LLM never decides *whether* corruption exists,
only *proposes a fix*.

## Consequences

**Positive:**

- The core signpost becomes reproducible and unit-testable — the same DAG state
  always yields the same next task, satisfying EPIC-002 "never wrong" where an LLM
  ranker (rejected, Alternatives) could not.
- `depends_on:` / `supersedes:` frontmatter edges (decision §7) are backward
  compatible — absent = no cross-cutting edge, so existing epics/specs/ADRs need no
  migration.
- One pure function in `packages/shared` (decision §6) is consumed by the status-bar
  signpost, explorer rollup, and CI alike, so every surface agrees on the one next
  task. Resolves [SPEC-010](../../specs/minspec/SPEC-010-signpost-correctness/requirements.md)
  Open Question #1 (global ordering when several items pend).
- Deferral as a `depends_on` link (decision §3) auto-clears and is explainable
  ("hidden because blocked on EPIC-003"), avoiding the rotting `deferred: true`
  boolean (rejected, Alternatives).
- The explorer pane gains a single next-task signpost + collapsible pipeline rather
  than the rejected dedicated "Approvals list" view — collapsing a backlog to one
  pointer is the product.

**Negative:**

- New spec [SPEC-012 Next-Task Resolver](../../specs/minspec/SPEC-012-next-task-resolver/requirements.md)
  must be authored and carry the contract — net-new SDD work, not a free byproduct.
- Subjective branch weight must now be lifted into explicit `epic.order` frontmatter
  (decision §3) and hand-maintained by the human; prose in CLAUDE.md ("ScroogeLLM is
  future") stays invisible to the engine until lifted, so a forgotten lift silently
  mis-ranks.
- May restructure the existing Specs/Decisions explorer panes (downstream UX spec) —
  UI churn beyond the engine.
- The LLM's role shrinks to *suggesting* `order` / `depends_on` values for the human
  to accept — it never computes the live next task, so every cross-cutting edge needs
  a human accept step before it influences priority.

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| Human forgets to lift branch weight into `epic.order`, so independent branches mis-rank | Decision §3 — prose weight is invisible until lifted to frontmatter | No deterministic rescue exists (a forgotten lift yields a valid-but-stale `epic.order`, not a detectable error); residual risk accepted. The LLM *suggesting* `order` values for the human to accept (Consequences / decision §3) is the only surfacing mechanism — a soft prompt, not a gate |
| A `depends_on:` / `supersedes:` edge introduces a cycle, corrupting the DAG | Decision §7 — "The graph MUST be acyclic" | Acyclicity is itself a free correctness check (§7); a cycle surfaces as structural corruption handled by the deterministic-repair ladder (§8) |
| Status-incoherence (child outranks proposed parent) silently skews the order | Decision §5 — e.g. SPEC-004 `implementing` under proposed EPIC-004, caught by hand in the triggering session | Detected structurally and surfaced as the top **gate-violation** class (§2, §5), not left latent |
| Engine drifts from `packages/shared` Tier-0 constraints (pulls in `vscode`/network) | Decision §6 — single pure function, DR-004 Tier-0, DR-014 | One engine consumed by status bar + explorer + CI; a non-pure dependency breaks the Tier-0 boundary tests (DR-004/DR-014) |
| Deterministic-only corruption handling misses semantically-wrong-but-well-formed content | Decision §8 — "not assessable for semantics … no programmatic oracle" | Scoped honestly to structure only; LLM-escalation offered (confirm-before-write, bounded per DR-355) when no deterministic repair applies, never a silent fix |

## Costly to Refactor

Moderately costly, and deliberately so. The frontmatter contract — `depends_on:` /
`supersedes:` edges and `epic.order` (decisions §3, §7) — once authored across
epics/specs/ADRs becomes data embedded in every artifact; replacing the ranking model
(e.g. reverting to an LLM ranker, rejected in Alternatives) would orphan that data and
break the SPEC-012 contract plus the three consumers (status bar, explorer, CI) of the
single `packages/shared` function (§6). The *engine* itself is cheap to swap — it is one
pure function. **ADR-filter: NO, not undoable in <1 day** once the edges are populated
and SPEC-012 ships, because the cost is the propagated frontmatter and the multi-surface
contract, not the function body. Before adoption (no edges authored yet) it is trivially
reversible.

## Alternatives considered

- **LLM ranks pending items** — rejected. Non-reproducible, untestable, violates
  EPIC-002 "never wrong" and DR-004 Tier-0. A probabilistic core signpost is a
  trust liability.
- **Render a flat prioritised list (no single next task)** — rejected as the
  primary surface. A list is a backlog; collapsing it to one pointer is the
  product. List survives only as an opt-in expansion (SPEC-012 FR-6).
- **Dedicated unified "Approvals" tree view** — rejected in-session as too
  list-heavy; superseded by the single-next-task signpost direction.
- **Infer branch priority from CLAUDE.md prose / edit recency** — rejected as
  hidden, non-deterministic weight. Weight must be explicit data (decision §3).
- **`deferred: true` boolean flag** — rejected. A bare boolean rots (set once,
  never unset → item stays hidden after its blocker clears), carries no reason, and
  is a second mechanism beside `depends_on`. Deferral is a link to the blocker
  instead (decision §3) — auto-clearing, explainable, one mechanism.
