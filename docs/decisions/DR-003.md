> **Note:** Originally named *Bug Session Protocol*. Renamed to **RCDD (Root-Cause-Driven Debugging)** on 2026-05-29 to join the SDD/CDD/TDD acronym family and to surface the differentiator (root-cause discipline, not just "bug session"). Behaviour and gates are unchanged.

## Context

Contract-Driven Development (DR-359) established session discipline and T3 regression tests, but has no enforced diagnostic phase before bug fixes. Current rule ("commit WIP, fix separately, resume") prevents scope bleed but doesn't prevent symptom-fixing.

AI agents are especially prone to jumping straight to code changes — they optimize for completion, not understanding. Without a structural gate that prohibits code changes during diagnosis, root causes get papered over.

The fix:feat ratio across projects (1.38 overall, 2.0 in 333Method) confirms that shallow fixes are creating rework.

## Decision

All bug-fix sessions follow a four-phase protocol with a hard gate: **no code changes in Phases 1-2.**

### RCDD — Root-Cause-Driven Debugging

```
Session scope: "Fix <issue ref>"
Type: bug (RCDD)

Phase 1 — Reproduce (no code changes)
  1. Reproduce the bug. Capture exact steps + actual vs expected output.
  2. If cannot reproduce → close issue or request more info. Stop.

Phase 2 — Diagnose (no code changes)
  3. Identify root cause. Write it as one sentence.
  4. Identify why existing tests didn't catch it.
  5. Check: is this a symptom of a deeper issue?
     If yes → file separate issue for the deeper problem.

Phase 3 — Fix
  6. Write T3 regression test FIRST (must fail before fix).
  7. Fix root cause (not symptom).
  8. Verify regression test passes.
  9. Run full test suite — no regressions introduced.

Phase 4 — Harden
  10. If root cause was "no test coverage" → add T0/T1 test
      for the invariant that was violated.
  11. If root cause was "contract violation" → tighten the contract.
  12. Commit fix + tests. Separate commit from any WIP.
```

### Key constraints

- **No code changes in Phases 1-2.** This is the structural gate. If an agent edits a file before writing the root-cause sentence, it violated the protocol.
- **Root-cause sentence is mandatory.** Must be written (in session output or commit message) before any fix code. Format: `Root cause: <one sentence>`.
- **T3 test before fix.** Test must fail before the fix is applied. This proves the test actually covers the bug.
- **Deeper-issue check.** If the bug is a symptom, the deeper issue gets its own issue — not a scope expansion of the current session.

### AI agent enforcement

Add to every bug-type agent prompt:

```
RCDD RULES (Root-Cause-Driven Debugging):
- Phase 1-2: diagnostic only. NO file edits, NO code changes.
- You MUST write "Root cause: <sentence>" before any fix.
- Write failing T3 test BEFORE implementing fix.
- If bug is symptom of deeper issue, output "DEEPER: <description>" and continue with immediate fix only.
```

## Consequences

- Bug sessions take slightly longer upfront (diagnosis phase)
- Significantly fewer repeat fixes for the same underlying issue
- T3 regression test coverage grows with every bug fixed
- AI agents forced to understand before acting
- Commit messages gain `Root cause:` line → searchable history of why bugs happened
