> **Note:** Originally named *Bug Session Protocol*. Renamed to **RCDD (Root-Cause-Driven Debugging)** on 2026-05-29 to join the SDD/CDD/TDD acronym family and to surface the differentiator (root-cause discipline, not just "bug session"). Behaviour and gates are unchanged.

## Context

Contract-Driven Development (DR-359) established session discipline and T3 regression tests, but has no enforced diagnostic phase before bug fixes. Current rule ("commit WIP, fix separately, resume") prevents scope bleed but doesn't prevent symptom-fixing.

AI agents are especially prone to jumping straight to code changes — they optimize for completion, not understanding. Without a structural gate that prohibits code changes during diagnosis, root causes get papered over.

The fix:feat ratio across projects (1.38 overall, 2.0 in 333Method) confirms that shallow fixes are creating rework.

## Decision

All bug-fix sessions follow a four-phase protocol with a hard gate: **no code changes in Phases 1-2.**

### RCDD — Root-Cause-Driven Debugging

```
Session scope: "Fix <issue ref>"
Type: bug (RCDD)

Phase 1 — Reproduce (no code changes)
  1. Reproduce the bug. Capture exact steps + actual vs expected output.
  2. If cannot reproduce → close issue or request more info. Stop.

Phase 2 — Diagnose (no code changes)
  3. Identify root cause. Write it as one sentence.
  4. Identify why existing tests didn't catch it.
  5. Check: is this a symptom of a deeper issue?
     If yes → file separate issue for the deeper problem.

Phase 3 — Fix
  6. Write T3 regression test FIRST (must fail before fix).
  7. Fix root cause (not symptom).
  8. Verify regression test passes.
  9. Run full test suite — no regressions introduced.

Phase 4 — Harden
  10. If root cause was "no test coverage" → add T0/T1 test
      for the invariant that was violated.
  11. If root cause was "contract violation" → tighten the contract.
  12. Commit fix + tests. Separate commit from any WIP.
```

### Key constraints

- **No code changes in Phases 1-2.** This is the structural gate. If an agent edits a file before writing the root-cause sentence, it violated the protocol.
- **Root-cause sentence is mandatory.** Must be written (in session output or commit message) before any fix code. Format: `Root cause: <one sentence>`.
- **T3 test before fix.** Test must fail before the fix is applied. This proves the test actually covers the bug.
- **Deeper-issue check.** If the bug is a symptom, the deeper issue gets its own issue — not a scope expansion of the current session.

### AI agent enforcement

Add to every bug-type agent prompt:

```
RCDD RULES (Root-Cause-Driven Debugging):
- Phase 1-2: diagnostic only. NO file edits, NO code changes.
- You MUST write "Root cause: <sentence>" before any fix.
- Write failing T3 test BEFORE implementing fix.
- If bug is symptom of deeper issue, output "DEEPER: <description>" and continue with immediate fix only.
```

## Costly to Refactor

- **Low — primarily process/methodology**, reversible by editing the protocol. The one hard seam: the **commit-msg gate token** (`Root cause:`); changing it re-tools `.githooks/commit-msg` + retrains authors/agents. *Check: the gate token is stable.*

## Consequences

**Positive:**

- Significantly fewer repeat fixes for the same underlying issue — directly attacks the 1.38 overall / 2.0-in-333Method fix:feat ratio that motivated this DR (Context).
- T3 regression test coverage grows with every bug fixed — Phase 3 step 6 mandates a failing T3 test before each fix.
- AI agents forced to understand before acting — the "no code changes in Phases 1-2" structural gate and mandatory `Root cause:` sentence block the agent's completion-bias jump-to-code (Context).
- Commit messages gain a `Root cause:` line → searchable history of *why* each bug happened, enforced by `.githooks/commit-msg`.

**Negative:**

- Bug sessions take slightly longer upfront (the diagnosis phase adds Phases 1-2 before any fix lands).
- The `Root cause:` commit-msg gate has a false-pass mode: any sentence satisfies it, including a restatement of the bad state — exactly the SPEC-004 data-only "fix" the Addendum (2026-06-01) had to retrofit Phase 2/3/4 refinements to close. The token gate proves a sentence *exists*, not that it names a mechanism.
- Authors/agents must learn the `Root cause:` token and the four-phase shape; an honest `RCDD_GATE_OFF=1` bypass exists (CLAUDE.md Pre-Commit Checks), so the gate is advisory, not airtight.

## Addendum — RCDD on the RCDD (2026-06-01)

**Trigger.** "SPEC-004 has no epic" was first "fixed" by adding the missing
`epic:` frontmatter line — a pure data edit — and committed with a `Root cause:`
line, satisfying the gate while explaining nothing. The bad *state* was restated
as if it were the *cause*.

**Root cause of the RCDD miss.** Phase 2's "write root cause as one sentence"
accepts any sentence, including a description of the bad end-state, because nothing
forces the diagnosis past the state to (a) the mechanism that produced it and
(b) the gate that should have rejected it. So a data-only fix passed as a
root-cause fix.

**Refinements (now folded into the four phases above):**

1. **Phase 2 — mechanism, not state.** A root cause must name a *mechanism*: the
   code/process that produced the bad state, or the gate that failed to reject it.
   "X is missing/null/wrong" is a symptom. "Y allowed X and gate Z never checked
   for it" is a cause. Phase 2 is unfinished until both *what produced it* and
   *what should have caught it* are named.

2. **Phase 3 — data-only-fix corollary.** A pure data/config/frontmatter fix means
   the root cause is not yet found. A shouldn't-be-possible bad state implies a
   missing or asymmetric gate. Patch the data to unblock; then fix the gate.

3. **Phase 4 — gate asymmetry check.** Add/repair the gate that makes the bad state
   un-committable, and inspect it for asymmetry: does it validate values that
   *exist* but never assert one *should* exist? That was the actual defect —
   `validateSpec` warned on **dangling** epic refs (`epic.unresolved`) but never on
   **missing** ones, so an untagged spec was invisible.

**Materialized gates (not prose-only — DR-023):**
- Extension validator: `epic.missing` rule in
  `packages/minspec/src/lib/spec-validator.ts` + tests (warning, FR-9 keeps epics soft).
- CI: epic referential-integrity check in `scripts/validate-frontmatter.ts`
  (hard error for this repo, which has adopted epics; run by `.github/workflows/ci.yml`).

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| `Root cause:` token rewards a restatement of the bad state, not a mechanism — gate passes while diagnosis explains nothing. | This already fired: the SPEC-004 missing-`epic:` data-only "fix" passed the `.githooks/commit-msg` gate (Addendum, 2026-06-01). | Phase 2 "mechanism, not state" + Phase 3 data-only-fix corollary + Phase 4 gate-asymmetry check fold the lesson in; backed by the `epic.missing` rule in `packages/minspec/src/lib/spec-validator.ts` and `scripts/validate-frontmatter.ts`. |
| Gate is bypassable, so it's advisory not enforced. | `RCDD_GATE_OFF=1 git commit` documented escape hatch (CLAUDE.md Pre-Commit Checks). | Accepted by design (avoids blocking legitimate WIP); the searchable `Root cause:` history makes silent skips auditable after the fact. |
| Changing the `Root cause:` token string breaks every consumer at once. | `.githooks/commit-msg` parses the literal token; authors + every bug-type agent prompt embed it (Decision → AI agent enforcement block). | Treat the token as a stable contract (see Costly to Refactor); don't rename it casually. |
| Phase 1-2 "no code changes" slows trivial/obvious bugs where the cause is self-evident. | The hard gate applies to all bug sessions uniformly (Key constraints). | One-line root-cause sentence is cheap; the deeper-issue check (Phase 2 step 5) prevents the slowdown from compounding into scope creep. |

## Alternatives Considered

- **Rely on DR-359 (Contract-Driven Development) session discipline alone.** Rejected: DR-359 established T3 regression tests and "commit WIP, fix separately, resume" but has *no enforced diagnostic phase* — it prevents scope bleed but not symptom-fixing (Context, ¶1). RCDD adds the missing no-code-in-Phases-1-2 gate.
- **Soft convention / guideline with no commit-msg gate.** Rejected: AI agents optimize for completion and paper over root causes without a *structural* prohibition (Context, ¶2). The `.githooks/commit-msg` `Root cause:` requirement makes the diagnosis a hard precondition of a `fix:` commit landing.
- **Accept the original Phase 2 "write root cause as one sentence" unmodified.** Rejected after it admitted the SPEC-004 data-only fix; replaced (not removed) by the mechanism-not-state / data-only-fix-corollary / gate-asymmetry refinements (Addendum, 2026-06-01) rather than discarding the four-phase shape.
