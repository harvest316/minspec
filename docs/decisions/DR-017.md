# DR-017: Agent-execute Layer-2 execution substrate

## Context

DR-008 (accepted) makes unattended `claude -p` dispatch conditional on **Layer 2**:
the agent must execute inside an isolation boundary with **no host credentials**,
egress denied by default, and the branch leaving the sandbox as a diff/bundle that a
credentialed host process reviews and pushes. DR-015 (accepted) places that Layer-2
work in a **third Tier-1 extension** (`aiclarity.agent-execute`) — the natural and
only home for the sandbox. DR-016 established the in-extension `claude -p` Tier-1
delegation pattern (availability check + graceful fallback mandatory).

What neither DR answers: **how** Layer 2 is physically built. The recurring framing
— "is this better done in a docker container than a vsix?" — contains a category
error worth settling explicitly, because the wrong mental model produces an
unsatisfiable design.

**A vsix cannot be the sandbox.** A VS Code extension runs in the extension host,
which is the user's own process: same machine, same `$HOME`, same `gh` token,
`CLOUDFLARE_API_TOKEN`, FTP/psql/wrangler creds, same `~/.claude` and keychain. If
`claude -p` runs *inside* the extension host, every Layer-2 condition is violated by
construction. Docker (or an equivalent boundary) is therefore not an *alternative* to
the vsix — it is the **execution substrate the vsix structurally lacks**.

## Decision

Build agent-execute as **two planes**. The vsix is the credentialed **control plane**;
an isolated container is the credential-free **execution plane**. The vsix *manages*
the substrate (spawns/monitors a container via the docker CLI/SDK); it never *becomes*
the sandbox and the agent never runs in the extension host.

| Plane | Where | Host creds | Network | Responsibilities |
|---|---|---|---|---|
| **Control** | `agent-execute` vsix (ext host) | yes | yes | pick `agent-ready` issue, tier-gate via shared classifier (DR-014), HITL approval UI (DR-012 ethos), review returned diff, `git push` + `gh issue comment` |
| **Execution** | container, clean env | **none host** (model access only, see below) | **deny-by-default + narrow allowlist** | run `claude -p` against the issue, edit code, run build/test, commit locally, emit diff/bundle + `.agent-summary.md` |

This maps DR-008 Layer 2 line-for-line:
- clean env, no creds mounted → execution plane gets no `~/.claude`, no tokens, no
  `~/.config`/`~/.cache/wrangler-home`/keychain;
- egress denied (`--network none`, or allowlisted to the package registry only);
- branch exits the sandbox as a **diff/bundle**, not a push;
- the credentialed control plane reviews + pushes; human (or trusted non-agent gate)
  approves the merge — the existing `needs-review` flow for T3–T4 already supplies the
  human gate.

The `npm run <script>` = arbitrary-code escape DR-008 named is now *contained*: the
container is where that arbitrary code is expected to run, and it holds nothing worth
stealing and no egress to exfiltrate it through.

### Substrate selection

Docker is the v1 substrate but **not** assumed as the only one. The axis is
**isolation strength × install friction × cross-platform reach**:

- **Docker / devcontainer** — ubiquitous; clean env trivial (`--network none`, no
  cred mounts); devcontainer reuses VS Code's own remote-container UX. Namespace
  isolation (not kernel). **Chosen for v1**, devcontainer-flavoured to reuse existing
  primitives.
- **microVM (Firecracker) / gVisor** — kernel-level boundary, stronger against
  container escape; heavier to ship. **Hardening path**, revisit before untrusted
  (non-self-authored) issue bodies are dispatched unattended.
- **Remote/cloud sandbox** — removes the local dependency but reintroduces network,
  a credential-handoff surface, and per-run cost. Deferred.
- **Linux-only (bubblewrap/nsjail)** — rejected as sole substrate: no cross-platform
  story.

### Model access — host-side broker, credential-free sandbox (decided)

`claude -p` needs model access (auth) **and** egress to the model endpoint, but
attestation forbids any credential or open egress in the sandbox. **Decision: a
host-side broker is the single model-access seam.** It is not one of two options — it
is the only design in which the sandbox's endpoint never has to change.

- The sandbox's `ANTHROPIC_BASE_URL` **always** points at the broker socket — a fixed,
  loopback/unix-socket seam that is the *only* allowlisted egress. The sandbox `claude
  -p` runs **credential-free**: it sends model requests with no real token; the broker
  (host-side) **injects** the real credential and makes the actual outbound call.
- **Installing ScroogeLLM repoints the broker, never the sandbox.** Whether the broker
  routes direct→Anthropic or via the local Scrooge proxy is a host-side config flip.
  Were the sandbox to call `api.anthropic.com` directly (rejected scoped-key design),
  enabling Scrooge would mean editing `ANTHROPIC_BASE_URL` *inside* an isolated,
  egress-denied box whose loopback isn't the host's — unworkable. The broker makes the
  sandbox endpoint stable for all time.

#### Billing mode — subscription default (no PAYG), API/Scrooge opt-in

`claude -p` authenticates on the dev's **Claude subscription** (Pro/Max quota), **not**
pay-as-you-go API tokens. Most devs will prefer this: the agent system spends their
existing subscription quota, not metered API credit. So:

- **Default: subscription mode.** Broker carries the dev's subscription credential
  (held host-side), routes direct→Anthropic. No PAYG. No API key anywhere.
- **Opt-in: API-key mode.** For devs who want Scrooge's cost-optimization/routing or
  higher concurrency than a subscription allows, the broker injects a spend-capped API
  key and routes via Scrooge. PAYG, but optimized + measured (the three-verb synergy
  applies *here*, not in the default).
- Scrooge value differs by mode: in API mode it cuts **$**; in subscription mode the
  constraint is **quota/rate**, so its routing (cheap tasks → cheap models) stretches
  subscription hours rather than dollars. Still useful, different axis.

> **Open question for the Specify cycle:** can the claude CLI's **subscription oauth**
> be broker-injected — sandbox sends sans-credential, broker adds the subscription
> token — or does subscription auth bind the token to where the CLI runs (and/or does
> ToS/endpoint-pinning forbid pointing subscription `claude -p` at a proxy)? **API-key
> mode is definitely broker-injectable; subscription mode is the unknown.** If
> injectable → clean. If not → fallback: inject the subscription token at spawn
> (weaker — attestation must then whitelist exactly that one token, nothing else).
> Resolve before committing the default-mode plumbing.

Model access is the **one** narrow exception to "no creds / no egress," confined to the
broker seam. Everything in the attestation manifest below still applies — in the broker
design the sandbox holds **no** Anthropic key and **no** `~/.claude`, so attestation
stays clean; only the spawn-token fallback would force a documented exception.

### Usage/quota limits (not install limits)

There is **no per-install license cap** on the `claude` CLI binary — the wall is
**usage**, metered on the account: Pro ≈45 prompts/5h-window + weekly cap; Max larger
but with a ~50-sessions/month throttle; Team seats 1.25×/6.25× Pro. Implication for
autonomous fanout: N concurrent sandboxes = N concurrent `claude -p` against the
**shared** account quota. Therefore the dispatch loop carries a **concurrency cap**
always, plus — in API mode — a **spend cap** on the injected key. In the
subscription default the binding limit is the 5h-window/weekly/session quota above, so
the concurrency cap must respect it or runs will throttle mid-flight; API mode trades
PAYG dollars for headroom past the subscription ceiling. (Verify exact current limits
against the active Anthropic plan before wiring caps.)

### Graceful degrade when no substrate present — fall back to Layer 1, not "off"

Per DR-016's mandatory-fallback rule: the extension **detects** a usable container
runtime (mirrors `isGhAvailable()` / `claude` detection). Absent it, the boundary the
sandbox provides is required only for **unattended** runs (DR-008 Layer 2). So the
correct degrade is **not "disabled"** — it is **fall back to Layer-1 manual dispatch**,
which DR-008 already permits (human-initiated, per-issue, trusted/self-authored body,
unsandboxed). Two distinct modes:

| Mode | Trigger | Sandbox | DR-008 |
|---|---|---|---|
| **Manual dispatch** | human, per-issue, trusted body | not required | **Layer 1 — already permitted** |
| **Autonomous / cron** | unattended loop | **required** | Layer 2 |

This corrects the product framing: the reason to install agent-execute is **any
code-writing dispatch**, not autonomy alone — manual HITL dispatch is the floor (works
without docker), and **docker unlocks the autonomous premium tier**. The agent never
runs in the extension host regardless; "no docker" downgrades *autonomy*, never the
*boundary*.

## Boundary verification — control-plane attestation

The Layer-2 conditions must be **verified, not assumed**. Before handing any issue to
the agent, the control plane spawns the container, injects a **probe**, captures a
structured result, compares it to an **expected-deny manifest**, and **fails closed**:
any should-be-denied capability that *succeeds* aborts the dispatch — the agent never
runs — and a red report goes to the dev.

Probe = negative assertions (each must **fail** inside the sandbox to pass):

| Check | Attempt | Pass = |
|---|---|---|
| Egress | connect a canary host (gh API / control-plane beacon) | refused / timeout |
| Creds (env) | read `$GH_TOKEN`, `$CLOUDFLARE_API_TOKEN`, `$ANTHROPIC_API_KEY` | unset |
| Creds (files) | stat `~/.claude`, `~/.config/gh`, `~/.cache/wrangler-home`, `~/.aws`, `~/.ssh`, `.netrc` | absent |
| Auth | `gh auth status` | fails |
| Push | `git push origin` | fails (no cred / no net) |
| FS boundary | read host `$HOME` / `/host` | not mounted |
| Privilege | docker socket present? running privileged? | no |

Two requirements stop the attestation from lying:

1. **Negative checks need positive controls.** A failed `curl` could mean
   egress-blocked, curl-missing, or canary-down. Pair each deny-check with a control:
   the *allowlisted* endpoint (model API / broker socket) **must succeed**. If both the
   deny-check and its control fail → broken probe, not a proven boundary → verdict
   **FAIL**. Never infer "secure" from a dead probe.
2. **Attest the box you execute in.** The probe runs in the *same* container lifecycle,
   same env / mounts / netns, **before** any agent code. Network policy and mounts are
   fixed at container creation and immutable from inside without privilege, so a clean
   attestation at spawn holds for the run.

**Scope honesty:** attestation proves *"as configured, this box cannot reach X"* — a
config-correctness gate. It does **not** prove the absence of a kernel/container-escape
0-day; that residual is what the microVM/gVisor hardening path addresses.

**Two-for-one:** this probe suite **is** the substrate integration test. One mechanism
serves both as a per-dispatch runtime gate and as the CI test that runs against a
docker-capable runner.

## Consequences

- DR-008 Layer 2 is satisfiable and lands in exactly one place (agent-execute), per
  DR-015. Unattended/cron dispatch becomes permissible **only** once this substrate
  ships, **passes attestation**, and clears its own security review.
- **Sandbox is a port, not a hardcoded docker call.** A `SandboxRunner` interface
  (spawn → attest → run → collect-diff → teardown) keeps ~95% of agent-execute —
  tier-gating, HITL state, diff handoff, summary→comment — testable with a **mock
  runner**, no docker needed. Only the docker/devcontainer *adapter* requires a real
  daemon. Also future-proofs the microVM/cloud substrates as alternate adapters.
- New runtime dependency for the *autonomous* path: a container engine on the user's
  machine. Tier-1, opt-in, separately installed (DR-004) — consistent with the
  three-verb product story. The Tier-0 MinSpec core gains nothing and stays air-gapped.
- Two-tier test strategy follows from the port: **control-plane + logic tests** run
  anywhere (mock `SandboxRunner`, incl. this docker-less dev container); **substrate
  integration tests** (= the attestation suite) need a real daemon and run on a
  docker-capable runner. Nesting (this docker → sibling docker) is possible via
  docker-out-of-docker (mount the host socket) or docker-in-docker (privileged), but
  the cleaner path is a dedicated CI runner with native docker — keep this dev
  container's socket unmounted. Per platform CLAUDE.md, docker is host-only here.
- A credential-free egress-denied container is a strong but **namespace-level**
  boundary; container escape is the residual risk, which is why microVM/gVisor is the
  named hardening path before untrusted input.
- Diff-handoff (not in-sandbox push) means the agent cannot self-link its PR mid-run;
  the control plane composes the comment from `.agent-summary.md` — already the DR-008
  Layer-1 posture, now the permanent contract.

## Alternatives considered

- **Agent runs in the extension host (no container).** Simplest; ships nothing extra.
  Rejected: violates every DR-008 Layer-2 condition — this is the category error the
  Context section names.
- **vsix bundles its own mini-sandbox (seccomp/namespaces by hand).** Reinvents
  Docker badly, Linux-only, large security-review surface. Rejected.
- **microVM from day one.** Strongest boundary, but heavy install + slower per-job;
  disproportionate for v1 self-authored (trusted) issues. Kept as the hardening path.
- **Cloud execution.** No local dep, but adds network + cred-handoff + cost; conflicts
  with the local-first posture for v1. Deferred.
- **Scoped key in the sandbox, direct to `api.anthropic.com`** (no broker). Rejected:
  (1) puts a credential inside the box; (2) it is PAYG-only — can't use the dev's
  subscription quota, which is the billing mode most devs will want; (3) enabling
  Scrooge later would require repointing `ANTHROPIC_BASE_URL` *inside* an isolated
  egress-denied sandbox, which is unworkable. The broker dominates it on all three.

## Status

Proposed. Implements DR-008 Layer 2 as a verified, attested boundary; scoped by DR-015
(third extension); follows DR-016's Tier-1 detect-or-degrade pattern; tier-gates via
DR-014 shared classifier; HITL ethos from DR-012; model access via a host-side broker
(subscription `claude -p` default, no PAYG; API/ScroogeLLM opt-in). Next step:
agent-execute Specify cycle (naming + spec deferred per DR-015 / #56) must adopt this
substrate model — the `SandboxRunner` port, the model-access broker (with the
subscription-oauth-injection open question — [#74](https://github.com/harvest316/minspec/issues/74) — resolved first), the attestation gate, and
the manual/autonomous mode split are its load-bearing contracts. Cron/unattended
dispatch stays **blocked** until this ships, **passes attestation**, and clears
security review. microVM/gVisor hardening (kernel boundary, required before
**untrusted** issue dispatch) is tracked separately in
[#73](https://github.com/harvest316/minspec/issues/73).
