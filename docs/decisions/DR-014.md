# DR-014: Shared-code boundary and co-existence rules for MinSpec + ScroogeLLM

## Context

MinSpec (`aiclarity.minspec`) and ScroogeLLM (`aiclarity.scroogellm`) are two
independent VS Code extensions in one monorepo. They can be installed **together**
— either separately, or via the MinSpec Pro pack (`aiclarity.minspec-pro`), which
is byte-identical at runtime to installing both (the pack only references them).
So "both installed" is the case that must be safe; the pack is not a safeguard.

Two facts force a boundary decision now:

1. **The classifier is becoming a real second-consumer shared module.** ScroogeLLM's
   router will route cheap-model vs SOTA partly on the task **complexity tier**, so
   it needs the same tier vocabulary and scoring engine MinSpec uses. Until now the
   classifier lived in `packages/minspec/src/lib/classifier.ts` with one consumer;
   CLAUDE.md already (incorrectly) described `@aiclarity/shared` as "the shared
   classifier." It is time to make that true — correctly.

2. **DR-004 split the product into network/trust tiers.** `@aiclarity/shared` and
   `packages/minspec` are **Tier 0** (air-gapped: no `http`/`https`/`fetch`/`net`,
   no process spawn). That offline guarantee is MinSpec's marketplace selling point.
   Anything shared must not break it.

When two extensions touch the same project's `.minspec/` directory and each bundle
their own copy of shared code, three overlap hazards appear that the manifest layer
does **not** catch (commands/config/views are already namespaced `minspec.*` /
`scroogellm.*`):

- **Concurrent writers to a shared on-disk artifact** (e.g. `.minspec/calibration.json`)
  → race / corruption.
- **Version skew between the two bundled copies of `@aiclarity/shared`** → MinSpec
  classifies a task T3 while ScroogeLLM routes it as T2. No crash; silent disagreement.
- **Tier-0 contamination** — pulling Tier-1 logic (CLI delegation, process spawn)
  into `shared` to "reuse" it would void the air-gapped claim for both extensions.

## Decision

### 1. Tier → package map (canonical)

| Package | Tier | May contain | May NOT contain |
|---|---|---|---|
| `@aiclarity/shared` | **0** | Pure TS: contract types, classifier engine, pure logic, `fs`/`path` reads of `.minspec/` | `vscode`, `http`/`https`/`fetch`/`net`, child-process spawn |
| `packages/minspec` | **0–1** | Extension UI/commands, signal producers, MinSpec-owned writes | network code in Tier-0 paths |
| `packages/scroogellm` | **2** | Proxy, routing, network (localhost-default per INV-11) | writes to MinSpec-owned artifacts |
| Tier-1 agent runtime | **1** | CLI delegation, process spawn (see DR-015) | — must live **outside** `shared` |

Tier-1 process-spawn / CLI-delegation code is **never** added to `@aiclarity/shared`.
If shared between extensions it gets its own package (DR-015), preserving shared's
Tier-0 air-gapped guarantee.

### 2. Classifier split — engine to shared, signal producers stay local

Move to `@aiclarity/shared` (Tier 0, no vscode):

- `Tier` / `Phase` types, `ClassificationSignal` / `ClassificationResult` shapes.
- The scoring function (signals → tier), tier-compare, suggested-phase mapping.
- Calibration **read** (`.minspec/calibration.json`).

Stay in `packages/minspec`:

- **Signal producers** — `git-analyzer`, `ast-analyzer`. They measure a *code diff*.

ScroogeLLM produces its **own** signals from the prompt/request it is about to route,
then calls the shared scorer. Same engine, different signals in. This is the seam:
**share the engine + vocabulary, each extension brings its own signal producers.**

### 3. Single-writer per on-disk artifact

Every file under a shared location has exactly **one writer at any time**; all other
consumers are read-only.

| Artifact | Writer | Readers |
|---|---|---|
| `.minspec/calibration.json` | **MinSpec when present; ScroogeLLM when MinSpec absent** (elected — see below) | the non-writer (read-only, to route) |
| Conformance contract export | **MinSpec** (`bridge.ts`) | ScroogeLLM (read-only) — already the design |
| `.minspec/config.json`, specs, ADRs | **MinSpec** | ScroogeLLM read-only |

#### Calibration writer election (issue #53)

Calibration ownership is **conditional, not absolute** — ScroogeLLM is a standalone
product and must still self-calibrate when installed alone:

- **MinSpec present (installed + enabled) → MinSpec is sole writer.** ScroogeLLM
  read-only, consumes calibration to route.
- **MinSpec absent → ScroogeLLM is the writer.** Falls back to owning the file.

The check is **asymmetric, which removes any election race**: MinSpec writes
*unconditionally* (never checks for ScroogeLLM); only ScroogeLLM tests for MinSpec
(`vscode.extensions.getExtension('aiclarity.minspec')`) before taking the writer
role. So two writers can never coexist — MinSpec always wins by construction, no lock
or handshake needed. ScroogeLLM re-evaluates on `vscode.extensions.onDidChange` and
**drops** the writer role the moment MinSpec appears (and takes it when MinSpec is
removed/disabled).

Handoff is seamless because the **shared engine writes one identical schema**
(§2, §4): whichever extension owns the file, the format and semantics match, so the
other reads it without translation. Conformance/config/specs remain
MinSpec-only-writer regardless — no fallback, since they are MinSpec artifacts with no
meaning when MinSpec is absent.

Shared calibration also means a tier correction a user makes in MinSpec fixes
ScroogeLLM's routing for that project — a feature, not just a constraint (sell it; #55).

### 4. Version lockstep + detectable mismatch

Each `.vsix` bundles its own copy of `@aiclarity/shared` (esbuild). Two copies are
safe **only at the same version**.

- Both extensions pin and ship the **same `@aiclarity/shared` version**, bumped together.
- The Pro pack only ever ships **matched** versions; never publish one extension on a
  newer shared than the other.
- Stamp an **engine/contract version** in the artifacts that cross the boundary
  (`ConformanceContract.version` already exists; add an engine-version field to the
  calibration read path) so a skew is **detected and surfaced**, not silent.

### 5. Namespacing (already holding — keep it)

Commands, config keys, view IDs stay per-extension prefixed (`minspec.*` /
`scroogellm.*`). Never a shared key. The cross-extension upsell nudge is already
suppressed when the other extension is installed (`bridge.ts`); ScroogeLLM must
**mirror** this (no Pro upsell when MinSpec present).

## Consequences

- Classifier engine + types relocate from `packages/minspec/src/lib/classifier.ts`
  to `@aiclarity/shared`; minspec imports them via the package name (never deep paths
  — see the `index.ts` barrel rationale). Signal producers stay put.
- A single-writer rule must be enforced in review for every `.minspec/` artifact.
- CI/publish gate: assert both extensions resolve the **same** `@aiclarity/shared`
  version before packaging the pack.
- An engine-version field is added to the calibration/contract read path.
- ScroogeLLM's Specify phase inherits these as hard constraints (read-only consumer,
  own signal producers, mirrored nudge).

## Alternatives considered

- **Leave classifier in minspec, have ScroogeLLM depend on the minspec package** —
  rejected: makes ScroogeLLM depend on a vscode-coupled extension package and its UI
  surface; the Tier-0 engine should not drag the extension with it.
- **Both extensions write calibration** — rejected: concurrent-writer race on a JSON
  file with no locking; corruption under normal use.
- **Allow independent shared versions per extension** — rejected: silent tier
  disagreement is the worst failure mode (no error, wrong routing/classification).
- **Fold the Tier-1 agent runtime into `shared` to reuse it** — rejected: voids the
  air-gapped Tier-0 marketplace claim for both extensions (DR-004).

## Status

Proposed. Establishes the boundary that ScroogeLLM's Specify phase and the
classifier relocation build against. Tier-1 packaging is decided separately in
DR-015. Single-writer + version-lockstep enforcement, the classifier move, and the
calibration-benefit marketing are tracked as issues.
