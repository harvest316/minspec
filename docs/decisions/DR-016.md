# DR-016: AI-assisted epic backfill

## Context

DR-013 shipped registered epics, but a project that adopts the feature mid-life
has dozens of existing specs/ADRs/issues carrying no `epic:` reference. Tagging
them by hand is the exact tedium the tool should remove. We want MinSpec to
**propose** an epic taxonomy + an artifact→epic mapping, and to **offer** it
during onboarding.

The obvious tension: good taxonomy-guessing wants semantic reasoning over titles
and prose — an LLM job — but **DR-004 Tier 0** forbids any network/AI in the core
`packages/minspec`. Does this break the air-gapped promise?

**It does not.** DR-004 already resolves it. `claude -p` is **Tier 1** (local tool
delegation), the same category as `gh` (already shelled via `execFile` in
`backlog.ts`) and `claude --bg` (named explicitly in DR-004). The extension makes
**zero outbound connections**; the locally-installed `claude` binary owns its own
networking and auth. No invariant is rewritten and no customer promise is broken,
provided backfill stays Tier 1: opt-in, clearly labeled, and gracefully degrading
when `claude` is absent. The air-gapped core gains no AI dependency.

## Decision

Ship epic backfill as a **two-engine, offer-only, human-gated** feature.

### Two engines

1. **Tier 0 — heuristic pass (always available, air-gapped).** Cluster existing
   specs/ADRs into candidate epics from signals already in the repo:
   - spec-kit subdirectory name (`specs/minspec/epic-grouping/` → epic
     "Epic Grouping"),
   - shared title/tag token overlap (reuse the Jaccard tokenizer behind
     `findSimilarAdrs`),
   - existing `epic:` refs and DR↔spec `Triggered by:` cross-links.
   Pure file-system. No `claude`, no network.

2. **Tier 1 — AI pass (opt-in, needs `claude`).** Shell `claude -p` with a
   compact digest of all artifacts (id, title, kind, path, first prose
   paragraph) and a strict "emit ONLY JSON matching this schema" instruction.
   Parse + validate into the same proposal shape. On any failure (binary absent,
   non-JSON, timeout) **fall back to the heuristic proposal** — never error.

Both engines produce the same `BackfillProposal` contract, so the review +
apply path is engine-agnostic.

### Offer-only onboarding (core stays Tier 0)

`auto-bootstrap` is already a detect-and-offer system. Add a
`hasUnbackfilledEpics()` detector and a bootstrap step that **offers** backfill.
The `claude -p` call runs only on explicit user action — never during first-run
itself, so onboarding remains air-gapped.

### Human-in-the-loop before any write (DR-012 ethos)

Backfill **proposes**; it never silently rewrites frontmatter. The proposal
(candidate epics + per-artifact mapping with rationale/confidence) is surfaced
for review; only on explicit approval does `applyBackfill` create the EPIC-NNN
files, insert `epic:` frontmatter into the mapped artifacts, and regenerate the
epic INDEX. Idempotent; touches only artifacts in the approved proposal; an
artifact that already has an `epic:` is left untouched unless the user opts to
override.

## Consequences

**Positive:**

- A single `BackfillProposal` contract makes the review + apply path
  engine-agnostic — the heuristic and AI engines are interchangeable, so the
  feature degrades from Tier 1 to Tier 0 without a second UI code path.
- The Tier-0 heuristic pass (spec-kit subdir name, `findSimilarAdrs` Jaccard
  token overlap, existing `epic:`/`Triggered by:` cross-links) means backfill
  works air-gapped with no `claude` installed — the core's DR-004 promise holds.
- Onboarding stays Tier 0: `auto-bootstrap` only *offers* backfill via the new
  `hasUnbackfilledEpics()` detector; the `claude -p` call fires only on explicit
  user action, never during first-run, so first-run makes zero outbound calls.
- HITL gate (DR-012 ethos): `applyBackfill` is idempotent, touches only approved
  artifacts, and leaves any artifact with an existing `epic:` untouched unless the
  user overrides — no silent frontmatter rewrites.
- Establishes the reusable Tier-1 LLM-CLI pattern (first in-extension `claude -p`
  delegation vs. the existing `scripts/` path) that later features can follow.
- `setArtifactEpic(filePath, ref)` is a generic frontmatter-insert helper
  (mirrors `setAdrStatus`), reused across specs and ADRs rather than duplicated.

**Negative:**

- New surface area to maintain: module `lib/epic-backfill.ts` (proposal types,
  heuristic engine, AI engine, apply), command `minspec.backfillEpics`, an
  onboarding offer step, and the `setArtifactEpic` helper.
- The Tier-1 `claude -p` delegation makes the availability check + graceful
  fallback *mandatory and load-bearing* — if a future edit assumes `claude` is
  present (skips the `isGhAvailable()`-style guard), the feature errors instead of
  degrading, silently violating "never error" (line 52).
- Cross-repo, non-code follow-up cost: marketplace copy / docs MUST label backfill
  as optional Tier-1 ("AI-enhanced; requires Claude Code") and state the heuristic
  pass works without it — prose that never enters the SDD flow (DR-023 leak risk).

## Alternatives considered

- **AI-only.** Simpler, but the feature is dead without `claude` — no air-gapped
  path. Rejected: a core-adjacent feature should have a Tier-0 mode.
- **Heuristic-only.** Stays purely in Tier 0, but weak taxonomy guesses (no
  semantic grasp of what a spec is *about*). Rejected as the sole engine; kept as
  the fallback + offline mode.
- **Auto-apply the mapping.** Fastest, but an AI silently rewriting frontmatter
  across every doc is exactly the trust failure DR-012 guards against. Rejected:
  HITL review is mandatory.
- **Rewrite the Tier-0 invariant to allow AI in core.** Unnecessary — DR-004's
  tiering already permits `claude -p` as Tier 1. Rejected: would weaken the
  air-gapped promise for no gain.

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|------|--------------------|------------|
| Air-gapped promise quietly broken | A future edit lets the `claude -p` path run during `auto-bootstrap` first-run, or skips the `isGhAvailable()`-style availability guard | First-run only *offers* via `hasUnbackfilledEpics()`; the AI call is gated behind explicit user action; availability check + heuristic fallback are mandatory (Decision §Two engines, item 2) |
| Malformed/hallucinated AI output corrupts frontmatter | `claude -p` returns non-JSON, a wrong-schema object, or a fabricated `EPIC-NNN` mapping | Strict "emit ONLY JSON matching this schema" prompt + parse/validate into `BackfillProposal`; any failure (non-JSON, timeout, binary absent) falls back to the heuristic proposal — never errors (line 52) |
| AI silently rewrites docs across the repo | `applyBackfill` could mass-edit `epic:` frontmatter without consent | HITL gate (DR-012): proposal surfaced for review; write happens only on explicit approval; idempotent; artifacts with an existing `epic:` left untouched unless overridden |
| Weak Tier-0 taxonomy guesses mislead the user | Heuristic engine has no semantic grasp of spec content (the reason AI-only was kept as fallback in Alternatives) | Proposal carries per-artifact rationale/confidence; the human reviews before any write, so a weak guess is rejected at review, not committed |
| Marketplace/docs overstate capability | Cross-repo copy claims "AI" without noting the Tier-1/`claude` requirement | Consequences §Negative makes the labeling a tracked follow-up (DR-023); copy must state the heuristic pass works without `claude` |

## Costly to Refactor

Low-to-moderate, mostly reversible. The `BackfillProposal` contract is the load-bearing commitment: once the heuristic engine, AI engine, and `applyBackfill` all consume it (and any other feature adopts the Tier-1 `claude -p` pattern this DR establishes), changing its shape is a multi-site edit. The user-facing decisions — opt-in/offer-only, HITL-before-write, Tier-1 classification of `claude -p` — are cheap to keep but expensive to *reverse* after release: flipping to auto-apply or running AI during first-run would break the DR-004 air-gapped and DR-012 trust promises customers were sold on. **ADR-filter: can this be undone in <1 day?** Yes for the code (delete `lib/epic-backfill.ts`, the `minspec.backfillEpics` command, the onboarding step, and `setArtifactEpic`); no for the trust posture once shipped and documented.
