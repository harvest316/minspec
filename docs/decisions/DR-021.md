# DR-021: Classifier is a ceremony floor, not a difficulty detector

## Context

[DR-009](DR-009.md) validated the tier classifier against SWE-bench-Verified via
out-of-tree fixtures ([SPEC-004](../../specs/minspec/SPEC-004-classifier-validation/requirements.md))
and **deliberately left the direction open** — it measures, it does not decide.
The measurement is now in
([tasks.md Findings](../../specs/minspec/SPEC-004-classifier-validation/tasks.md)) and forces
a product decision.

Strongest evidence (Run C, n=120, 11 repos; ground truth = majority of 3 blind LLM
labellers given only the problem statement + 1 human; **Fleiss κ = 0.80**):

- **34.2% exact, 86.7% adjacent.** The classifier predicts **T1 for 89/120 (74%)**
  while only 25 are truly T1.
- **It never over-tiers.** `pred ≥ T2` → `true ≥ T2` is **100% precise (31/31)**;
  whole-set over-tiering is **1/120**. A predicted tier is a near-perfect *lower
  bound*.
- **It systematically under-tiers subtle small fixes** — 48 true-T2 and 16 true-T3
  collapse to T1.

**Root cause (a design limit, not a bug):** every signal is mechanical
scope / blast-radius (file count, line count, cross-dir spread, file-type
diversity). **Cognitive difficulty is orthogonal to diff size** — a subtle 2-line
fix and a trivial 2-line fix are size-identical, and within the predicted-T1
bucket true-T1 (lines 1..15) and false-T1 (lines 2..20) fully overlap. **No
threshold separates them**, and lowering thresholds only inflates the genuine T1s.

The fork DR-009 left open: given the classifier structurally cannot see difficulty,
how should the product apply and describe a tier?

## Decision

**Ship the classifier as an upward-only ceremony ratchet, and reframe the docs to
say a tier measures mechanical scope — not difficulty.**

1. **Apply the predicted tier as a minimum (floor).** Ceremony ratchets **up,
   never auto-down.** Justified directly by the 100%-precise `pred ≥ T2` lower
   bound: when the classifier says "at least T2," it is right.
2. **Prompt-to-bump-up only.** Surface the predicted tier with a one-click "this
   is harder than it looks → raise tier" affordance. Never silently lower. Leans
   on the existing user-override + calibration path (invariant #5).
3. **Reframe documentation.** State plainly that a tier reflects
   mechanical/blast-radius scope and **does not detect cognitive difficulty**.
   Remove any copy implying the tool gauges how hard a change is.
4. **Drop AST-complexity augmentation (validation option 2).** Proven dead end:
   the 64 misses are median 1 file / 5 lines — there is no *local* complexity to
   measure, so AST would score them low too.
5. **Defer semantic/NLP difficulty scoring (option 3) to opt-in.** Reading the
   problem statement is the only signal that could close the gap, but it is
   near **invariant #1 (no-AI)**. It ships, if ever, as an **opt-in Tier-1+
   feature** behind AI consent — never in the Tier-0 core. Parked, not built.

## Costly to Refactor

- **Upward-only ratchet direction** (floor; never auto-down, Decision 1) — the floor is sound *only* because `pred ≥ T2 → true ≥ T2` is 100% precise (31/31, Context); reversing to allow auto-down re-opens the ceremony-safety argument, since the classifier under-tiers 64/120 and a downward step would land on a real false-T1. *Check: Decision 1 floor preserved; over-tiering stays 1/120.*
- **The "tier = scope, not difficulty" reframe** (Decision 3) — public copy/positioning per the `packages/minspec/src/lib/classifier.ts` floor + docs/marketing refs; walking it back resurrects the "34% accurate" framing this DR replaced with the 100%-precise floor claim. *Check: no doc copy implies the tool gauges difficulty (Decision 3).*

## Consequences

**Positive:**
- **Ships now, no-AI.** Invariants #1 (no AI) and #2 (network) untouched; pure
  analyzer path.
- **Honest positioning.** The reframe removes a promise the classifier cannot
  keep, and converts a "34% accurate" headline into a true claim: a 100%-precise
  *floor*.
- **Sound floor.** Guaranteed by the 31/31 (`pred ≥ T2 → true ≥ T2`) precision and 1/120 whole-set over-tiering (Context): MinSpec never under-ceremonies relative to its own prediction, so the floor (Decision 1) is correct by construction, not by tuning.

**Negative:**
- Subtle small fixes still land at T1 unless a human/agent bumps them — the tool
  will not catch them automatically.
- Marketing cannot claim "detects hard tasks"; the difficulty story is deferred.

**Risks accepted:**
- Users may still read the floor as difficulty detection. Mitigated by the
  reframed docs + the explicit bump-up prompt, not by tooling alone.

## Alternatives considered

- **Tune size thresholds.** Rejected — wrong axis (orthogonal to difficulty);
  only inflates genuine T1s.
- **AST complexity signals (option 2).** Rejected — dead end; no local complexity
  in the misses.
- **Ship semantic difficulty scoring now.** Rejected — reads issue text, collides
  with invariant #1; viable only as opt-in.
- **Keep current framing, change nothing.** Rejected — the framing over-promises
  difficulty detection the classifier structurally cannot deliver.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| 1 | **Floor read as difficulty detector** — users trust T1 as "easy" and skip thinking. | Med · Med | Reframed docs state tiers = scope, not difficulty; bump-up prompt makes raising tier one click. Decision 3. |
| 2 | **Bump-prompt fatigue** — a "raise tier?" nudge on every classify becomes noise. | Med · Low | Show only at the boundary (predicted-T1 with a plausible-harder signal), respect the advisory-not-blocking posture; dismissible like every MinSpec nag. |
| 3 | **Deferred semantic feature never ships** — the real gap (subtle small fixes) stays open indefinitely. | Med · Low | Accepted: the floor is correct and useful without it. Tracked as a parked opt-in path, not a regression. |
| 4 | **Evidence is single-corpus** — SWE-bench-Verified is all single-PR fixes (T4=0); the floor is unvalidated for architectural tiers. | Low · Med | Claim is scoped to T1–T3; the ratchet only asserts a *lower* bound, which a T4-blind corpus cannot falsify. Note the limit in docs. |
