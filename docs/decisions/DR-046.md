# DR-046: SealBox obeys rule #8 — worktree isolation + symmetric base-freshness

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records the root cause + the invariants; mandates the SPEC-019 edits (some applied
> with this DR, the rest Plan-phase). **No code** — SPEC-019 is a Specify-phase record.

## Context

[SPEC-019](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md)'s **FR-13**
hands the agent's branch out as a diff and has the credentialed control plane push it
**after the agent exits**. Its one concurrency guard is a *creation-time* sub-bullet: branch
off `origin/main` (fetched parent-side), never the stale local `main`. The session question:
SealBox does not run in a vacuum — concurrently the human **merges PRs** (`origin/main`
advances), **edits `main` directly**, and **other Claude Code sessions work in sibling
worktrees** on the same `.git`. Does FR-13 keep SealBox from getting confused by — or
*causing* — changes under its feet?

A 7-lens adversarial workflow (`wf_e2dd93db-888`: one race-hunter per concurrent-mutation
scenario, each finding then independently refuted) confirmed **7 races, every one
integrity-class (never-wrong)** — i.e. each makes SealBox emit a *factually-wrong result* or
*corrupt/strand the user's working tree*, not merely degrade safely. The maintainer **hit
race #1 this very session**: a DR push was rejected non-fast-forward because a PR merged
mid-flight, forcing a hand-rebase — which an unattended agent cannot do.

## Root cause (RCDD)

**Mechanism — a validator asymmetry, the recurring class
([validator-asymmetry](project_validator_asymmetry_class): SPEC-004 `epic:`, #115, #126,
DR-032 egress).** FR-13 gates base-freshness in **one direction only**: it asserts the base
is fresh at *creation* and never re-asserts it at *push*. `origin/main` is a **moving remote
ref**, and runs are unbounded (minutes–hours), so the exact "factually-wrong output" failure
FR-13's creation rule was written to kill **re-enters at exit**, where neither FR-13
(creation) nor CL-10 (pre-dispatch) looks.

**Second mechanism — unspecified locus.** FR-13 names the base *ref* but never the working-
*tree location* nor any prohibition on the control plane touching the user's shared checkout.
Layer-2 is safe by construction (container, host `$HOME` unmounted, FR-6). **Layer-1 ships
first (CL-1) and has no fs isolation** — so "branch off `origin/main`" is satisfiable by
`git checkout -b` *inside the user's primary checkout*, or by the spawned subprocess cwd
defaulting there ([bug #83](feedback_worktree_agent_bash_cwd_leak)). That moves the shared
HEAD and strands the human's edits — the rule-#8 corruption SealBox is contractually bound to
obey, with **SealBox as the culprit**.

**Missing gate.** No invariant binds SealBox — itself a concurrent automated git actor — to
the rule-#8 discipline the human follows; and there is no exit-time base re-validation gate.
A data/prose fix (e.g. "remember to rebase") would be the incomplete-RCDD tell; the gate must
be **enforce-by-construction** (CL-12).

## Decision

**1 — The load-bearing invariant (every lens converged on it).**

> **INV (T0) — SealBox obeys rule #8 as a product invariant.** SealBox is a concurrent
> automated git actor: every dispatched agent (L1 and L2) operates in a **dedicated worktree**
> rooted *outside every checkout* (`~/code/.worktrees/<repo>/sealbox-<runId>/` — never nested
> in a working tree, never `/tmp` where the dir vanishes but the `.git` worktree record
> survives). No SealBox code path mutates the user's primary checkout, moves the shared HEAD,
> force-pushes, or forces/deletes a ref a peer session may hold. Shared `.git` is left as
> found.

**2 — Make base-freshness symmetric (the missing exit-time half) + a safe push protocol.**
Six additive requirements close the 7 races; none adds cred/egress to the sandbox, moves push
into the box, or adds product surface:

| # | Requirement | Closes |
|---|---|---|
| a | **Pin the base as a SHA** (`rev-parse FETCH_HEAD` at spawn, create the exec context off that exact SHA), store `baseSha` on the OutcomeStore record (CL-4), inject it into the agent's DATA framing as a caveat ("tree pinned at `<sha>`; do not assert non-existence of what you cannot see"). | atomic snapshot vs concurrent fetch/merge |
| b | **Re-validate at push** — re-fetch `origin/main`; if advanced past `baseSha` → rebase **in-worktree** → re-run the gate + recompute the diff range/`changedFiles` on the fresh base; on conflict/overlap → **fail-soft to `needs-review`, never push/comment blind**. | base drift (Face 1) |
| c | **Push protocol** — per-dispatch-**unique** branch (`sealbox/<issue>-<ulid>`), **create/fast-forward-only, never `--force`/`--force-with-lease`/reset** of a remote ref. Enforce-by-construction: the push wrapper has no force code path (grep-gate in review, mirroring R7's cred grep). | concurrent-push clobber / silent revert |
| d | **cwd discipline** — every parent-side git op uses `git -C <worktree>` + an **explicit refspec** (never a bare `git push`, never `branch -f`/reset on the shared HEAD); verify the primary checkout's branch **NAME** == intended *and* HEAD SHA unmoved **before and after** handoff (name-check, not SHA-only — a SHA guard misses a concurrent `git switch`). Mismatch → STOP, typed `{ok:false,reason:'checkout-moved'}`. | bug #83 + locus (Face 2) |
| e | **Lifecycle** — `teardown` is **defined for L1** (`git worktree remove --force` + `git branch -D <tempBranch>` on *every* terminal exit, in a finally/trap, after diff+push). Extend CL-7 orphan reclamation from run-*records* to git-*state*; re-queue mints a **fresh** worktree off a **fresh** fetch (never reuse a stale orphan → reintroduces a stale base). | orphan-worktree leak / stale-reuse |
| f | **Typed reasons** (FR-11) — add `base-advanced`, `base-advanced-conflict`, `git-lock-contention`, `checkout-moved`; treat git-lock contention (`index.lock`/`packed-refs.lock`/worktrees lock) as retryable→fail-closed, never dispatch into a half-init worktree. CL-5 rejectable set += `stale-base`. | clean degrade, lock races |

**3 — `--force-with-lease` vs no-force, resolved: no force.** One race-hunter suggested
`--force-with-lease` on the agent's own branch; another said never force at all. With
per-dispatch-**unique** branches (2c) plus rebase-in-worktree-then-push-fresh, a force is
**never required** — so the push primitive carries no force path whatsoever. The stricter rule
wins; it is also the never-wrong-safe one.

## Alternatives considered

- **Leave it to Plan.** Rejected: the worktree-isolation + symmetric-freshness contract is a
  **T0 invariant on a T4 security substrate**, not undoable in <1 day; retrofitting a
  fail-closed/never-force gate after dispatch is wired is the costly path (mirrors FR-6's
  fail-closed Costly-to-Refactor entry). Record it now.
- **Container-only isolation (skip the L1 worktree rule).** Rejected: **L1 ships first**
  (CL-1) with no fs isolation, so the worktree IS L1's only isolation — the gap is live in
  v1, not deferred to the Layer-2 milestone.
- **Allow `--force-with-lease` on the agent branch.** Rejected (decision 3): unique branches
  make force unnecessary; a force code path is a standing clobber/regression surface.
- **Prose-only "rebase before push" reminder.** Rejected: the incomplete-RCDD tell
  (DR-003 data-only-fix corollary). The gate must be enforce-by-construction (CL-12).

## Consequences

- **Positive:** SealBox can neither be *confused by* concurrent mutation (pinned base +
  exit-time reconcile) nor *cause* corruption (worktree-only, never touches the shared
  HEAD/checkout) — both halves of the maintainer's worry, closed at the invariant level. The
  asymmetry root cause is named, so the symmetric gate covers both directions.
- **Cost (owed) — SPEC-019 edits.** Applied **with this DR** (spec is pre-approval, editable
  in place): the new T0 INV, the two FR-13 sub-bullets (creation pin+locus / exit
  reconcile+push+cwd), the FR-11 typed reasons, and risk **R10**. Deferred to **Plan**: the
  CL-4 (`baseSha` on OutcomeStore) / CL-5 (`stale-base`) / CL-7 (git-state GC) refinements,
  the FR-2 `execContext`-required parameter, and the T0 tests. All fold into the SPEC-019 →
  SealBox migration cascade (DR-044).
- **No code.** SPEC-019 is unbuilt; FR-16 / Tier-0 untouched. No cred/egress added; push stays
  parent-side; no new product surface.

  `--force`/reset ever invoked; (iii) simulated crash between commit and teardown → reclamation
  sweep leaves zero orphan worktrees/branches.
