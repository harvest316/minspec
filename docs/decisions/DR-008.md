## Context

`scripts/dispatch-issue.sh` runs a headless `claude -p` agent against a GitHub issue, in a git worktree, and (on success) commits + pushes a branch and comments on the issue. The goal is "full steam": a loop/cron driver that auto-dispatches every `agent-ready` issue with no human in the loop.

Two automated commit security reviews flagged the dispatch path:

1. The agent ran with `--permission-mode bypassPermissions` while the issue body was concatenated raw into the prompt — a prompt-injected issue could drive any tool. (Fixed: `<untrusted_issue_body>` delimiters + scoped allowlist.)
2. The scoped allowlist was itself a semantic escape: `Bash(node:*)`, `Bash(npx:*)`, `Bash(npm:*)`, `Bash(cat:*)`, `Bash(git:*)` each permit arbitrary code execution or credential/file exfiltration (`node -e`, `npx <pkg>`, a tampered `npm run` script, `cat ~/.config/*`, `git push`/`git config`).

The unavoidable conclusion: **a tool allowlist cannot sandbox a dev agent.** An agent that runs the project's own build and tests is executing arbitrary code by definition — the test files are code the agent just wrote, and `package.json` scripts are mutable by the agent. No allowlist that permits `npm test` can also prevent arbitrary execution.

The agent also runs in the container's environment, which (per the platform CLAUDE.md) holds live credentials: a `gh` token, `CLOUDFLARE_API_TOKEN`, FTP creds (`lftp`), `psql -d mmo` access, wrangler home. A compromised agent has direct reach to all of them.

## Decision

Separate the control into two independent layers, and gate unattended operation on the second.

### Layer 1 — applied now (defense-in-depth, not a boundary)

- Issue body wrapped in `<untrusted_issue_body>` with an explicit "data, not instructions" preamble.
- Agent allowlist limited to file tools + fixed `npm` subcommands (`test`/`validate`/`lint`/`build`/`ci`) + **local-only** git (`add`/`commit`/`status`/`diff`/`log`). No `node`, `npx`, `cat`, `gh`, or `git push`/`remote`/`config`.
- **Credentialed and network operations removed from the agent entirely.** The agent commits locally and writes `.agent-summary.md`. The dispatcher (parent process) does the `git push` and `gh issue comment` *after* the agent exits. The `gh` token is never in the agent's reach.

This is explicitly labelled defense-in-depth. It reduces casual escape and removes the highest-value credential (the gh token) from the agent, but it does **not** make an injected agent safe — `npm run <script>` still executes editable code.

### Layer 2 — required before unattended ("full-steam") dispatch

Unattended auto-dispatch (a loop/cron over `agent-ready` with no human approving each run) is **NOT permitted** until the agent executes inside an isolation boundary with **no host credentials in its environment**:

- Container/VM/sandbox with its own clean environment — no `gh` token, no `CLOUDFLARE_API_TOKEN`, no FTP/psql/wrangler creds, no access to `~/.config`, `~/.claude`, `~/.cache/wrangler-home`, or the host keychain.
- Network egress denied by default (or allowlisted to the package registry only).
- The branch leaves the sandbox as a diff/bundle; the credentialed host process reviews and pushes it.
- A human (or a trusted, non-agent gate) approves the merge — the existing PR-branch + `needs-review` flow for T3-T4 already provides this for higher-tier work.

Until Layer 2 exists, `dispatch-issue.sh` is **human-initiated only**, and only on issues whose body is trusted (self-authored). This is consistent with the existing tier-gated HITL: T3-T4 already require human review; this DR extends the same caution to the *execution environment* for all tiers when unattended.

## Consequences

### Positive
- The gh token — the most directly abusable credential — is out of the agent's reach today, without waiting for containers.
- The full-steam blocker is named and recorded, so it can't be skipped by accident when wiring the loop.
- Layer 1 / Layer 2 split lets validation and PR-branch automation proceed now while the dangerous part (unattended) waits.

### Negative
- Manual dispatch only, for now — the originally-desired auto-loop is deferred until isolation is built.
- Building Layer 2 (clean-env container, egress policy, diff-export handoff) is real work not yet scoped.
- Parent-side push/comment means the agent can't link its own PR mid-run; the dispatcher composes the comment from `.agent-summary.md`.

### Risks accepted
- Layer 1 allowlist still permits arbitrary code, because its fixed `Bash(npm run validate)`/`lint`/`build` entries (`scripts/dispatch-issue.sh:130`) execute editable `package.json` scripts the agent can rewrite. **Accepted only because** dispatch stays human-initiated on trusted issues until Layer 2. Not acceptable for untrusted input or unattended runs.

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| Operator wires the loop/cron anyway, treating Layer 1 as sufficient | The fixed `Bash(npm run validate)`/`lint`/`build` entries (`scripts/dispatch-issue.sh:130`) map to editable `package.json` scripts, so even without a wildcard `npm` entry an injected `agent-ready` body can rewrite e.g. the `validate` script and exfiltrate when the agent runs it (Context §"unavoidable conclusion": package.json scripts are mutable by the agent) | Layer 2 is recorded as a hard gate (Decision §"Layer 2"); `dispatch-issue.sh` stays human-initiated on self-authored issues only until the clean-env sandbox + egress policy exist |
| Allowlist drift re-admits a credentialed/escape tool | Re-adding `node`, `npx`, `cat`, `gh`, or `git push`/`remote`/`config` to the allowlist re-opens the exact escapes catalogued in Context §2 (`node -e`, `cat ~/.config/*`, `git push`) | Decision §"Layer 1" enumerates the closed list explicitly; `scripts/triage-inbox.sh` is the prior-art reference for the narrow allowlist shape |
| Parent-side push/comment mishandles agent output | The dispatcher composes the `gh issue comment` from `.agent-summary.md` *after* the agent exits; a malformed or injected summary file could carry attacker-chosen text into the comment | Summary is data the parent renders, not instructions; the credentialed `gh`/`git push` actions run in the parent only after the agent process has exited, so the agent never holds the token |
| Layer 2 egress policy conflicts with tiered network consent | Layer 2 denies/allowlists network egress, which must compose with DR-004's tiered network consent rather than contradict it | Tracked as a dependency in Related → DR-004; egress allowlist scoped to the package registry only |

## Alternatives Considered

- **Harden the tool allowlist alone (no isolation boundary).** Rejected: Context §"unavoidable conclusion" proves a tool allowlist cannot sandbox a dev agent — any allowlist permitting `npm test` permits arbitrary execution, because the test files and `package.json` scripts are code the agent itself just wrote. Tightening the list is defense-in-depth (Layer 1), never a boundary.
- **Keep `gh` push/comment inside the agent and rely on prompt-injection delimiters.** Rejected: the `<untrusted_issue_body>` delimiters reduce casual escape but do not make an injected agent safe; leaving the `gh` token in the agent's environment keeps the highest-value credential one editable `package.json` script away from exfiltration (the allowlisted `Bash(npm run validate)`/`lint`/`build`, `dispatch-issue.sh:130`). Hence push/comment moved to the parent (Decision §"Layer 1").
- **Allow unattended dispatch now, gated only on the existing T3-T4 `needs-review` merge approval.** Rejected: `needs-review` gates the *merge*, not the *execution environment*. An unattended agent with host credentials (`CLOUDFLARE_API_TOKEN`, FTP, `psql -d mmo`, wrangler home) can do damage before any merge review. This DR deliberately extends the caution to the execution environment for all tiers when unattended.

## Costly to Refactor

Moderate to reverse, asymmetric by layer. Layer 1 is cheap to undo (<1 day): the allowlist and the parent-side push/comment split live entirely in `scripts/dispatch-issue.sh`. The expensive, effectively irreversible commitment is the Layer 2 gate: once unattended dispatch is built on a clean-env sandbox with diff/bundle export and a denied-by-default egress policy, relaxing it back to "agent runs in the credentialed host env" would re-expose every host credential listed in Context §"the agent also runs in the container's environment" and cannot be undone in <1 day. ADR-filter: the Layer 1 split is reversible in <1 day; the Layer 2 isolation harness (clean-env container, egress policy, diff-export handoff — noted as "real work not yet scoped") is not, which is why this is recorded as a DR rather than done silently.
