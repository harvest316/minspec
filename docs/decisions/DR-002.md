## Context

Two extensions share a classification engine. Specs, docs, and decisions live at project level. Need a structure that supports both extensions without duplication.

## Decision

Single monorepo at `harvest316/minspec` using npm workspaces.

Packages:
- `packages/minspec` — VS Code extension A
- `packages/scroogellm` — VS Code extension B
- `packages/shared` — Classification engine + contracts (shared by both)
- `packages/extension-pack` — MinSpec Pro (references both)

## Rationale

- npm workspaces is built-in (zero added deps)
- Two extensions + one shared lib is simple enough — no Turborepo/Lerna needed
- Extensions are vsix packages, not npm publishes → no publish orchestration needed
- One issue tracker, one CI pipeline, shared specs/docs

## Consequences

- `packages/shared` must be built before either extension
- vsix packaging is per-extension (`npm run package` in each)
- Workspace hoisting means devDeps in root, but `@vscode/vsce` stays per-package
