## Context

Two extensions share a classification engine. Specs, docs, and decisions live at project level. Need a structure that supports both extensions without duplication.

## Decision

Single monorepo at `harvest316/minspec` using npm workspaces.

Packages:
- `packages/minspec` — VS Code extension A
- `packages/scroogellm` — VS Code extension B
- `packages/shared` — Classification engine + contracts (shared by both)
- `packages/extension-pack` — MinSpec Pro (references both)

## Rationale

- npm workspaces is built-in (zero added deps)
- Two extensions + one shared lib is simple enough — no Turborepo/Lerna needed
- Extensions are vsix packages, not npm publishes → no publish orchestration needed
- One issue tracker, one CI pipeline, shared specs/docs

## Costly to Refactor

- **The 4-package split + names** (`packages/minspec|scroogellm|shared|extension-pack`) — renaming or re-bounding a package later means updating every import, marketplace ID, and CI path. *Check: package names + boundaries settled before publishing.*
- **npm workspaces as the build tool** — switching to pnpm/turbo later is a re-tooling. *Check: workspaces meets needs at scale.*

ADR-filter answer: **Not undoable in <1 day.** Splitting a package out or swapping the build tool touches every import path, marketplace/extension ID, and CI path (per the two bullets above) — DR-027's `packages/scroogellm` split to `harvest316/scroogellm` is the worked example of how much moves when one package leaves. This DR earns its place in the register.

## Consequences

**Positive:**
- One issue tracker, one CI pipeline, and shared `specs/`/`docs/` across both extensions (no cross-repo sync of the decision register or specs).
- `packages/shared` (the classification engine + contracts, later formalised Tier-0 per DR-014) is imported directly by both extensions — no `npm publish` round-trip to consume it.
- Zero added build tooling: npm workspaces ships with npm, so no Turborepo/Lerna config to maintain.

**Negative:**
- Build ordering is load-bearing: `packages/shared` must be built before either extension, so CI/build scripts must encode that dependency.
- vsix packaging stays per-extension (`npm run package` in each) — there is no single "package the repo" command.
- Workspace hoisting puts devDeps in the root `node_modules`, but `@vscode/vsce` must stay per-package, so the hoisting model isn't uniform and has to be reasoned about per-tool.

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| A package needs to leave the monorepo, breaking the "no duplication" premise | `packages/scroogellm` was in fact split out to the private `harvest316/scroogellm` repo per DR-027 — the shared-engine import relationship had to be re-cut at that boundary | Keep `packages/shared` free of vscode/network deps (Tier-0, DR-014) so it can be vendored/republished to a split-out repo without dragging extension code along |
| npm workspaces proves too weak at scale, forcing a re-tooling | "Costly to Refactor" already flags pnpm/turbo migration as expensive | Defer the switch until a concrete pain (slow installs, cache misses) appears; the two-extension + one-lib shape (per Rationale) is below the threshold where Turborepo/Lerna pay off |
| Build-ordering footgun: an extension builds against a stale `packages/shared` | Consequences notes `shared` must be built first; hoisting masks which copy is linked | Encode the `shared`-before-extensions order in CI build scripts; treat a stale-shared build as a build break, not a warning |

## Alternatives Considered

- **Turborepo / Lerna / pnpm workspaces** — rejected: the Rationale states two extensions + one shared lib is "simple enough — no Turborepo/Lerna needed", and these add a dependency and config surface for caching/orchestration this repo does not yet need (see also the "npm workspaces as the build tool" entry under Costly to Refactor).
- **Separate repos per extension with a published shared package** — rejected: extensions are vsix packages, not npm publishes, so there is no publish orchestration to amortise; separate repos would also fragment the single issue tracker, CI pipeline, and shared `specs/`/`docs/` that the Decision keeps together. (Note: this *was* later partially adopted for scroogellm via DR-027, but driven by privacy, not build needs.)
