# DR-013: Registered epics as a first-class grouping dimension

## Context

MinSpec tracks three artifact kinds that all describe the same underlying work
but are siloed in the UI:

- **Specs** (`specs/**/*.md`, `id: SPEC-NNN`) — what to build.
- **ADRs** (`docs/decisions/DR-NNN.md`) — why we chose an approach.
- **Issues** (GitHub, label-driven via `backlog.ts`) — discrete units of work.

The Traceability Convention already links these one-to-one (commit → issue →
DR → spec), but there is no **grouping** dimension above the individual artifact.
A body of work like "telemetry & RUM" today spans DR-010, DR-011, several specs,
and N issues with nothing tying them into one visible bucket. The explorer panels
(`spec-tree-provider`, `adr-tree-provider`, `backlog-view`) each list their own
artifact kind flat, so a user cannot see "everything under epic X" in one place.

A flat freeform `epic: some-string` was considered and rejected: typos silently
fragment groups, and there is nowhere to hang an epic's own title, status,
ordering, or completion roll-up.

## Decision

Introduce **registered epics** — a small new artifact type that the other three
reference, plus an epic grouping layer in all three explorer panels.

### 1. Epic registry — `docs/epics/EPIC-NNN.md`

One file per epic, sequential from `EPIC-001`, frontmatter:

```yaml
---
id: EPIC-001
slug: telemetry          # short kebab handle; used for the GitHub issue label
title: Telemetry & RUM
status: active           # proposed | active | done | abandoned
order: 1                 # explorer sort key; lower = higher
---
```

Body is freeform prose describing the epic. An `INDEX.md` is generated alongside,
mirroring the ADR register pattern (marker-bounded, DR-011). Next-number =
`max(existing EPIC-NNN) + 1`, computed by a **MinSpec: Create Epic** command
mirroring **Create ADR**.

### 2. Artifact → epic reference

- **Specs & ADRs:** optional `epic:` frontmatter field. Accepts either the id
  (`EPIC-001`) or the slug (`telemetry`); both resolve to the same registry entry.
- **Issues:** optional GitHub label `epic:<slug>` (e.g. `epic:telemetry`),
  consistent with the existing `wsjf:N.NN` / lifecycle / priority label
  conventions in `backlog.ts`. No new GitHub primitive (no Milestones) — labels
  only, so this stays detection-grade, not a GitHub integration.

A single `resolveEpic(ref)` in a new `epic-manager.ts` is the one place that maps
id-or-slug → registry entry, used by every consumer.

### 3. Explorer grouping

Each of the three panels gains an **epic grouping layer** as the top tree level,
sorted by epic `order` then id:

```
▼ Telemetry & RUM (EPIC-001)        3/7
   DR-010, DR-011
   SPEC-003
   #41, #42, #43
▼ (no epic)
   …ungrouped artifacts…
```

Grouping is a **view toggle**, default on; flat view remains available. Artifacts
with no/invalid epic ref collect under a synthetic "(no epic)" group — never
hidden.

### 4. Validation

`spec-validator` / ADR validation gains a soft rule: an `epic:` ref that does not
resolve to a registry file is a **warning** (not a hard block) — surfaced as a
diagnostic and a "(no epic)" placement. Keeps ceremony proportional: an unknown
epic never blocks work, it just nags.

## Consequences

**Positive:**

- One visible bucket for a body of work that today scatters across multiple
  artifacts — e.g. "telemetry & RUM" spanning DR-010, DR-011, several specs and
  N issues now collects under `EPIC-001` in all three explorer panels.
- `SpecFrontmatter` and `AdrFrontmatter` gain an optional `epic?: string`, so the
  change is backward compatible — an absent field just means ungrouped, no
  migration of existing specs/ADRs required.
- Epic is an orthogonal grouping axis layered *above* the one-to-one traceability
  chain (commit → issue → DR → spec) — that chain is untouched, so grouping adds
  reach without re-plumbing the existing links.
- The registry (`EPIC-NNN.md` + generated `INDEX.md`, marker-bounded per DR-011)
  gives each epic its own title/status/order/roll-up — a hanging point a freeform
  string could never provide.

**Negative:**

- New artifact type + `docs/epics/` directory + generated INDEX that scaffold
  (`minspec init`) must now create and maintain — a fourth artifact kind to keep
  in lockstep with the other three.
- Three tree providers (`spec-tree-provider`, `adr-tree-provider`, `backlog-view`)
  restructured from flat → two-level — the most surface area of the change, and
  the part most likely to regress.
- Frontmatter completion + templates must learn the `epic:` field, and a new
  `resolveEpic(ref)` / `epic-manager.ts` becomes a shared dependency every
  consumer (validation, all three panels) routes through.

## Alternatives considered

- **Freeform `epic:` string** — rejected (typos fragment groups, no epic
  identity/order/roll-up).
- **GitHub Milestones for issue grouping** — rejected for v1: introduces a real
  GitHub write-integration surface and a second source of truth; the `epic:<slug>`
  label reuses the existing label-convention machinery with zero new integration.
- **Single combined "work item" tree** merging all three kinds — rejected:
  collapses the deliberate spec/ADR/issue distinction the methodology depends on.

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| Id/slug drift between the two reference styles | Specs/ADRs accept either `EPIC-001` or `telemetry`; issues carry `epic:<slug>` labels in `backlog.ts` — two spellings of the same epic | `resolveEpic(ref)` in `epic-manager.ts` is the single mapper for id-or-slug → registry entry; every consumer routes through it, so there is one place to keep the resolution consistent. |
| Dangling `epic:` refs silently fragment groups | A spec/ADR pointing at an `EPIC-NNN` with no registry file would otherwise vanish from any group | Section 4 makes an unresolvable `epic:` a **warning** (not a hard block) in `spec-validator`/ADR validation, and the artifact lands in the synthetic "(no epic)" group — never hidden. Note this is the *missing-ref* direction; per DR-003 Phase 4 asymmetry check, validation must also not assume an `epic:` should always exist (it is optional). |
| Two-level tree regression in the explorer | Restructuring `spec-tree-provider`, `adr-tree-provider`, `backlog-view` flat → two-level is the largest surface area (see Consequences) | Grouping ships as a **view toggle, default on**; flat view remains available as a fallback if grouping misbehaves, so a regression degrades to the prior flat UX rather than breaking the panel. |
| Epic numbering collision | Hand-picking an `EPIC-NNN` could reuse a number, like the DR out-of-sequence hazard called out in project CLAUDE.md | Next-number is computed as `max(existing EPIC-NNN) + 1` by the **MinSpec: Create Epic** command (mirrors **Create ADR**), not chosen by hand. |

## Costly to Refactor

Moderate blast radius, but mostly additive. The cheap-to-reverse parts: `epic?:`
is optional on `SpecFrontmatter`/`AdrFrontmatter` and `epic:<slug>` is just
another label in `backlog.ts`, so dropping the feature leaves all existing specs,
ADRs and issues valid (absent/ignored field). The expensive-to-undo part is the
explorer: once the three tree providers are restructured flat → two-level and
users rely on epic grouping as the default top level, reverting means re-flattening
three panels and retraining muscle memory — and the `docs/epics/` registry + its
generated `INDEX.md` become real artifacts other DRs/specs may start citing.
ADR-filter answer: **No** — not undoable in <1 day once the tree restructure ships
and epics are referenced across artifacts; hence this DR (T3) rather than a silent
change.
