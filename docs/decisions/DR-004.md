## Context

MinSpec extension has invariant #2: "No backend — zero network calls, no accounts, no telemetry, all local." This binary rule is already being bent:

- The extension shells out to `gh` CLI for GitHub issue creation (parking lot feature from DR-360).
- The MinSpec Pro extension pack adds ScroogeLLM, which is inherently network-dependent.
- Agent dispatch (`claude --bg`, `dispatch-issue.sh`) needs network access.
- Usage analytics could inform product decisions but are impossible under a strict no-network rule.

A binary "no network" invariant no longer reflects reality. The team wants to replace it with a tiered consent model that preserves the core offline story while allowing network features to grow cleanly.

## Decision

Replace invariant #2 with a three-tier network consent model.

### Tier 0: Fully Offline (Core MinSpec)

- Classification, specs, lifecycle, UI, harness generation — zero network calls.
- Works air-gapped, no auth, no accounts.
- This is the free extension's default and primary selling point.
- No code path in `packages/minspec` or `packages/shared` may import `http`, `https`, `fetch`, or any networking module.

### Tier 1: Local Tool Delegation (Opt-in)

- GitHub integration via `gh` CLI (issues, PRs, labels).
- Agent dispatch (`claude -p` per issue, roles, triage) is Tier 1, but it does
  **not** ship inside MinSpec. Per DR-015 it ships as a separate third "Execute"
  extension (`aiclarity.agent-execute`); the `scripts/` bash tooling remains the
  dev-time path. MinSpec itself stays Tier 0 / air-gapped.
- Extension delegates to locally-installed CLI tools — no network code in the extension itself.
- Requires: `gh` CLI installed and authenticated, Claude Code installed.
- Clearly labeled as optional features in UI and docs.
- Failure mode: features degrade gracefully when CLI tools are absent (show "install gh to enable" prompts, never error).

### Tier 2: Network Services (MinSpec Pro / ScroogeLLM)

- ScroogeLLM proxy (localhost by default, remote opt-in per invariant #11).
- Future: usage analytics, telemetry (opt-in only, never default-on).
- Requires MinSpec Pro extension pack installation.
- Separate privacy policy, explicit consent flow before any data leaves localhost.

## Rationale

- Preserves the core MinSpec privacy/offline story for the marketplace — the free extension genuinely works air-gapped.
- The delegation model (Tier 1) means `gh` and `claude` handle their own auth and networking; the extension process itself makes zero outbound connections.
- MinSpec Pro becomes the natural upgrade path when network-dependent features are needed.
- Explicit tiers make marketplace listing, documentation, and privacy policy unambiguous.

## Consequences

**Positive:**

- Tier 0 keeps the air-gapped story honest and enforceable: a single, mechanical code-review rule (no `http`/`https`/`fetch`/`net` imports in `packages/minspec` or `packages/shared`) now stands in for the vague "no backend" invariant #2 — easy to grep, easy to gate in CI.
- Network features (`gh` CLI parking lot from DR-360, agent dispatch via the `agent-execute` extension per DR-015, the ScroogeLLM proxy's "localhost by default, remote opt-in" path from the Tier 2 Decision clause) get a sanctioned home (Tier 1 / Tier 2) instead of silently bending invariant #2 (the tiered-consent rule this DR rewrites).
- The delegation model means Tier 1 adds zero outbound connections from the extension process itself — `gh` and `claude` own their auth/networking — so the Tier 0 marketplace claim survives even as Tier 1 features ship.
- Marketplace listing, docs, and the Pro privacy policy can state tier boundaries unambiguously instead of explaining why a "no network" extension shells out to `gh`.

**Negative:**

- Invariant #2's authoritative text in `specs/minspec/requirements.md` must be rewritten from the old blanket "no network" to this tiered model, and every place that paraphrases it (the Context's quoted binary rule, the marketplace/Pro privacy copy) re-stated to match; leave any copy on the old wording and it contradicts this DR while still reading as a live constraint.
- Tier 1 features carry a permanent UX tax: each must probe CLI availability at activation and degrade gracefully (e.g. "install gh to enable") rather than error — more code paths and more failure modes than a pure-offline feature.
- The Pro extension pack now requires a separate, maintained privacy policy covering Tier 2; an empty or stale policy is a compliance hole that core MinSpec never had.
- "Three tiers" is a classification surface a contributor can get wrong — placing a feature in the wrong tier (e.g. a network call slipped into a "Tier 1" delegation) reintroduces exactly the offline-claim breakage this model exists to prevent.

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| Tier 0 boundary silently broken | A future PR adds an `import https` (or `fetch`) into `packages/minspec` or `packages/shared`, breaking the air-gapped marketplace claim while code review waves it through | Make the boundary a mechanical gate, not a human judgement — grep/lint/CI check for `http`/`https`/`fetch`/`net` imports in those two packages (the same rule named in Consequences), so the violation is un-mergeable rather than review-dependent. |
| Mislabelled "delegation" leaks network into the extension process | The Decision claims Tier 1 has "no network code in the extension itself," but a contributor could call an HTTP API directly instead of shelling out to `gh`/`claude`, then still file it under Tier 1 | Tie the Tier 0/1 lint to the *process*, not the feature label: any networking module in the extension code fails the gate regardless of which tier the feature claims to be. |
| Tier 1 features hard-error when `gh`/`claude` absent | Failure-mode rule in the Decision ("degrade gracefully ... never error") is prose-only; an activation path that assumes `gh` is present will throw for offline/unconfigured users | Require an availability probe at activation per the Consequences rule; cover the missing-CLI path with a test so "show install prompt, don't throw" is enforced, not aspirational. |
| Invariant #2 left stale where it is paraphrased | The rule's authoritative form is `specs/minspec/requirements.md` invariant 2, but its old blanket "no network" wording is also quoted in this DR's Context and will appear in marketplace/Pro privacy copy; rewrite the spec and miss a paraphrase, and the contradicted blanket rule stays live (the cross-doc drift DR-029 flags) | Land the spec rewrite and every paraphrase edit in the change that adopts this DR; the discrepancy is exactly the cross-artifact asymmetry the RCDD gate (DR-003) exists to catch. |
| Tier 2 consent treated as default-on | "usage analytics, telemetry" listed under Tier 2 (Decision §Tier 2) could ship enabled-by-default, violating that section's "localhost by default, remote opt-in" line and its "explicit consent flow before any data leaves localhost" clause | Decision already pins these as "opt-in only, never default-on" + explicit consent flow; the Pro privacy policy (the Negative-consequence policy this DR creates) must document it before any Tier 2 data path ships. |

## Alternatives Considered

- **Keep the binary "no network" invariant #2 unchanged.** Rejected: the Context shows it is *already* violated in production — the `gh` CLI parking-lot shell-out (DR-360), the ScroogeLLM Pro dependency, and `claude --bg` dispatch all make outbound activity. A rule the codebase already breaks is worse than no rule: it makes the marketplace privacy claim a lie rather than a constraint.
- **Drop the offline guarantee entirely and just disclose network use.** Rejected: Tier 0 air-gapped operation is "the free extension's default and primary selling point" per the Decision; discarding it throws away the core differentiator and the honest marketplace story to save a classification step.
- **Two tiers (offline vs. networked), collapsing Tier 1 into Tier 2.** Rejected: it erases the load-bearing distinction that Tier 1 *delegates* to local CLIs (zero outbound connections from the extension process) whereas Tier 2 (ScroogeLLM, telemetry) actually sends data off localhost. Collapsing them would force `gh`-backed features to carry Tier 2's consent-flow and privacy-policy burden they don't need.

## Costly to Refactor

Moderately costly, and asymmetric by tier. Reversing the *model itself* (re-merging tiers or reverting to binary "no network") is cheap on paper but the Tier 0 boundary becomes load-bearing the moment it's published: once the marketplace listing and Pro privacy policy assert "core MinSpec is air-gapped," that promise is made to users, not just to the codebase — walking it back is a trust/compliance event, not a code edit. The mechanical pieces (the import-lint gate, the two invariant-#2 edits, the activation availability probes) are each undoable in well under a day. ADR-filter: the *wording/structure* of the tiers can be undone in <1 day; the *public offline guarantee* it underwrites cannot, so this DR is worth recording rather than just doing.
