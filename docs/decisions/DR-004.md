## Context

MinSpec extension has invariant #2: "No backend — zero network calls, no accounts, no telemetry, all local." This binary rule is already being bent:

- The extension shells out to `gh` CLI for GitHub issue creation (parking lot feature from DR-360).
- The MinSpec Pro extension pack adds ScroogeLLM, which is inherently network-dependent.
- Agent dispatch (`claude --bg`, `dispatch-issue.sh`) needs network access.
- Usage analytics could inform product decisions but are impossible under a strict no-network rule.

A binary "no network" invariant no longer reflects reality. The team wants to replace it with a tiered consent model that preserves the core offline story while allowing network features to grow cleanly.

## Decision

Replace invariant #2 with a three-tier network consent model.

### Tier 0: Fully Offline (Core MinSpec)

- Classification, specs, lifecycle, UI, harness generation — zero network calls.
- Works air-gapped, no auth, no accounts.
- This is the free extension's default and primary selling point.
- No code path in `packages/minspec` or `packages/shared` may import `http`, `https`, `fetch`, or any networking module.

### Tier 1: Local Tool Delegation (Opt-in)

- GitHub integration via `gh` CLI (issues, PRs, labels).
- Agent dispatch via `claude --bg` / `dispatch-issue.sh`.
- Extension delegates to locally-installed CLI tools — no network code in the extension itself.
- Requires: `gh` CLI installed and authenticated, Claude Code installed.
- Clearly labeled as optional features in UI and docs.
- Failure mode: features degrade gracefully when CLI tools are absent (show "install gh to enable" prompts, never error).

### Tier 2: Network Services (MinSpec Pro / ScroogeLLM)

- ScroogeLLM proxy (localhost by default, remote opt-in per invariant #11).
- Future: usage analytics, telemetry (opt-in only, never default-on).
- Requires MinSpec Pro extension pack installation.
- Separate privacy policy, explicit consent flow before any data leaves localhost.

## Rationale

- Preserves the core MinSpec privacy/offline story for the marketplace — the free extension genuinely works air-gapped.
- The delegation model (Tier 1) means `gh` and `claude` handle their own auth and networking; the extension process itself makes zero outbound connections.
- MinSpec Pro becomes the natural upgrade path when network-dependent features are needed.
- Explicit tiers make marketplace listing, documentation, and privacy policy unambiguous.

## Consequences

- Invariant #2 in `CLAUDE.md` and `specs/minspec/requirements.md` must be updated to reference this tiered model instead of the blanket "no network" rule.
- Tier 0 boundary is enforced in code review: no `http`/`https`/`fetch`/`net` imports in `packages/minspec` or `packages/shared`.
- Tier 1 features must check for CLI availability at activation and surface clear UI when tools are missing.
- Marketplace listing documents tier boundaries and optional dependencies.
- MinSpec Pro extension pack privacy policy must cover Tier 2 network behavior.
