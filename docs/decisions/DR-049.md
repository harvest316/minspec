# DR-049: SealBox is public — moat lives in Scrooge, not the boundary

> **Born `proposed` per DR-029** (acceptance is a separate human act). Reverses DR-044 §2
> (private repo). DR-044 itself reserved "flip to public at/with launch if desired" — this
> exercises that option.

## Context

[DR-044](DR-044.md) §2 put SealBox in a **private** repo, mirroring [DR-027](DR-027.md)'s
ScroogeLLM split, to protect "the credential-boundary design as the pre-launch
differentiator." The founder questioned that: SealBox is **hard to monetize directly**, so what
does privacy actually protect? A 3-lens pressure-test (`wf_0a57ce70-32b`: steelman-private,
steelman-public, scrooge-funnel) returned **2 public (high confidence) : 1 private (med)** —
and the private lens **conceded its own core**: *"do not defend privacy on the
credential-boundary code; that case is genuinely weak."*

## Rationale

1. **Security product → open IS the trust wedge.** SealBox's load-bearing claim is FR-6
   attestation ("an agent that physically can't betray you"). By Kerckhoffs, a boundary that
   needs secrecy is a *weak* boundary; the claim is only believable if a buyer's security team
   can **audit the sealed box themselves**. Closed-source undercuts the one positioning that
   differentiates SealBox from the crowded autonomous-engineer lane.
2. **No code moat to protect.** Sandbox primitives (gVisor/Firecracker/devcontainers) are
   commodity; SPEC-019 itself says it "invents nothing." Privacy guards an illusion.
3. **SealBox is a Scrooge funnel — so maximise distribution.** Every model call routes through
   the host-side broker (FR-3), and "installing Scrooge repoints the broker" (FR-4). Public +
   free → more installs → more agent dispatch → more model spend → more for Scrooge (the real
   moat) to monetize. Privacy throttles the exact spend SealBox exists to create.
4. **The moat is one layer away and stays private.** Measurement/routing lives in ScroogeLLM
   ([DR-027](DR-027.md), unchanged). Concentrate secrecy where value actually is.

## Decision

1. **SealBox repo → PUBLIC / open-source.** Supersedes [DR-044](DR-044.md) §2. `harvest316/sealbox`
   flips to public (exercising DR-044's reserved option).
2. **ScroogeLLM stays PRIVATE** ([DR-027](DR-027.md) unchanged). Only the meter is secret; the
   spend-pump in front of it is open.
3. **Protect the funnel *intent*, not the code (the private lens's one valid point).** The
   non-commodity insight is the *strategy* — "SealBox is a demand-generator engineered to feed
   our paid proxy." Keep that **out of public SealBox artifacts**: present the broker as
   **user benefit** (run past your subscription quota; see per-agent cost), **never** as a
   monetization choke point. The funnel-as-strategy framing lives in the **private Scrooge**
   docs, not in the public README/specs.

## Funnel mechanics (recorded)

- Path: public install → agent dispatch → fleet concurrency → subscription-ceiling hit → PAYG
  overage → **routed through Scrooge** (where it's monetized).
- **Caveat — subscription-default bounds the take.** FR-5 defaults to the dev's own Pro/Max
  quota (zero PAYG), so Scrooge only ever monetizes the **overage** past the ceiling. Revenue
  scales with **fleet concurrency**, so SealBox should encourage high-concurrency use
  (cron, multi-issue fan-out); light single-agent use yields ~zero Scrooge pull-through.
- **Competitive risk (fork-and-repoint).** A public SealBox can be forked and its broker
  repointed away from Scrooge — by design (FR-4 is a config flip). **Accept it:** Scrooge's
  moat was never lock-in (it's measurement); secrecy wouldn't prevent the fork anyway. Win by
  being the easy **default** + actually cheaper, not by trapping.

## Consequences

- **Migration scrub is MOOT.** [DR-044](DR-044.md)'s owed public-history scrub of minspec is
  dropped — nothing to hide. Finishing the move = tombstone minspec for tidiness (reversible),
  flip the repo public. No irreversible history rewrite.
- **No change to Scrooge.** DR-027 stands.
