# DR-020: Risks & Mitigations on every spec + DR (depth scales with tier)

> **Reframed by [DR-029](DR-029.md) (2026-06-03).** The section's value is now framed
> as forcing the *LLM* to reason (the human verifies a signal, not the content); the
> tier-proportional depth table + T1 floor below are kept verbatim. See DR-029 §1.

## Context

Specs and DRs were being written without an explicit **Risks & Mitigations**
section — SPEC-012 shipped its first draft without one.

An initial reading gated the section by tier (required only on DRs + T3/T4 specs)
to honour MinSpec's **ceremony-proportional-to-tier** principle. That reading was
**reversed**: the primary value of the section is not the document artifact — it
is **forcing the author (human or LLM) to reason from the risk angle at all**.
That cognitive prompt is *most* valuable exactly where it is easiest to skip:
small, "obvious" T1/T2 changes, where unconsidered risk does the quiet damage.
Omitting it there removes the prompt from the cases that most need it.

The tension with tier-proportionality is resolved not by *who must have it* but by
*how deep it must be*: the section is **always required**, but its **depth scales
with tier** — one line at T1, a full table at T4. Mandatory thinking; proportional
ceremony.

## Decision

**Every spec and every DR MUST carry a Risks & Mitigations section. Its depth is
proportional to tier.**

| Artifact | Required? | Expected depth |
|---|---|---|
| **DRs (all)** | **Required** | Table, alongside Consequences/Alternatives. |
| **Specs T4** | **Required** | Full table; likelihood · impact · mitigation→FR. |
| **Specs T3** | **Required** | Table (the SPEC-012 R1–R7 form). |
| **Specs T2** | **Required** | Short table or 2–4 bullets — the top real risks. |
| **Specs T1** | **Required** | ≥1 line — the single biggest risk + its mitigation. "None material" is an allowed answer, but it must be *written*, forcing the thought. |
| **Epics** | Optional / lightweight | A one-line risk in the Goal suffices; no mandated table. |

The point of the T1 floor: even "no material risk" must be *stated*, not skipped —
the value is the author having looked.

### Shape

A table — `# · Risk · Likelihood · Impact · Mitigation` — each mitigation tracing
to a concrete FR / decision section / invariant (not hand-waving). At T1/T2 a
bullet list or single line is acceptable in place of the full table. The table in
SPEC-012 (R1–R7) is the reference exemplar.

### Enforcement — soft, never blocking

The frontmatter/structure validator gains a **warning** (not a hard block) when a
spec or DR lacks the section, mirroring the existing soft `epic:` unresolved-ref
rule (DR-013 §4). A missing risks section nags; it does not stop work. Consistent
with the advisory-not-blocking posture — the only hard gate is DR-012 approval.

## Costly to Refactor

- **Low — a policy**, now reframed by DR-029 (LLM reasons; human verifies the signal). The only code seam is the validator rule (SPEC-013), itself advisory/reversible. *Check: stays aligned with DR-029.*

## Consequences

- **Templates updated** to emit a Risks & Mitigations stub on **every** spec tier
  (sized to tier) and on the **Create ADR** template — `specify` / `plan` skill
  scaffolds + the ADR command. No tier omits the stub.
- **Validator** gains a soft rule (warn) for any spec or DR missing the section.
- **Backfill** of the section into the 19 existing DRs is a separate follow-up —
  not auto-done (avoids a 19-file churn commit); offered, not forced.
- No change to tier classification or to any existing required phase.
- Implemented via a dedicated spec ([SPEC-013](../../specs/minspec/risk-section-policy/requirements.md))
  so the template + validator work is itself specified and reviewed.

## Alternatives considered

- **Tier-gated (DRs + T3/T4 only; T1/T2 optional).** Initially chosen, then
  reversed: it removes the risk-thinking prompt from the small "obvious" changes
  that most need it. The proportionality concern is better met by scaling *depth*,
  not by exempting tiers.
- **DRs only.** Too narrow — specs carry the implementation, and a T1 fix can be as
  risky as a T4 in blast radius even when small in diff.
- **Hard validator block.** Rejected: only the DR-012 approval gate blocks; every
  other MinSpec check is advisory. A blocking risks-rule would be inconsistent and
  punish drafts.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| 1 | **Rubber-stamping** — authors write "None material" reflexively, defeating the prompt. | Med · Med | Soft-warn only; the T1 floor still forces a *written* answer. Review (DR-012 approval) is where empty risk sections get challenged. Not solvable by tooling alone — accepted residual. |
| 2 | **Ceremony backlash** — mandatory section on T1 feels like the bureaucracy MinSpec sells against. | Med · Med | Depth scales to one line at T1; "None material" is valid. The stub is pre-filled by the template so the cost is editing, not authoring from blank. |
| 3 | **Template/validator drift** — stub shape and validator check disagree, so compliant docs warn. | Low · Med | SPEC-013 makes the stub and the validator rule share one definition of "has a risks section"; one T1 test pins them together. |

Proposed.

## Addendum — 2026-06-02: Consequences is also a required section (on DRs)

The SPEC-013 review generalised the enforcement mechanism from a risks-only check to
a small **required-section registry** (one section-agnostic predicate). This addendum
extends the *policy* to match, so the registry has a cited source for its second
entry (symmetry with the Risks entry).

**Decision.** Every DR MUST carry a `## Consequences` section — the classic
ADR-format section the **Create ADR** template already emits and all current DRs
already have. The registry enforces it as a required section on **DRs only** (specs
use Risks & Mitigations; Consequences is a DR/ADR convention).

**Shape — better than presence (per review OQ2).** The Consequences section is *not*
satisfied by mere presence. It requires a **minimal ± shape**: at least one
**positive** and at least one **negative** consequence, detected deterministically
via configurable **polarity cues** (e.g. `Positive`/`Negative`, `Pros`/`Cons`,
`Benefits`/`Drawbacks`, `+`/`−`). An explicit one-sided answer — "Negative: none" —
satisfies the missing side (mirrors the Risks "None material" escape, DR-020 above):
the value is the *written thought* that there is no downside, not a blank. This stays
**structural, never a quality oracle** (no judgement of whether a consequence is
"good" — SPEC-013 R4 / SPEC-012 FR-15).

**Timing + surfacing.** Consequences (like Risks) is a **cross-cutting** section: it
summarises the whole DR, so it is **completed last / due at the ready point** and its
completeness is **freshness- and coverage-bound**, never latching on presence
([DR-028](DR-028.md)). A missing/stale section is surfaced as a **visible one-click
offer, never silently written and never blocking** ([DR-026](DR-026.md)).

**Backfill (sequencing).** Existing DRs carry Consequences as **prose**, not the ±
shape — so enabling the ± check warns on the back-catalogue. Per SPEC-013 RD-1 the
backfill (restructure existing Consequences into ±) lands **before/with** enabling
the rule, so it first runs against a clean tree (avoids the warning-flood, R2).

**Implemented by** [SPEC-013](../../specs/minspec/risk-section-policy/requirements.md)
(now *Required-Section Policy Enforcement*), which owns the registry, the single
predicate, the templates, and the validator rule.
