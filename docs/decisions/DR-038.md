# DR-038: Unified next-task graph surface (one DAG, subsumes #48 / #182 / #211)

## Context

The next-task signpost ([DR-019](DR-019.md), SPEC-012) emits **one** next human task
from a deterministic cross-artifact DAG. Today the *answer* (one task) and the
*reasoning* (the DAG behind it) are separate ideas with separate, unbuilt surfaces:

- **[#48](https://github.com/harvest316/minspec/issues/48)** — a "dependency-map"
  explorer minimap, scoped to **SPEC/DR/EPIC** nodes only.
- **[#182](https://github.com/harvest316/minspec/issues/182)** — PR-review tasks *in*
  the signpost when gate = PR.
- **[#211](https://github.com/harvest316/minspec/issues/211)** — a separate PR-review
  *queue* explorer pane.

Three surfaces, three node vocabularies, one underlying graph. SPEC-010 FR-4 already
requires the signpost to **show its evidence**; a local graph centred on the signpost
node *is* that evidence rendered spatially — it turns an opaque verdict ("do X next")
into an auditable one ("…because Y and Z block everything else"). Splitting that across
three surfaces fragments the one thing that sells the product: a trustworthy pointer.

## Decision

**One unified graph surface renders the SPEC-012 DAG, with the signpost node at its
centre. It subsumes #48, #182 and #211.**

### 1. Node set = every artifact kind

Nodes are **specs, DRs, epics, issues, and PRs** — a superset of #48 (which saw only
SPEC/DR/EPIC). Edges are the union of implicit SDD-tree edges and explicit
`depends_on:`/`supersedes:`/`relates_to:` edges (DR-019 §7). PRs and issues enter as
the gate/work nodes that #182 and #211 described, now first-class in the same graph.

### 2. A view, never a source of truth

The graph **renders** the DAG the next-task resolver already computes (SPEC-012); it
holds no independent state and cannot disagree with the status-bar signpost or CI. One
engine, many surfaces (DR-019 §6).

### 3. Single-task primacy is an invariant, not a style

The render shows the **signpost node + its first two dependency layers**, with a
floating **"+N" rollup** label on each second-layer node counting what is collapsed
behind it. The centre/primary node is **visually dominant and the only node carrying an
imperative call-to-action**; layer-1/2 nodes are dimmed context with no action
affordance. This is what keeps the surface from degrading into "here's a list" — the
exact anti-pattern [DR-019](DR-019.md) rejects. It is stated here as a hard invariant.

### 4. Two queues stay visually distinct

DR-019 §4 keeps the human-decision queue separate from the agent-dispatch queue. The
graph **encodes queue membership** (e.g. human-gate nodes outlined) so the merge does
not visually conflate "approve this spec" (human) with an agent phase-action.

### 5. Keyboard: a hotkey for the frequent action, mouse OK for the rest

Per the amended global keyboard rule (high-frequency actions need a keyboard path; rare
ones may use the mouse): the **frequent** action — "jump to the next review task" — gets
a **global hotkey** (#258). Graph panning/clicking and node reordering are
*low-frequency* and may be mouse/drag-and-drop. The surface is **not** gated on full
keyboard navigation of the graph (this relaxes #48's "clickable minimap" framing rather
than inheriting it as a violation).

### 6. Downstream of SPEC-012, not parallel to it

This is the "signpost UI surface" SPEC-012 parks in *Out of Scope*. The DAG **data**
does not exist yet (`resolveNextTask` is unbuilt). So this is **spec-after-SPEC-012**:
the resolver ships first; this graph is authored against its output. No graph work
starts before the engine exists.

## Consequences

**Positive:**

- One surface, one node vocabulary, one engine — the signpost, its PR-gate facet, and
  its dependency view can never disagree, because they are the same render.
- "Show the reasoning behind the signpost" becomes literal: the +N-rollup graph makes
  the deterministic derivation inspectable, strengthening the "never wrong" claim.
- #48 / #182 / #211 collapse into facets of one spec, removing three half-overlapping
  UI backlogs.

**Negative:**

- Larger single surface to spec and build than any of the three issues alone — more
  rendering complexity (graph layout, rollup counts, queue encoding) in one place.
- Node-set expansion to issues + PRs is real new scope beyond #48; accepted
  deliberately here (it is the user's explicit ask), not slid in silently.
- Hard-couples the visual surface to the SPEC-012 output contract — a change to the
  resolver's `NextTask` shape ripples into the graph.

## Alternatives considered

- **Keep #48 / #182 / #211 as three surfaces.** Rejected — three node vocabularies over
  one graph, guaranteed to drift and to fragment the product's core pointer.
- **Flat list / table instead of a graph.** Rejected — a list is a backlog; the graph's
  centre-plus-rollup form is what preserves single-task primacy (DR-019).
- **Render all nodes, not two layers.** Rejected — a full graph re-introduces the
  "everything to do" reading; two layers + "+N" rollup collapses depth deliberately.
- **Build now.** Rejected — the DAG data does not exist; the resolver (SPEC-012) is the
  keystone and must ship first.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | **List-leakage** — two fan-out layers read as a backlog, undoing single-task primacy. | Med · High | Decision §3 as a hard invariant: primary node dominant + sole CTA; layers dimmed, no action affordance. T0 test asserts exactly one actionable node. |
| R2 | **Queue conflation** — human-gate and agent nodes look alike, undoing DR-019 §4. | Med · Med | Decision §4 — queue membership encoded in the render; verified against the resolver's node `kind`. |
| R3 | **Scope creep** — issues/PRs node-set balloons the spec. | Med · Med | Decision §6 — strictly a view over the existing resolver DAG; no new node kinds beyond what SPEC-012 already emits. |
| R4 | **Build-before-engine** — graph work starts before `resolveNextTask` exists. | Low · High | Decision §6 — explicitly spec-after-SPEC-012; this DR authorises no code until the engine ships. |

## Costly to Refactor

Cheap *now*, costly *later*. Today nothing is built (no resolver, no graph), so this is
trivially reversible — it is a direction note plus three relinked issues. Once the graph
ships against the SPEC-012 `NextTask` contract, the surface and that contract are
coupled (Decision §6), so reverting to three separate surfaces would mean re-splitting a
shipped UI. **ADR-filter: borderline — undoable in <1 day only while unbuilt**, which is
why the decision is recorded before any code. The node-set expansion to issues/PRs is
the one-way-ish part; it is taken deliberately per the user's explicit ask.
