## Context

Building VS Code extensions for two related but distinct problems:
1. SDD (Spec-Driven Development) productivity tooling — helps developers write just enough spec
2. LLM cost optimization proxy — reduces LLM API costs via middleware chain

Initial concept was one extension ("LLMProxy"). Market research showed SDD tooling is an underserved niche with strong SEO opportunity and no dominant player.

## Decision

Split into three separate extensions:
- **MinSpec** (`aiclarity.minspec`) — free SDD tool, top-of-funnel (Tier 0 / air-gapped)
- **ScroogeLLM** (`aiclarity.scroogellm`) — freemium proxy, monetization vehicle
- **Agent Execute** (`aiclarity.agent-execute`) — Tier 1 agent dispatch / execution substrate. Productized dispatch ships here, NOT inside MinSpec. See DR-015 (packaging), DR-004 (tier model), DR-008 (dispatch security), DR-017 (execution substrate).
- **MinSpec Pro** (`aiclarity.minspec-pro`) — extension pack, premium upsell

Use SDD as the primary development methodology for building both extensions (dogfooding).

## Rationale

- SDD tool has lower scope and faster path to v1 → ships first
- Two free tools create broader surface area for discovery
- Extension pack is natural upsell when both are valuable independently
- Dogfooding SDD builds credibility and surface issues early

## Consequences

**Positive:**
- Two free tools (MinSpec + ScroogeLLM) create broader surface area for marketplace discovery than the single original "LLMProxy" concept would have.
- Shared classification engine extracted to `packages/shared` (Tier-0, no vscode/network — see DR-014), single-sourcing the classifier instead of forking it per extension. Realised for MinSpec today (consumed via `packages/minspec/src/lib/bridge.ts`); agent-execute is still spec-stage (no package yet, per DR-017) so its consumption is the projected, not yet realised, payoff of the extraction.
- MinSpec's lower scope ships first, decoupling its release from the heavier proxy work; ScroogeLLM specs can defer until MinSpec reaches stable v1.

**Negative:**
- Three extension IDs and three marketplace listings to maintain (`aiclarity.minspec`, `aiclarity.scroogellm`, `aiclarity.agent-execute`) plus the `aiclarity.minspec-pro` pack — four publish/version surfaces instead of one.
- agent-execute may have to list on the marketplace without its own domain, weakening its SEO/landing story relative to minspec.dev and scroogellm.com.
- ScroogeLLM work is blocked behind MinSpec shipping, delaying the monetization vehicle (later compounded by the DR-027 split into the private `harvest316/scroogellm` repo).

## Risks & Mitigations

- **Split overhead never pays back** — three+ extensions multiply CI, versioning, and marketplace upkeep; if cross-tool discovery (the main "Positive") doesn't materialise, the monorepo carries the cost of a strategy that bought nothing. Mitigation: the MinSpec Pro pack (`aiclarity.minspec-pro`) bundles the listings so a user who finds one is funnelled to the others, and `packages/shared` keeps the classifier single-sourced so the split doesn't fork core logic.
- **Tier-boundary leakage** — putting agent dispatch in MinSpec (Tier 0 / air-gapped) would violate the tier model; this decision explicitly routes dispatch into `aiclarity.agent-execute` (Tier 1). Mitigation/anchor: enforced by DR-004 (tier model), DR-008 (dispatch security), DR-015 (packaging), DR-017 (execution substrate) — the boundary is owned by those DRs, not just prose here.
- **ScroogeLLM starvation** — gating ScroogeLLM specs behind MinSpec v1 risks the monetization vehicle never getting attention. Realised partially: ScroogeLLM was later split out to private repo `harvest316/scroogellm` (DR-027) and is only at Specify (SPEC-100/101/102). Mitigation: tracked as its own repo with its own DR register so it can progress independently of MinSpec's Implement phase.

## Alternatives Considered

- **Single "LLMProxy" extension** (the initial concept named in Context) — rejected because market research showed SDD tooling is an underserved niche with strong SEO and no dominant player, value a bundled proxy-only extension could not capture; collapsing SDD and proxy into one listing also forfeits the broader top-of-funnel surface area two free tools provide.
- **Ship ScroogeLLM (proxy/monetization) first** — rejected because the SDD tool has lower scope and a faster path to v1 (per Rationale); leading with the heavier proxy work would delay any shippable artifact and the dogfooding credibility MinSpec provides.

## Costly to Refactor

Moderately costly, exceeds the <1-day ADR-filter, hence this DR. Reversing the split (re-merging into one extension) means collapsing three+ marketplace IDs and their published version histories, un-extracting `packages/shared`, and reworking the tier model owned by DR-004/DR-008/DR-015/DR-017 — multi-day work touching every package. The choice has already accreted dependent decisions (DR-014 shared package, DR-027 ScroogeLLM repo split), each of which would also need unwinding, so the effective blast radius grows over time rather than staying fixed.
