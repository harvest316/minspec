# DR-034: Committed, attributed approval ground truth + derived spec status

> Triggered by #112 (status=done with no approval record) â€” a higher-touch fix
> dead-ended on architecture: approval ground truth is **local + per-checkout**, so
> the invariant it needs to enforce is structurally unenforceable. Resolves the
> blocker by promoting approvals to a **committed, attributed** artifact (#95) and
> **deriving** status from {phases, approval} (#116). Amends DR-012 (approval design)
> and builds on DR-031 (canonical approval resolution).

## Context

DR-012 made spec approval an explicit human act: a content hash recorded in
`.minspec/approvals.json`, with a PreToolUse gate (DR-031 / spec-gate.py) that
denies source edits while any T3/T4 spec is `status: implementing` without a current
approval. It is the only enforcement that survives bypass-permissions mode.

Three coupled defects make the gate's core invariant â€” **`status âˆˆ {implementing,
done} â‡’ a current approval record exists`** (#112) â€” unenforceable, and let the
status signpost lie:

1. **Ground truth is local + per-checkout.** `.minspec/approvals.json` is gitignored
   (`.gitignore:39`). DR-031 worked around *linked worktrees* by resolving the main
   checkout's file via `git --git-common-dir` â€” but that only saves same-machine
   worktrees. A **fresh clone, CI runner, or teammate** has an empty store, so the
   gate would flag every implementing/done spec â†’ build always red. Verified: the
   primary checkout holds 15 records; a fresh worktree sees 7 unbacked primary specs.
   The two disagree, so "status backed by ground truth" cannot be enforced (#95).

2. **Two status representations diverge (#148).** `parseSpec` (spec.ts) reads the
   **literal** `status:` frontmatter line as authoritative for display; `getSpecStatus`
   (lifecycle.ts) **derives** status from the phases map. Approval flips only the
   literal line (approve.ts), leaving phases stale â€” so a spec can read `status:
   implementing` while `phases.plan: pending`. The literal line is hand-editable to a
   value nothing produced (the #112/#116 signpost-lie).

3. **The hash voids itself (#116/#166).** The approved hash is sha256 over **raw file
   bytes** (approval.ts). Any edit voids approval â€” including the tool's *own* status
   flip (forcing the fragile flip-then-hash ordering in approve.ts), a deterministic
   lifecycle transition, and a cosmetic prose-link fix (#166, left unfixed precisely
   because editing the body would void an intact approval and trip the repo-wide stale
   gate). Raw-byte hashing is also CRLFâ†”LF-fragile across machines (#95 fan-out 1).

The common root is **the ground truth's durability and shape**, not any one symptom.
Patching status data or re-approving by hand are data-only fixes (DR-003) â€” the gates
that should make the bad state un-representable are what's missing.

## Decision

Five coupled changes. They ship **warn-first** and promote to error once the corpus
is clean (the #137/#103/#111 pattern).

### 1. Approval ground truth is COMMITTED, as per-spec, path-keyed sidecars

Un-gitignore approvals. Replace the single `.minspec/approvals.json` with one sidecar
file per spec, **keyed by the spec's repo-relative path**:

```
.minspec/approvals/specs/minspec/SPEC-007-foo/requirements.md.json
```

- **Path-keyed, not id-keyed** â†’ the key is inherently unique, so it sidesteps the
  cross-product SPEC-id collision (#58, scrooge vs minspec `SPEC-001`) *in the approval
  keyspace* without depending on the broader id-policy fix.
- **Per-spec file** â†’ a merge conflict arises **only when two devs approve the same
  spec** (a genuine conflict worth surfacing), never when they approve different specs.
  `git log .minspec/approvals/<path>.json` is that spec's approval history.
- DR-031's `--git-common-dir` canonical-resolution becomes a **fallback** for an
  uncommitted local approval during authoring, not the load-bearing path: a committed
  sidecar is present in every clone/worktree/CI checkout by construction.

### 2. Records are attributed (Tier-0 identity)

Each sidecar carries:

```jsonc
{
  "specPath": "specs/minspec/SPEC-007-foo/requirements.md",
  "specHash": "<canonical-hash, see Â§3>",
  "approvedAt": "2026-06-06T...Z",
  "approvedBy": "paul@harvest316.com",   // git config user.email at approval time
  "tier": "T3",
  "migrated": false                       // true only for Â§5 backfilled records
}
```

`approvedBy` = `git config user.email`, captured by the extension at approval time â€”
**Tier-0**: offline, headless, no identity service (DR-004). **Any committer may
approve** (v1): the value is the *attributed audit trail* (who / what hash / when),
not an authority gate. A CODEOWNERS-style reviewer-set is explicitly deferred
(follow-up) â€” it is identity/authority infra that pushes toward Tier 1.
Single-approver-suffices (one record clears the spec for the team), per #95.

### 3. Hash covers CANONICALIZED content; lifecycle fields are excluded

The approved hash is sha256 over a **canonical form** of the spec, defined precisely
so independent implementations reproduce it byte-identically:

1. Split frontmatter block from body.
2. From frontmatter, **remove exactly the lifecycle keys `status` and `phases`**
   (and the `phases:` map's indented sub-lines). Everything else in frontmatter â€”
   `id`, `tier`, `type`, `epic`, `aspects`, `title`, `superseded-by`, â€¦ â€” is
   **content**.
3. Rejoin frontmatter-minus-lifecycle + body.
4. Normalize: EOL â†’ `\n`; strip trailing whitespace per line; ensure exactly one
   trailing newline.
5. `sha256` the result.

Consequences:
- Editing `status`/`phases` (the tool's own lifecycle transitions, #148; a deterministic
  advance) **no longer voids** a content approval â€” kills the flip-then-hash dance.
- Editing the **body or any other frontmatter field still voids** approval (by design,
  DR-012) â€” substantive change re-triggers review.
- CRLFâ†”LF and trailing-WS no longer flip the hash (#95 fan-out 1). Belt-and-suspenders:
  add `.gitattributes` `specs/** text eol=lf`.

**#166's "lock-safe edit region" (hash-excluding a prose span) is OUT OF SCOPE** â€” a
separate, larger feature. This DR excludes *lifecycle frontmatter fields* only, not
arbitrary body regions.

**Implementation discipline (the main risk).** The canonicalization is the contract.
Implement it **once** in a shared module (Node, consumed by the extension) and **once**
in Python (spec-gate.py). **Drop the DR-012 bash `sha256sum` path** â€” the gate is
Python now, and a third raw-byte impl cannot reproduce canonicalization without
divergence. A **T0 invariant test asserts Node â‰¡ Python** over the entire `specs/`
corpus; CI runs it on every change to either implementation.

### 4. Status is DERIVED (source of truth) + the literal line is a validated mirror

A single function is the source of truth:

```
deriveStatus(phases, approvalState, explicitTerminal) -> SpecStatus
  explicitTerminal (archived | superseded)  -> that terminal      // human act, not derived
  all phases pending                         -> new
  not approved                               -> specifying         // cannot pass specify/clarify unapproved
  approved + implement phase in progress     -> implementing
  approved + all required phases done         -> done
```

- The literal `status:` frontmatter line becomes a **tool-written mirror** (already
  written via `setSpecStatus`), never hand-authored as truth.
- The validator asserts **literal == derived** via the #137 symmetric primitive â†’
  **warning** on drift (catches hand-edits and the #148 phases/status desync).
- The gate and CI read the **derived** status.

This makes `implementing`/`done` **structurally impossible without a current approval
record** â€” that IS the #112 fix, enforced rather than asserted.
`archived` (and a future `superseded`, #162) form the **explicit-terminal class** â€”
human acts feeding the derivation, not derived from phases.

**Caveat (#116):** full `done`-from-task-completion needs task-tracking wired for
split-layout specs (no task checkboxes today). v1 derives `new`/`specifying`/
`implementing` from {phases, approval}; `done` continues to rely on the
implement-phase signal until task-tracking lands (follow-up).

### 5. Migration â€” warn-first, no flag day

A one-time script:
1. Converts the 15 local `approvals.json` records â†’ committed, path-keyed sidecars,
   **recomputing each hash under Â§3 canonicalization** (raw-byte hashes are invalid
   under the new scheme), backfilling `approvedBy` = repo owner's `git user.email`.
2. For the **7 shipped specs that are `implementing`/`done` with no record**, writes a
   sidecar marked **`migrated: true`** (attributed to owner + migration date). The
   gate/validator treat migrated records as **valid-but-flagged** ("approval migrated,
   not a recorded human act â€” re-approve to clear"). This is honest about provenance
   (no manufactured "human approved at hash X" claim â€” the evidence-discipline value)
   **and** non-blocking on day 1.
3. The gate ships **warn**, promoting to **error** once the corpus is clean (no migrated
   records, no literal/derived drift).

## Invariants (T0 â€” tests before implementation)

- **INV-1 (#112):** `deriveStatus(...) âˆˆ {implementing, done}` â‡’ a current approval
  record exists whose `specHash` matches the canonical hash of the spec's content.
- **INV-2 (hash agreement):** the canonical hash of every spec in `specs/` is
  byte-identical across the Node and Python implementations.
- **INV-3 (lifecycle-edit non-void):** editing only `status` and/or `phases` does NOT
  change the canonical hash; editing the body or any other frontmatter field DOES.
- **INV-4 (mirror consistency):** for every spec, literal `status:` == `deriveStatus`;
  a mismatch yields a validator warning (never silently passes).
- **INV-5 (key uniqueness):** approval sidecar keys (spec paths) are unique within the
  repo â€” no two specs share an approval record.
- **INV-6 (terminal honesty):** `archived`/`superseded` are set only by an explicit
  human act, never inferred from phases.

## Contracts (define before implementation)

- `ApprovalRecord` (sidecar JSON) â€” Â§2 shape, as a TS type + the on-disk JSON schema.
- `canonicalizeSpec(rawSpec) -> string` and `specHash(rawSpec) -> string` â€” the Â§3
  contract, with the Python twin specified line-for-line.
- `deriveStatus(phases, approvalState, explicitTerminal) -> SpecStatus` â€” Â§4.
- Approval store read/write keyed by spec path (replaces the id-keyed

## Risks & Mitigations

- **Canonicalization divergence across impls (highest risk).** â†’ one shared Node
  module + one Python twin; drop the bash path; INV-2 corpus test gates CI.
- **Migration manufacturing approvals.** â†’ `migrated: true` provenance flag; never
  emit an unflagged human-approval record the human did not perform (evidence
  discipline).
- **Committed-store merge conflicts.** â†’ per-spec path-keyed sidecars; conflict only on
  same-spec double-approve, which is a real conflict to surface.
- **Promoting warnâ†’error too early.** â†’ stays warning until the corpus has zero migrated
  records and zero literal/derived drift; promotion is a separate, observable step.
- **Hidden literal-status consumers.** â†’ audit every reader of `frontmatter.status`
  before flipping it to mirror semantics; the validator surfaces any that re-introduce
  drift.

## Costly to Refactor

- **Medium-high.** The hash scheme is a content contract embedded in committed records;
  once specs are approved under Â§3, changing canonicalization re-voids them. Status
  derivation is reversible (re-introduce literal-as-truth). Committing the store is
  reversible (re-gitignore) but loses the team/CI value. This is why it is DR-gated and
  warn-first.

## Consequences

- The #112 invariant becomes **enforceable** in CI and fresh checkouts â€” status can no
  longer claim implementing/done without committed, attributed ground truth.
- #148 (status/phases desync) and the #116 self-voiding hash are resolved at the root;
  #166's specific stale-link can be fixed under the lifecycle-exclusion once its edit is
  body-only (still voids) â€” its full "lock-safe region" remains a separate feature.
- Approval gains a git-visible audit trail (who/what-hash/when) â€” the team-readiness
  story DR-012 promised but local-only state could not deliver (#95).
- DR-012's bash `sha256sum` hashing path is retired; DR-031's canonical resolution is
  demoted to a fallback.

## Alternatives considered

- **Keep approvals local; only fix worktrees (status quo + DR-031).** Rejected: CI /
  fresh clone / teammate still see an empty store â€” the #112 invariant stays
  unenforceable. The root is durability, not worktree propagation.
- **Single committed `approvals.json`.** Rejected: every concurrent approval of
  *different* specs collides on one JSON object â€” merge pain that scales with team size.
- **Id-keyed sidecars.** Rejected: reintroduces #58 cross-product id collision in the
  approval keyspace.
- **Keep literal status as truth; only validate against derived.** Rejected: status can
  still momentarily lie until validation runs â€” a check, not a structural guarantee; a
  weaker #112 fix.
- **Build a reviewer-authority registry now.** Rejected for v1: identity/authority infra
  (Tier 1); the audit trail delivers the v1 value. Deferred.
- **Backfill the 7 unbacked specs as real approvals.** Rejected: manufactures human
  approvals that never happened â€” the exact signpost-lie this project forbids. Use the
  `migrated` flag instead.
- **Require re-approval of all 7 before cutover.** Rejected: a flag-day gate, contra the
  no-flag-day constraint.
