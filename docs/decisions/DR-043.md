# DR-043: Approval baseline as a pinned git blob, referenced from the committed ledger

## Context

SPEC-017 (Trust Dashboard) needs an **approval baseline** — the exact approved
spec body at approval time — so it can later char-diff current-vs-approved and
report rework % (M1). SPEC-017 FR-OQ4 originally resolved this *by engineering
default* to: gzip the latest-approved body into a **git-ignored**
`.minspec/snapshots/` sidecar.

DR-042 then found the load-bearing gap one layer down: `.minspec/approvals.json`
(and that gzip sidecar) are git-ignored, so the baseline is **per-machine, lost on
a fresh clone — there is no committed ground truth** to join a review act to a
downstream outcome across a project's life. DR-042 #300 already calls for a
*committed, content-free approval ledger* (hash + tier + approver + timestamps).

In the SPEC-017 clarify session the founder asked the sharp question: *"can't we
use git versioning to store approvals instead of gzip?"* The key fact: **the
approved body is already in git** — the spec file is tracked — so a gzip copy
duplicates bytes git already holds, and makes them per-machine. The right move is
to reference the bytes git already has, not copy them.

The one complication: DR-012 approval hashes the **on-disk** (possibly
uncommitted/dirty) body, so the approved bytes may never be committed exactly as
approved — there is not always a natural commit/blob to point at.

## Decision

Store the SPEC-017 approval baseline **git-natively**, unifying it with the
DR-042 #300 committed ledger:

1. **Mint a blob at approve-time.** On approval, write the approved body to git's
   object store as a content-addressed blob — `git hash-object -w` (zlib-
   compressed, content-addressed, deduped by git; **no duplicate copy** of the
   body). This works whether or not the working tree is clean, so the dirty-tree
   case is covered.
2. **Pin it against gc.** Create a ref `refs/minspec/snapshots/<specId>` (and/or
   `…/<specId>@<approvedAt>`) pointing at the blob, so `git gc` cannot prune an
   otherwise-unreachable object.
3. **Record only the SHA in the committed ledger.** The DR-042 #300 ledger entry
   becomes `{ specId, baselineBlob: <sha>, specHash, tier, approver, approvedAt }`
   — content-free (a SHA + metadata, never body text), **committed** to the repo.
   One mechanism now serves both SPEC-017's baseline and #300's ground truth.
4. **Recover by plumbing.** Rework % recovers the baseline with
   `git cat-file blob <sha>` and char-diffs against the current body (SPEC-017
   FR-2). The git read lives in the `vscode` glue (like file I/O); the metric
   functions stay pure (SPEC-017 FR-12).
5. **Gzip is the fallback, not the default.** When the project is **not** a git
   repo, fall back to the original gzipped `.minspec/snapshots/` sidecar (baseline
   then per-machine again, as before). MinSpec does not *require* git for M1/M2 to
   function; it *prefers* it for committed ground truth.

This **supersedes** SPEC-017 FR-OQ4's original gzip-sidecar resolution and amends
SPEC-017 Costly-to-Refactor #4.

## Rationale

- **Closes the DR-042 ground-truth gap at its source.** The baseline becomes a
  committed, shareable, auditable git object tied to author + timestamp — exactly
  what "prove value across a project's life" needs — instead of a per-machine file.
- **No duplication.** Git already stores the spec bytes; we add a 40-char SHA, not
  a second compressed copy. R3 (snapshot bloat) drops from Med to Low.
- **One mechanism, two needs.** SPEC-017's snapshot and #300's ledger were separate
  designs; the blob-SHA-in-the-ledger collapses them into one.
- **Dirty-tree-safe.** `git hash-object -w` mints the blob from the approved bytes
  directly — no dependency on the user having committed (the founder delegates git
  and it "mystifies" them; approval must not force a commit).

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Unreachable blob pruned by `git gc` | Pin with a `refs/minspec/snapshots/` ref; reachable objects are never pruned. T0/T1 test: `gc --prune=now` then `cat-file` still returns the body. |
| Writing into `.git` surprises users | Custom ref namespace (`refs/minspec/`) is invisible to branch UIs; blobs are ordinary git objects; **no commits** are created (no history noise). Documented in onboarding. |
| "Shared ground truth" assumed but refs are not pushed by default | The *ledger* (a tracked file) is the committed/shared record; the blob refs are a local optimisation. Cross-machine baseline recovery needs an explicit `refs/minspec/*` push refspec — a tracked follow-up, not an assumption. |
| Non-git project | Gzip `.minspec/snapshots/` fallback (decision pt 5); M1/M2 still compute, baseline just per-machine. |
| Approval write path now also touches git | Confined to the single `approval.ts` write site (SPEC-017 A1); T1 contract test on the approve→blob→ref→ledger sequence. |

## Consequences

- SPEC-017 FR-1 / FR-2 / FR-OQ4 / Costly #4 / R3 / A4 / INV-Deterministic updated to
  the git-blob baseline (this DR's clarify PR).
- DR-042 #300's ledger schema gains `baselineBlob: <sha>`; the ledger becomes the
  join key *and* the snapshot index.
- A new ref namespace `refs/minspec/snapshots/` appears in the repo (local).
- Adds a hard dependency on `git` for the *preferred* path; the gzip fallback keeps
  the no-git story alive.
