# DR-040: Friction-free materialization of DR follow-ups (auto-create, not block)

## Context

[DR-023](DR-023.md) requires every DR to materialize its surfaced work as tracked
issues/specs in a `## Follow-ups (tracked)` section, with only a **soft validator
warning** when items lack a ref. The session asked whether to *harden* this into a
blocking gate — because un-materialized follow-ups are the mechanism by which "newer
specs/DRs not yet turned into issues/PRs" stay invisible to the next-task DAG (DR-019):
the resolver ranks structural edges, so prose-only follow-ups are simply not there.

A blocking gate would close the leak but **surface friction to the human** at commit
time — the opposite of the product's "the tooling handles the bookkeeping" ethos. The
user's steer: keep the guarantee, drop the friction — *auto-create the missing issues*.

This also reconciles with DR-023's own rejected alternative ("auto-create an issue for
every Consequence bullet" — rejected as noise): the difference is **scope**. We auto-file
only the **curated `## Follow-ups (tracked)`** items the author already chose as
actionable, never every descriptive consequence.

## Decision

**Replace DR-023's soft-warning with friction-free auto-materialization. On DR save,
un-materialized follow-ups become real issues automatically; only genuinely broken
refs surface to the human.**

### 1. Auto-create for un-materialized follow-up bullets

On DR save (and in CI as the backstop), scan the `## Follow-ups (tracked)` section. For
each bullet that has **no** resolvable issue/spec ref, `gh issue create` from the bullet
text and **write the new `#ref` back** into the DR in place. The author sees a populated
section; no gate, no prompt, no commit rejection.

### 2. Dangling refs are a real error, still surfaced

A bullet that **does** carry a ref which does **not** resolve (a deleted/typo'd `#N`, or
a `SPEC-NNN` not in the index) is **corruption, not absence** — we cannot fabricate an
issue of unknown shape. This stays a surfaced error (the existing dangling-ref /
structural-corruption handling, DR-019 §8), never an auto-create.

### 3. Scope is the curated section only

Auto-create runs **only** over `## Follow-ups (tracked)`, never over `## Consequences`
or prose. This is what makes auto-create safe where DR-023 rejected the blanket version:
the human has already curated *which* lines are actionable; the tooling only spares them
the filing.

### 4. "None" remains valid and untouched

An explicit `None` is a no-op — the author asserting there is no orphan work. Auto-create
neither fires nor warns.

## Consequences

**Positive:**

- The DR-023 guarantee (no orphan consequences) becomes **enforced by construction**
  rather than relying on author discipline — and with *less* friction than the status quo
  warning, not more.
- Closes the DR-019 input leak honestly: un-materialized follow-ups become real nodes the
  DAG can see, instead of teaching the resolver to infer priority from prose (which would
  reintroduce LLM ranking — explicitly out of bounds).
- Dogfoods itself: this DR's own `## Follow-ups (tracked)` is the first to be auto-filled.

**Negative:**

- Auto-created issues are only as good as the bullet text — a terse bullet yields a terse
  issue (mitigated: the author is editing the DR anyway and can refine; an LLM polish pass
  is optional, not required).
- Generating issues on save is an outward/irreversible side effect of an editor action —
  must be **idempotent** (never double-file the same bullet) and scoped to the user's own
  repo.

## Alternatives considered

- **Hard-block the commit until follow-ups are materialized** (the session's first
  framing). Rejected — surfaces friction to the human; only DR-012 approval blocks in
  MinSpec, and this would be a second blocking gate for bookkeeping.
- **Keep DR-023's soft warning** (status quo). Rejected — relies on discipline and leaves
  the DR-019 leak open; the warning is friction without the guarantee.
- **Auto-create over every Consequence bullet.** Rejected for the same reason DR-023 did —
  noise; most consequences are descriptive. Scope is the curated section (Decision §3).

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | **Double-filing** — re-save creates duplicate issues for the same bullet. | Med · Med | Decision §1 must be idempotent: a bullet with a written-back ref is skipped; match on bullet identity, not position. |
| R2 | **Surprise outward action** — editor save silently opens GitHub issues. | Med · Med | Scoped to the user's repo; visible in the diff (the written-back `#ref`); CI is the authoritative backstop, the editor path is the convenience. |
| R3 | **Low-quality auto-issues** from terse bullets. | Med · Low | Author is editing the DR and curates the section; optional LLM polish; the issue is a stub to refine, better than orphaned prose. |
| R4 | **Dangling ref mis-handled as "absent"** → wrong auto-create. | Low · Med | Decision §2 — a present-but-unresolvable ref is corruption, surfaced, never auto-created. |

## Costly to Refactor

Cheap to reverse — **undoable in well under a day**, like DR-023 itself. It is one
behaviour change on the validator/save path (warn → auto-create) plus an idempotency
guard; no schema, no stored state, no cross-repo contract. Reverting means restoring the
soft warning. The only lingering artifacts are the auto-created issues, which are ordinary
issues either way. Passes the ADR filter as a low-blast-radius process rule; recorded
because it changes the DR-023 contract a future author relies on.
