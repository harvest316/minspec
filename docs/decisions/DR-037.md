# DR-037: Scaffold editor-independent git hooks into user projects

## Context

MinSpec's SDD gates (spec `id:` frontmatter, RCDD root-cause line, ref-egress leak
DR-032) only fire when the user goes through the VS Code Command Palette. A terminal
`git commit`, a different editor, or an AI agent committing via Bash bypasses all of
them. The RCDD Phase-4 rule says bad states should be **un-committable** — the current
setup violates that for any workflow outside VS Code.

MinSpec is a VS Code extension with no `bin` field — there is no `minspec` shell
command. All validators are TypeScript modules inside the extension process, so they
cannot be called from a shell hook without a separate shippable entrypoint.

The exploration was prompted by evaluating Husky. Husky solves hook *management*, not
the validator entrypoint problem; `core.hooksPath` is sufficient once hooks exist.
Husky itself is not adopted (the monorepo already has an equivalent lighter pattern;
pushing a new dep into user repos for non-Node projects is invasive).

## Decision

**Scaffold a 3-tier detection-chain hook into user projects during `Initialize` /
`Refresh Harness`**, plus a companion GitHub Actions workflow as the authoritative
post-push gate.

### Hook detection chain

Scaffolded as `.minspec/hooks/pre-commit` (shell) + `.minspec/hooks/commit-msg` (shell):

```sh
#!/bin/sh
# MinSpec gate — runs the highest-fidelity validator available
if command -v node >/dev/null 2>&1; then
  npx --yes @aiclarity/minspec-validator "$@"
elif command -v python3 >/dev/null 2>&1; then
  python3 "$(dirname "$0")/validate.py" "$@"
else
  "$(dirname "$0")/validate.sh" "$@"
fi
```

| Tier | Requirement | Coverage |
|---|---|---|
| Node (`npx`) | `node` on PATH | Full — YAML parse, cross-ref resolution, egress check (DR-032), RCDD gate |
| Python 3 | `python3` on PATH | Good — `PyYAML` frontmatter parse, regex RCDD gate, basic cross-ref |
| Shell (`bash`/`sh`) | Always present | Minimal but never-wrong — `grep` for `id: SPEC-`, `Root cause:` in fix commits |

Shell fallback covers the two highest-value gates because they are pattern-matchable;
full structural validation requires a parser.

### Activation in user repo

`scaffold.ts` writes the hook files and sets `core.hooksPath=.minspec/hooks` in the
user project's git config (not `package.json prepare` — avoids Node/npm assumption):

```ts
execSync('git config core.hooksPath .minspec/hooks', { cwd: rootDir });
```

Idempotent — no-op if already set to that value.

### GitHub Actions backstop

`Initialize` also scaffolds `.github/workflows/minspec-validate.yml`. This always uses
the Node tier (CI has Node) and is the authoritative gate — it catches contributors
without local hooks and any bypass. Local hook = fast fail; CI = never-merge guarantee.

### New package: `@aiclarity/minspec-validator`

A thin standalone Node package (separate from the extension VSIX) exposing:

```
npx @aiclarity/minspec-validator [--pre-commit | --commit-msg <file>]
```

It re-uses the same validator logic already in `packages/shared` — no duplication,
just a new entry point. Published to npm independently of the extension release cycle.

## What this does NOT do

- Does not add Husky or any hook-management dependency to user repos.
- Does not require Node in user repos — shell fallback always works.
- Does not gate on Python being installed — it is an opportunistic upgrade tier.
- Does not compile a standalone binary — the npx+python+sh chain covers the use cases
  without cross-platform binary distribution complexity.
