## Context

MinSpec writes content into shared harness files (`CLAUDE.md`, `AGENTS.md`,
`.cursorrules`, `DESIGN.md`, `.minspec/constitution.md`, `docs/decisions/INDEX.md`)
inside explicit markers: `<!-- minspec:NAME:start -->` … `<!-- minspec:NAME:end -->`
(e.g. `dr-index`, `slash-commands`, `active-spec`). `refreshHarnessFiles()` and
`adr-manager` already merge by replacing only the content between those markers,
preserving everything outside (invariant #6).

Two prompts still interrupt updates to these MinSpec-owned regions:

1. **Claude Code agent edits** — the harness prompts for permission whenever the
   dev agent edits `CLAUDE.md`/`AGENTS.md`/etc., even when the edit lands entirely
   inside a `minspec:*` block.
2. **Extension refresh toast** — DR-006's auto-bootstrap offers a "Refresh harness
   files?" toast on drift, re-asking each time.

User request (2026-05-30): once our sections exist, stop asking permission to
update *our* sections.

## Decision

Stop prompting for updates **confined to MinSpec's own marker blocks** — at both
layers — while leaving all content outside the markers fully protected.

### Layer 1 — Claude Code agent edits (this repo)

A `PreToolUse` hook `scripts/hooks/marker-guard.mjs` auto-approves `Edit` /
`MultiEdit` calls **only** when every edited span lies entirely within a
`minspec:*` marker block of a managed file. **Default-deny by construction:** the
only behaviour-changing output is an `allow` decision emitted on proven
containment; every other case (edit outside markers, ambiguous match, whole-file
`Write`, unmanaged file, new/unreadable file) emits nothing and defers to the
normal prompt. A bug can only fail to allow (→ prompt), never over-approve. Runs
alongside `spec-gate.sh`; a `deny` from any hook still wins, so the src/spec gate
(DR-012) is unaffected.

### Layer 2 — extension refresh (SPEC-005 FR-6)

The auto-bootstrap "refresh" path may re-sync MinSpec marker sections silently
(no toast) once the project is initialized, because the merge touches only
generated content between markers. The init/missing-structure offer and any
change that would alter content **outside** markers still prompt. Gated by the
existing `minspec.autoBootstrap.enabled` master toggle.

## Rationale

- Marker-bounded edits never touch user-authored content (invariant #6), so the
  prompt was guarding a region the user does not own — pure friction.
- Symmetry: same "only our sections, never yours" guarantee enforced at both the
  agent layer (hook) and the product layer (merge).
- Pure file-system; no AI, no network (Tier 0, DR-004).

## Costly to Refactor

- **The `minspec:*` marker syntax** — the contract defining what's auto-editable; changing it re-marks every managed file. *Check: marker syntax stable before adoption.*
- **The PreToolUse default-deny design** — inverting it (default-allow) is a safety regression, costly to re-audit. *Check: default-deny preserved.*

## Consequences

- **Positive**: no permission interruptions for routine `dr-index` /
  `slash-commands` / `active-spec` updates.
- **Risk**: hook misjudges containment and auto-approves a non-marker edit.
  Mitigation: default-deny — containment must be *proven* (single unambiguous
  match fully inside a block) or the hook stays silent; whole-file `Write` is
  never auto-approved.
- **Negative**: a managed file with no markers yet gets no auto-approval (correct
  — there is no "our section" to bound to until init writes one).
