## Context

MinSpec writes content into shared harness files (`CLAUDE.md`, `AGENTS.md`,
`.cursorrules`, `DESIGN.md`, `.minspec/constitution.md`, `docs/decisions/INDEX.md`)
inside explicit markers: `<!-- minspec:NAME:start -->` … `<!-- minspec:NAME:end -->`
(e.g. `dr-index`, `slash-commands`, `active-spec`). `refreshHarnessFiles()` and
`adr-manager` already merge by replacing only the content between those markers,
preserving everything outside (invariant #6).

Two prompts still interrupt updates to these MinSpec-owned regions:

1. **Claude Code agent edits** — the harness prompts for permission whenever the
   dev agent edits `CLAUDE.md`/`AGENTS.md`/etc., even when the edit lands entirely
   inside a `minspec:*` block.
2. **Extension refresh toast** — DR-006's auto-bootstrap offers a "Refresh harness
   files?" toast on drift, re-asking each time.

User request (2026-05-30): once our sections exist, stop asking permission to
update *our* sections.

## Decision

Stop prompting for updates **confined to MinSpec's own marker blocks** — at both
layers — while leaving all content outside the markers fully protected.

### Layer 1 — Claude Code agent edits (this repo)

A `PreToolUse` hook `scripts/hooks/marker-guard.mjs` auto-approves `Edit` /
`MultiEdit` calls **only** when every edited span lies entirely within a
`minspec:*` marker block of a managed file. **Default-deny by construction:** the
only behaviour-changing output is an `allow` decision emitted on proven
containment; every other case (edit outside markers, ambiguous match, whole-file
`Write`, unmanaged file, new/unreadable file) emits nothing and defers to the
normal prompt. A bug can only fail to allow (→ prompt), never over-approve. Runs
alongside `spec-gate.sh`; a `deny` from any hook still wins, so the src/spec gate
(DR-012) is unaffected.

### Layer 2 — extension refresh (SPEC-005 FR-6)

The auto-bootstrap "refresh" path may re-sync MinSpec marker sections silently
(no toast) once the project is initialized, because the merge touches only
generated content between markers. The init/missing-structure offer and any
change that would alter content **outside** markers still prompt. Gated by the
existing `minspec.autoBootstrap.enabled` master toggle.

## Rationale

- Marker-bounded edits never touch user-authored content (invariant #6), so the
  prompt was guarding a region the user does not own — pure friction.
- Symmetry: same "only our sections, never yours" guarantee enforced at both the
  agent layer (hook) and the product layer (merge).
- Pure file-system; no AI, no network (Tier 0, DR-004).

## Costly to Refactor

- **The `minspec:*` marker syntax** — the contract defining what's auto-editable; changing it re-marks every managed file. *Check: marker syntax stable before adoption.*
- **The PreToolUse default-deny design** — inverting it (default-allow) is a safety regression, costly to re-audit. *Check: default-deny preserved.*

**ADR-filter (can this be undone in <1 day?):** Yes for this DR's own addition —
deleting `scripts/hooks/marker-guard.mjs` (or its `PreToolUse` registration)
restores the agent prompt immediately, and clearing `minspec.autoBootstrap.enabled`
restores the Layer 2 toast, with no data migration and no change to already-merged
content. The two items above are the *pre-existing* contract (marker syntax,
default-deny), not introduced here; this decision only adds a withdrawable silent
`allow` path on top of them.

## Consequences

**Positive:**

- No permission interruptions for routine `dr-index` / `slash-commands` /
  `active-spec` updates inside MinSpec marker blocks (Layer 1 hook +
  `refreshHarnessFiles()` merge).
- One "only our sections, never yours" guarantee enforced symmetrically at both
  the agent layer (`scripts/hooks/marker-guard.mjs`) and the product layer
  (SPEC-005 FR-6 merge), so the user's mental model is uniform across surfaces.
- Removes DR-006's repeated "Refresh harness files?" toast on drift for
  marker-only re-syncs, while the init/missing-structure offer still prompts.

**Negative:**

- A managed file with no `minspec:*` markers yet gets no auto-approval (correct —
  there is no "our section" to bound to until init writes one), so first-time
  edits before init still prompt.
- Adds a second `PreToolUse` hook (`marker-guard.mjs`) running alongside
  `spec-gate.sh` on every `Edit`/`MultiEdit`; the parse+containment check is now
  on the hot path for all agent edits, not just harness ones.
- Silent (toast-less) Layer 2 re-syncs reduce the visible audit trail of when
  MinSpec rewrote a marker block, traded for less friction (gated by the
  `minspec.autoBootstrap.enabled` toggle so it can be turned off).

## Risks & Mitigations

| Risk | Mechanism / anchor | Mitigation |
|---|---|---|
| Hook misjudges containment and auto-approves a non-marker edit, mutating user-authored content (breaks invariant #6) | `scripts/hooks/marker-guard.mjs` containment matcher | Default-deny by construction — the only behaviour-changing output is `allow` on *proven* containment (single unambiguous match fully inside a `minspec:*` block); every other case emits nothing and defers to the normal prompt. A bug can only fail to allow (→ prompt), never over-approve. |
| Whole-file `Write` slips an edit past the marker guard | `Write` tool bypasses span-level `Edit`/`MultiEdit` matching | Hook never auto-approves a whole-file `Write` — only `Edit`/`MultiEdit` whose every span lies inside a block; `Write` always defers to prompt. |
| Layer 2 silent re-sync overwrites content the user expected to keep | SPEC-005 FR-6 auto-bootstrap refresh path | Merge touches only generated content between markers; any change that would alter content *outside* markers still prompts, and the whole path is gated by `minspec.autoBootstrap.enabled`. |
| Composing with the src/spec gate could weaken a `deny` | runs alongside `spec-gate.sh` (DR-012) | Deny-wins composition — a `deny` from any hook still wins, so DR-012's gate is unaffected; this hook can only *add* `allow`, never override a `deny`. |

## Alternatives Considered

- **Add `CLAUDE.md`/`AGENTS.md` to a blanket `Edit` allow-list** (e.g. settings
  `permissions.allow`) — rejected: that auto-approves *any* edit to the whole
  file, including user-authored content outside the markers, directly violating
  invariant #6 (the prompt was guarding exactly that region). The marker guard is
  span-scoped instead of file-scoped precisely to keep the "never yours"
  guarantee.
- **Keep prompting, just suppress the DR-006 refresh toast** — rejected: solves
  only Layer 2; the Claude Code agent (Layer 1) would still interrupt on every
  `dr-index`/`slash-commands` update, leaving the core friction the user reported
  (2026-05-30) unaddressed.
- **Have the hook re-implement the marker merge itself / rewrite the edit** —
  rejected: scope creep and a second source of truth for marker semantics. The
  hook only *decides* (allow/defer) on containment; `refreshHarnessFiles()` and
  `adr-manager` remain the sole writers, keeping this Tier-0, pure file-system
  (DR-004).
