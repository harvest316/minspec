# DR-032: No internal DR/SPEC/EPIC numbers in user-facing output

> Triggered by: session request 2026-06-05 (Paul Harvey) — *"if a MinSpec user has
> a decision register in their project that also numbers DR-XXX then we should prevent
> any DR-XXX references from being published in templates or displayed on screen by our
> vsix to prevent confusion."*

## Context

MinSpec **dogfoods** its own SDD methodology — its developers write internal
`DR-NNN` references throughout the source (DR-012 = the approval gate, DR-003 = RCDD,
etc.). Those references belong in MinSpec's *code comments*, which never ship to or
display in a user's project.

One leaked. [`spec.ts:310`](../../packages/minspec/src/lib/spec.ts) appends a footer to
**every spec file MinSpec generates** in a user's project:

```
# Editing voids approval (hash in .minspec/approvals.json → stale); re-run "MinSpec: Approve Spec". DR-012
```

That `DR-012` is MinSpec's *private* register number. It is written into the user's
spec files (confirmed: it sits in this very repo's
[`specs/minspec/requirements.md`](../../specs/minspec/requirements.md) frontmatter) and
is rendered on screen by the spec panel. The behaviour the footer describes is fully
stated without it — the trailing tag adds nothing but MinSpec-internal provenance the
user cannot resolve.

The triggering concern is **collision**: if the user's own register has a `DR-012`,
the footer becomes a *false cross-reference* pointing at an unrelated decision. But the
collision condition is effectively **always true** — MinSpec scaffolds the user's
`{{decisionsDir}}/` and the *Create ADR* command mints `DR-001`, `DR-002`… in every
project. So "only suppress on collision" would be conditional logic guarding a
near-universal state. And even with *no* collision, a bare internal `DR-012` in the
user's file is meaningless noise. The right primitive is therefore **unconditional**:
MinSpec's user-facing output carries no specific internal DR number at all.

This is the recurring **asymmetry-gate** class (DR-003 Phase 4; the SPEC-004 /
`#115` / `#126` validator-asymmetry lineage): MinSpec has gates over the *user's*
content (spec-validator, command-references) but **no gate over its own emitted
output**. The validators check what flows *in*; nothing checks what flows *out*.

### Audited surfaces (2026-06-05)

- **Leak — 1:** the spec-footer `DR-012` above. Only `DR-<number>` literal that
  reaches a user-facing surface.
- **Clean — placeholders:** harness templates
  ([`template-registry.ts`](../../packages/minspec/src/lib/template-registry.ts)) and
  the ADR template
  ([`adr-manager.ts`](../../packages/minspec/src/lib/adr-manager.ts)) reference only the
  `DR-NNN` *pattern* / the user's own computed next number. Correct.
- **Clean — comments:** every other `DR-<number>` in `src/` is a code comment —
  internal provenance, never shipped or displayed.
- **Out of audit / exempt:** README, CHANGELOG, THIRD-PARTY-NOTICES cite real MinSpec
  DRs (DR-006/018) — but that is MinSpec's *own marketplace/legal copy describing
  itself*, linking to MinSpec's GitHub. Not a user-project collision (see OQ-2).

## Decision

1. **MinSpec emits no specific internal `DR-`/`SPEC-`/`EPIC-` number into any
   user-facing surface.** (Generalised from DR-only per OQ-1.) The only permitted forms
   in emitted/displayed output are the literal placeholder tokens `DR-NNN` / `SPEC-NNN`
   / `EPIC-NNN` (the user's own number goes there). "User-facing surface" =
   harness/scaffold templates written into the project, generated spec/ADR/INDEX
   content, webview HTML, walkthrough docs, and user-visible notification strings.
   MinSpec's own source comments and its own repo docs/specs/decisions are **not**
   user-facing and are exempt.

2. **Strip the leak.** Remove the trailing `DR-012` from the spec approval footer; keep
   the behaviour text verbatim. (Data fix alone = incomplete RCDD — the gate below is
   the actual fix.)

3. **Add a symmetric output-provenance gate (T0 invariant test).** A test renders every
   emit surface and asserts none contains a `(DR|SPEC|EPIC)-<digits>` literal
   (placeholders `DR-NNN` / `SPEC-NNN` / `EPIC-NNN` allowed). New emit paths register
   their output in the gate; a focused static scan
   covers surfaces the test cannot render (walkthrough markdown, webview HTML). A
   justified exception uses an inline `dr-ok` marker. This makes the leak
   **un-committable**, not merely patched.

4. **Unconditional, not collision-conditional.** No runtime scan of the user's register;
   no render-time rewriting. Rejected as more code guarding a near-universal state with
   no extra correctness (see Alternatives).

## Risks & Mitigations

- **Gate false-positives on legitimate copy** (README self-reference). → exempt
  MinSpec's own marketing/legal/docs/specs paths by allowlist; `dr-ok` for one-offs.
- **Gate misses an unregistered emit path** (the output test only sees what it renders).
  → pair the render test with a static scan over the designated emit-surface files, and
  require new emit functions to be added to the gate's surface list (documented in the
  test). Residual gap acknowledged, not silently hidden.
- **Over-stripping removes a genuinely useful pointer.** → if MinSpec ever must cite its
  own rationale to the user, use a *resolvable* form (full GitHub URL), never a bare
  `DR-NNN` that collides with the user's register.

## Costly to Refactor

- **The "no internal DR in output" contract (Decision 1)** — low to reverse (it only
  ever *removes* tokens from output), but it is a standing convention every future
  contributor's emitted strings must honour; its value is entirely in being enforced,
  hence the gate.
- **Gate surface enumeration** — low; a test-local list, extended as emit paths are
  added.

## Consequences

- A user's spec files and MinSpec's on-screen text never show a number that points into
  MinSpec's private register — no confusion, no false cross-reference, collision or not.
- The never-wrong signpost posture (EPIC-002 / DR-029) extends to *output provenance*:
  MinSpec does not display a reference the user cannot resolve.
- The asymmetry-gate lineage gains its output-side member: gates now cover both ingress
  (user content) and egress (MinSpec's emitted output).

## Alternatives considered

- **Conditional collision suppression** (scan user's register; rewrite refs only on
  clash). Rejected: the collision is near-universal (MinSpec scaffolds the register), so
  it is stateful machinery guarding the default case; a bare internal `DR-012` is noise
  even without a clash. Unconditional strip is simpler and at least as correct.
- **Data-only strip, no gate.** Rejected per DR-003 data-only-fix corollary — leaves the
  egress gate gap open; the next internal ref leaks the same way.
- **Blanket source grep for `DR-\d+`.** Rejected as the gate's *primary* form — it can't
  distinguish an emitted string from a code comment and would fire on the (correct)
  hundreds of internal comment refs. The render-output test is precise; the static scan
  is scoped to emit-surface files only.

## Open Questions (resolved at approval 2026-06-05)

- **OQ-1 — Generalize to SPEC-/EPIC- refs? → YES.** The gate regex is
  `(DR|SPEC|EPIC)-\d+`; placeholders `DR-NNN` / `SPEC-NNN` / `EPIC-NNN` are the only
  permitted emitted forms. Folded into Decisions 1 & 3.
- **OQ-2 — README/CHANGELOG self-reference? → KEEP EXEMPT.** Rationale (Paul Harvey):
  MinSpec is a **public repo** — anyone can look the docs up, so its own marketplace/
  legal copy may cite real MinSpec DRs and link to GitHub. The exemption is the
  allowlist in SPEC-021 FR-3 (README, CHANGELOG, THIRD-PARTY-NOTICES, `docs/**`,
  `specs/**`).
