# DR-024: Split DR-022 along the evidence line

## Context

[DR-022](DR-022.md) was accepted in-session as a T4 keystone: it reframes ceremony
around a consequence/**reach** risk profile, demotes the diff-size tier to a derived
label, supersedes [DR-020](DR-020.md), and drives a marketplace/SEO repositioning
([#86](https://github.com/harvest316/minspec/issues/86)).

Review surfaced a **rigor asymmetry**. The size axis had to *earn* acceptance with an
empirical study — [DR-009](DR-009.md) / SPEC-004, n=120, Fleiss κ=0.80. DR-022's new
**primary signal, call-graph impact-reach, is accepted on argument alone** — zero
validation — while it supersedes a risks policy and changes public positioning. The
load-bearing **risk→phase catalog** ([#89](https://github.com/harvest316/minspec/issues/89))
does not yet exist; DR-022 itself calls it "the real design surface" and rates its
ballooning risk *High*. Accepting an unvalidated model that demotes the validated
core IP (DR-018) and churns SEO/brand is **ahead of the evidence**.

But one sub-decision inside DR-022 is genuinely forced and **low-regret**: the
data-model fork. **Fork A** stores a scalar `tier`; **Fork B** stores a risk
*profile*. This forks the Tier-0 frontmatter + approval-record contract (DR-014);
choosing A-now-B-later is a **breaking migration of user data once shipped**. Nothing
has shipped, so the choice is free now — and a profile *container* is a strict
superset of a scalar tier (it can hold just `{ tier }` if reach never pans out). That
bit is decidable **independent of whether reach is any good.**

## Decision

**Split DR-022 along the evidence line.**

1. **ACCEPT (here): Fork B as the contract _direction_.** The unit of ceremony is a
   risk-profile container; `tier` becomes a derived field. Justified by the
   can't-ship-A-and-lock-users forcing function — **not** by reach's quality.
   Low-regret (superset of scalar). The *actual* frontmatter migration
   ([#90](https://github.com/harvest316/minspec/issues/90)) is sequenced **pre-first-ship,
   after reach validates** — it is **not** performed now.
2. **DOWNGRADE DR-022 `accepted → proposed`** for everything that depends on reach
   being good: the consequence/reach analyzers ([#88](https://github.com/harvest316/minspec/issues/88)),
   the risk→phase catalog (#89), screen-gated risks *enforcement*, and the
   positioning change (#86). All **gated on the reach-validation study
   ([#91](https://github.com/harvest316/minspec/issues/91))** clearing a bar
   comparable to the size study's.
3. **RESTORE DR-020 to in force (`proposed`)** as the interim risks rule. Screen-gating
   depends on the screen *existing*; superseding DR-020 before then would leave **no
   operative risks policy** — a regression. DR-022 supersedes DR-020 only on its own
   acceptance (i.e. after #91). SPEC-013 implements DR-020 in the interim.
4. **HOLD positioning (#86).** No site/marketplace copy change until #91 validates
   reach. Protects existing SEO/brand from churning on an unproven thesis.

The "can't defer the contract" argument survives the split intact — it justifies
locking the profile *shape*, which is exactly what (1) does. It never justified
stamping the whole *model* accepted.

## Consequences

**Positive:**

- **DR-020 → `proposed` (in force)** restores an operative risks policy: `superseded_by`
  cleared, banner flipped from SUPERSEDED to interim-in-force — closing the
  no-operative-policy gap that superseding DR-020 before the screen exists would have
  opened (Decision point 3).
- **No code or contract migration happens now.** The only immediate change is
  decision-status edits + keeping DR-020 as the operative risks rule — so the split
  costs almost nothing to make or to reverse.
- The accepted **Fork B contract direction** (Decision point 1) unblocks reach work to
  *validate* immediately against #91, since the profile-container shape is settled and is
  a strict superset of a scalar `tier`.

**Negative:**

- **DR-022 → `proposed`** demotes a keystone that was accepted in-session: its Decision
  now only *stands as the intended model* pending #91, its DR-020 supersession becomes
  contingent on acceptance, and its follow-ups (#88–#91, #86) remain open though now
  correctly gated. The record carries a downgraded keystone until #91 clears.
- **SPEC-013** must carry a dual-state banner — it implements **DR-020** in the interim
  while still holding `depends_on: [DR-022]` for the eventual re-scope and DR-022's
  screen-gating on acceptance — i.e. an artifact tracking two decisions at once.
- **#86 (positioning)** is annotated as blocked on #91, deferring the SEO/brand
  repositioning that DR-022 had set in motion.

## Alternatives considered

- **Keep DR-022 fully accepted.** Rejected — accepts an unvalidated keystone that
  demotes validated core IP and changes public positioning before evidence.
- **Reject Fork B / keep scalar tier.** Rejected — forks the Tier-0 contract;
  deferring guarantees a breaking post-ship migration; the Fork B container is
  low-regret now.
- **Downgrade _all_ of DR-022, including the contract direction.** Rejected — the
  contract choice is the one genuinely time-sensitive bit (free only pre-ship);
  folding it into the #91 gate risks shipping scalar-A and locking it in.

## Costly to Refactor

Mostly cheap to reverse — **except one durable bit.** The governance moves (Decision
points 2–4: DR-022→`proposed`, DR-020 restored, #86 held) are pure status/banner edits
across DR-022, DR-020, and SPEC-013; unwinding them on a passing #91 is the one-line
status flip per artifact already described in R1. **ADR-filter: yes, undoable in <1 day.**

The exception is the **accepted Fork B contract _direction_** (Decision point 1). The
direction itself is locked here, but its blast radius is bounded by what this DR
explicitly defers: the actual frontmatter + approval-record migration ([#90](https://github.com/harvest316/minspec/issues/90))
is sequenced **pre-first-ship, after reach validates — not performed now**. Because the
profile container is a strict superset of a scalar `tier` (it can hold just `{ tier }`),
even a *failing* #91 leaves no expensive undo — #90 is simply never run (R3). The only
genuinely costly-to-refactor object — a shipped frontmatter contract that would force a
breaking user-data migration to change (the very risk DR-024 exists to avoid, per the
Fork A/B analysis in Context) — is held behind the pre-ship gate and so is not adopted
by this decision.
