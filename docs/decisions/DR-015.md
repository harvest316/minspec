# DR-015: Where the agent dispatch system ships

## Context

The agent system — headless dispatch of `claude -p` per GitHub issue
(`scripts/dispatch-issue.sh`), role prompts (`scripts/roles/`), and inbox triage
(`scripts/triage-inbox.sh`) — is today **bash dev-tooling** living in `scripts/`.
It is used to develop the monorepo itself. Its security model (untrusted-body
delimiters, no-credential execution isolation) is already governed by **DR-008**.

The question this DR answers is **packaging**, not security. The gate question
— *is agent dispatch a shipped runtime feature, or dev-tooling?* — is now
**resolved by the user: it is a shipped runtime feature**, and it is needed in
**both** MinSpec and ScroogeLLM. The dev-tooling option is closed. Two facts
constrain where it lives:

1. **It is Tier 1** (DR-004 / DR-014): it spawns processes, shells out to `gh`,
   creates git worktrees. Per DR-014 it **cannot** live in Tier-0 `@aiclarity/shared`
   — that would void MinSpec's air-gapped marketplace claim.
2. **Both MinSpec and ScroogeLLM could want it**, but if both ship their own
   dispatcher they will both react to the same GitHub issue → **double-dispatch /
   label race** (the open overlap from DR-014).

The user's framing: ship the agent system **in both extensions**, **or** as a
**third extension**.

## Decision

**Gate the packaging on one question:** *is agent dispatch a shipped runtime feature,
or developer tooling for building this monorepo?*

- **Dev-tooling only (status quo)** → it stays in `scripts/`. Both extensions
  "share" it by living in the same repo. Nothing to package, no overlap. **Done.**
- **Shipped runtime feature** → a **third extension** (not duplicated in both).

**Recommendation: the third extension**, conditional on dispatch being productized.
Until then, keep it in `scripts/`.

### Why third extension over "in both"

- **Single home = single dispatch owner**, which dissolves the double-dispatch
  overlap from DR-014 *by construction* — there is exactly one thing that can react
  to an issue.
- **Tier-1 trust profile is isolated.** MinSpec stays genuinely Tier-0/air-gapped
  for the marketplace; users opt into autonomous, credential-touching execution by
  installing a clearly-separate product (DR-004 consent model).
- **No duplicated dispatcher** to keep in sync across two codebases.
- **DR-008's Layer-2 (no-credential execution isolation) lands in one place** — the
  agent extension is the natural and only home for the sandbox work.

### Why NOT "in both extensions"

- Two dispatchers → must coordinate a lock label or owner anyway; you reinvent
  "single owner" without the clean separation.
- Forces Tier-1 process-spawn code into MinSpec, eroding its air-gapped story.
- Doubles maintenance of the riskiest (security-sensitive) subsystem.

## The third-extension pitch

If productized, the agent system completes a **three-verb product line**:

> **MinSpec — Specify.** What to build (specs, tiers, traceability). Tier 0, offline.
> **ScroogeLLM — Optimize.** What it costs (route, cache, measure). Tier 2.
> **Agent extension — Execute.** Get it done (issue → spec → code → PR, tier-gated HITL). Tier 1.

Positioning points:

- **It's the hub that demonstrates the whole family.** It consumes the shared
  classifier to **tier-gate** work (DR-014: T1–T2 auto-dispatch, T3–T4 → human
  approval — already the model in CLAUDE.md), and it can **route its own agent calls
  through ScroogeLLM** so cheap tasks run on cheap models. Owning all three shows the
  flywheel: spec defines tier → tier gates dispatch → dispatch routed for cost.
- **Honest trust boundary as a feature.** "MinSpec never touches your network or
  credentials. When you want autonomy, you install the agent runtime — explicitly,
  separately, with its own permissions." That clarity is a selling point, not a tax
  (mirrors DR-004 tiers and DR-010's opt-in-value framing).
- **Tier-gated HITL is the differentiator.** Not "an AI that does everything" — an
  agent that knows, via the same classifier the IDE uses, when to ask a human. The
  validator-gate / fabricated-researcher cautionary angle reinforces "agents you can
  trust because they stop."
- **The Pro pack becomes the full pipeline** — three extensions, Specify → Optimize
  → Execute, one install.

Naming is **out of scope** for this DR (track separately with the existing naming
decision). Working placeholder only.

### Downsides (accepted / to weigh)

- A third marketplace listing + icon + README + LICENSE + privacy policy to maintain.
- Brand fragmentation risk across three products.
- The Pro pack grows to three members; pack version-lockstep (DR-014 §4) now spans
  three bundled `@aiclarity/shared` copies.

## Consequences

- **No code yet.** This DR records the decision *shape*; nothing ships until the gate
  question is answered (productized vs dev-tooling) — tracked as an issue.
- If third extension proceeds: new `packages/<agent-ext>` (Tier 1), a Specify cycle
  of its own, and it is where DR-008 Layer-2 isolation is implemented.
- The Tier-1 agent runtime is **never** merged into `@aiclarity/shared` (DR-014 §1).
- Shared classifier (DR-014) is the dependency that lets the agent extension tier-gate.

## Alternatives considered

- **Ship in both extensions** — rejected for shipped use (double-dispatch, Tier-0
  erosion, duplicated maintenance); acceptable only while it remains dev-tooling,
  which is the status-quo branch above.
- **Keep forever in `scripts/`** — viable and correct *if* it never becomes a user
  feature. This DR does not force productization; it decides where it goes *if* it is.
- **Fold into ScroogeLLM** (since both are network/Tier-aware) — rejected: conflates
  cost-optimization with autonomous execution; muddies the three-verb story and makes
  dispatch hostage to the proxy's release cadence.
