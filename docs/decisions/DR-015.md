# DR-015: Where the agent dispatch system ships

## Context

The agent system — headless dispatch of `claude -p` per GitHub issue
(`scripts/dispatch-issue.sh`), role prompts (`scripts/roles/`), and inbox triage
(`scripts/triage-inbox.sh`) — is today **bash dev-tooling** living in `scripts/`.
It is used to develop the monorepo itself. Its security model (untrusted-body
delimiters, no-credential execution isolation) is already governed by **DR-008**.

The question this DR answers is **packaging**, not security. The gate question
— *is agent dispatch a shipped runtime feature, or dev-tooling?* — is now
**resolved by the user: it is a shipped runtime feature**, and it is needed in
**both** MinSpec and ScroogeLLM. The dev-tooling option is closed. Two facts
constrain where it lives:

1. **It is Tier 1** (DR-004 / DR-014): it spawns processes, shells out to `gh`,
   creates git worktrees. Per DR-014 it **cannot** live in Tier-0 `@aiclarity/shared`
   — that would void MinSpec's air-gapped marketplace claim.
2. **Both MinSpec and ScroogeLLM could want it**, but if both ship their own
   dispatcher they will both react to the same GitHub issue → **double-dispatch /
   label race** (the open overlap from DR-014).

The user's framing: ship the agent system **in both extensions**, **or** as a
**third extension**.

## Decision

The gate question is resolved: agent dispatch **is** a shipped runtime feature wanted
by both MinSpec and ScroogeLLM. It ships as a **third extension** — a single Tier-1
"Execute" extension shared by both — **not** duplicated inside each.

The `scripts/` bash remains the **dev-time** path for building this monorepo; the
third extension is the **productized** path. The two coexist: dev-tooling does not
ship in any `.vsix`, the third extension is the only thing users install.

### Why third extension over "in both"

- **Single home = single dispatch owner**, which dissolves the double-dispatch
  overlap from DR-014 *by construction* — there is exactly one thing that can react
  to an issue.
- **Tier-1 trust profile is isolated.** MinSpec stays genuinely Tier-0/air-gapped
  for the marketplace; users opt into autonomous, credential-touching execution by
  installing a clearly-separate product (DR-004 consent model).
- **No duplicated dispatcher** to keep in sync across two codebases.
- **DR-008's Layer-2 (no-credential execution isolation) lands in one place** — the
  agent extension is the natural and only home for the sandbox work.

### Why NOT "in both extensions"

- Two dispatchers → must coordinate a lock label or owner anyway; you reinvent
  "single owner" without the clean separation.
- Forces Tier-1 process-spawn code into MinSpec, eroding its air-gapped story.
- Doubles maintenance of the riskiest (security-sensitive) subsystem.

## The third-extension pitch

The agent system completes a **three-verb product line**:

> **MinSpec — Specify.** What to build (specs, tiers, traceability). Tier 0, offline.
> **ScroogeLLM — Optimize.** What it costs (route, cache, measure). Tier 2.
> **Agent extension — Execute.** Get it done (issue → spec → code → PR, tier-gated HITL). Tier 1.

Positioning points:

- **It's the hub that demonstrates the whole family.** It consumes the shared
  classifier to **tier-gate** work (DR-014: T1–T2 auto-dispatch, T3–T4 → human
  approval — already the model in CLAUDE.md), and it can **route its own agent calls
  through ScroogeLLM** so cheap tasks run on cheap models. Owning all three shows the
  flywheel: spec defines tier → tier gates dispatch → dispatch routed for cost.
- **Honest trust boundary as a feature.** "MinSpec never touches your network or
  credentials. When you want autonomy, you install the agent runtime — explicitly,
  separately, with its own permissions." That clarity is a selling point, not a tax
  (mirrors DR-004 tiers and DR-010's opt-in-value framing).
- **Tier-gated HITL is the differentiator.** Not "an AI that does everything" — an
  agent that knows, via the same classifier the IDE uses, when to ask a human. The
  validator-gate / fabricated-researcher cautionary angle reinforces "agents you can
  trust because they stop."
- **The Pro pack becomes the full pipeline** — three extensions, Specify → Optimize
  → Execute, one install.

`aiclarity.agent-execute` is a **working technical id only** (the "Execute" verb).
The **user-facing name + marketplace positioning are deferred pending marketing
analysis** — keyword/SEO + competitive scan before any public name is committed
(mirrors the MinSpec/ScroogeLLM naming process). Do not treat `agent-execute` as the
product name on any public surface.

### Downsides (accepted / to weigh)

- A third marketplace listing + icon + README + LICENSE + privacy policy to maintain.
- Brand fragmentation risk across three products.
- The Pro pack grows to three members; pack version-lockstep (DR-014 §4) now spans
  three bundled `@aiclarity/shared` copies.

## Consequences

**Positive:**

- Single dispatch owner: neither MinSpec nor ScroogeLLM ships a dispatcher; both
  depend on / coexist with the third extension → no double-dispatch, the DR-014
  label-race overlap is closed *by construction*.
- The Tier-1 agent runtime is **never** merged into `@aiclarity/shared` (DR-014 §1),
  so MinSpec keeps its genuinely Tier-0/air-gapped marketplace claim.
- DR-008 Layer-2 (no-credential execution isolation) lands in exactly one home —
  `packages/agent-execute` — instead of being duplicated across two codebases.
- Shared classifier (DR-014) becomes the dependency that lets the agent extension
  tier-gate (T1–T2 auto-dispatch, T3–T4 → human approval), demonstrating the
  three-verb flywheel: spec defines tier → tier gates dispatch → dispatch routed
  through ScroogeLLM for cost.
- `scripts/` dispatch survives as dev-tooling for building this monorepo — not
  deleted, not shipped in any `.vsix`.

**Negative:**

- New `packages/agent-execute` (`aiclarity.agent-execute`, Tier 1) + extension-pack
  member must be created and carry its own Specify cycle, marketplace listing, icon,
  README, LICENSE, and privacy policy (the "Downsides" list above).
- Pro pack grows from two to three members; version-lockstep (DR-014 §4) now spans
  three bundled `@aiclarity/shared` copies that must stay in sync.
- Brand fragmentation risk across three separately-listed products (MinSpec /
  ScroogeLLM / Execute).
- The riskiest (security-sensitive) subsystem now has its own release cadence and
  publishing surface to keep current alongside the other two.

## Alternatives considered

- **Ship in both extensions** — rejected for shipped use (double-dispatch, Tier-0
  erosion, duplicated maintenance); acceptable only while it remains dev-tooling,
  which is the status-quo branch above.
- **Keep forever in `scripts/` only** — rejected once dispatch was confirmed a shipped
  feature wanted by both extensions; `scripts/` survives as dev-tooling but cannot be
  the product surface (not bundled in any `.vsix`).
- **Fold into ScroogeLLM** (since both are network/Tier-aware) — rejected: conflates
  cost-optimization with autonomous execution; muddies the three-verb story and makes
  dispatch hostage to the proxy's release cadence.

## Risks & Mitigations

| Risk | Mechanism | Mitigation |
|---|---|---|
| Three-product brand fragmentation dilutes the line | A third separate marketplace listing + icon + README (the "Downsides" list) splits attention three ways | Position as one three-verb pipeline (Specify → Optimize → Execute) and ship the Pro pack (`extension-pack`) as the single-install bundle that re-unifies them |
| Version skew across three `@aiclarity/shared` copies | DR-014 §4 lockstep now spans three bundled copies of the Tier-0 shared package (classifier + contract types); a stale copy in `agent-execute` mis-tiers work and breaks tier-gating | Treat the pack as the version-lockstep boundary (DR-014 §4); the agent extension depends on the *same* classifier the IDE uses, so a single shared release covers all three |
| Tier-0 erosion creeps back in | Pressure to "just embed" dispatch into MinSpec for convenience would re-introduce process-spawn/`gh`-shelling Tier-1 code into the air-gapped extension (the rejected "in both" branch) | DR-014 §1 hard rule: Tier-1 runtime never enters `@aiclarity/shared`; this DR makes `packages/agent-execute` the only home, so there is no in-MinSpec target to embed into |
| Security work concentrated in one unfinished package | DR-008 Layer-2 (no-credential execution isolation) is *not yet implemented*; it is deferred to `agent-execute`'s own Specify cycle (#56) | Single home means one place to land and audit the sandbox; dispatch stays governed by DR-008's existing untrusted-body delimiters + no-credential isolation until Layer-2 ships |
| Naming committed prematurely on a public surface | `aiclarity.agent-execute` is a working technical id only; treating it as the product name pre-empts the marketing/SEO scan | Decision explicitly defers user-facing name + positioning pending keyword/competitive analysis (mirrors MinSpec/ScroogeLLM naming) |

## Costly to Refactor

Moderate blast radius, but mostly forward-looking — the third extension does **not
yet exist** (its Specify cycle and DR-008 Layer-2 are tracked in #56), so reversing
the *decision* today costs little: delete the plan, keep `scripts/` as dev-tooling
(the rejected status-quo branch). **ADR-filter: undoable in <1 day right now.**

The expensive-to-undo moment is **after** `aiclarity.agent-execute` is published and
installed: at that point the single-dispatch-owner guarantee (DR-014 overlap closure)
and the Tier-0/air-gapped marketplace claim both depend on dispatch living *outside*
MinSpec and ScroogeLLM. Folding it back into either extension would then mean
re-introducing Tier-1 process-spawn code into a published Tier-0 listing (voiding the
air-gap claim users installed on) and re-litigating the double-dispatch label race —
a marketplace-relisting + trust-story rewrite, not a code move. So: cheap to reverse
pre-publish, costly post-publish.

## Status

Accepted. Agent dispatch ships as a third Tier-1 "Execute" extension shared by MinSpec
and ScroogeLLM; `scripts/` remains dev-tooling. Builds on DR-004 (tiers), DR-008
(dispatch security), DR-014 (shared boundary, double-dispatch overlap). Follow-up
tracked in #56 (productization confirmed; naming + Specify cycle to schedule).
