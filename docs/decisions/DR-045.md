# DR-045: Host-IDE background tasks are Layer-1 visibility, not a Layer-2 substrate

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). This ADR records how the host IDE's background-task pattern interacts with the
> SPEC-019 mode split, and the one-clause prose edit it owes FR-10. **Nothing here is
> built** — SPEC-019 is a Specify-phase record; this refines its requirements, it ships no
> code.

## Context

The host IDE (the Claude Code VS Code extension) now surfaces **pending background tasks**
in the IDE whenever it spins up a batch of background agents — a fan-out queue the human can
glance at and interrupt. This raised a design question against
[SPEC-019](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md)'s
dispatch model: how does it interact with **FR-9** (manual Layer-1 vs autonomous Layer-2
mode split) and **FR-10** (no container runtime → degrade to Layer-1 manual, never "off")?

The load-bearing fact: that host-IDE task runner executes its tasks **in the extension host
— the user's own credentialed process** (`$HOME`, `gh`, `CLOUDFLARE_API_TOKEN`,
`~/.claude`, wrangler). By [SPEC-019](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md)'s
own definitions it is therefore **Layer-1** (credentialed, unsandboxed), dressed with an
IDE visibility layer. It is **not** a sandbox, does **not** satisfy the FR-6 attestation
boundary, and is precisely the category error [DR-017](DR-017.md)/FR-1 names: *a vsix is the
user's credentialed process*.

FR-9 and FR-10 are written as a **binary** environmental read — *container present* → Layer-2
autonomous; *no container* → Layer-1 manual. The host-IDE background-task runner introduces a
**third environmental fact neither FR enumerates**: an async runner that is
**autonomy-capable but boundary-incapable** — it can fan out unattended work, but only inside
the credentialed host. The risk is an implementer (or operator) mistaking it for a valid
degrade substrate for *autonomous* dispatch when no container is present.

## Decision

1. **Classify the host-IDE background-task runner as Layer-1 + IDE visibility — never a
   Layer-2 substrate.** Its presence is *autonomy-capable but boundary-incapable*; it never
   satisfies FR-6 attestation, and its availability **never flips autonomy on**. The axis
   FR-9 splits on is the **boundary** (credential-free sandbox + attestation + broker seam),
   not parallelism or visibility. Many tasks + a visible queue ≠ the isolation boundary.

2. **FR-9 needs no edit — it is a mental mapping.** The host-IDE fan-out maps onto FR-9's
   **Manual dispatch (Layer-1)** row (human-initiated, trusted body, credentialed, no
   sandbox); the pending-task panel is a UX layer *over* that row, orthogonal to the mode
   split.

3. **FR-10 owes one explicit clause.** FR-10's binary text must name the third fact:
   *an ambient async runner (e.g. the host IDE's background tasks) is not a container; its
   presence degrades to manual **visibility**, never to unsandboxed **autonomy**. Autonomous
   dispatch requires an attested container (FR-6) regardless of any host-provided task
   runner.* This makes explicit what FR-10's principle ("no docker downgrades autonomy,
   never the boundary") already implies, closing the misread where the IDE runner is taken
   as a no-container fallback for autonomous work.

4. **FR-11's typed-fallback union is unchanged.** Host-IDE-present is still `no-runtime`
   *for autonomy purposes* — no new discriminated-union member. Only the **affordance**
   rendered on `no-runtime` gets richer: when the host is Claude Code, surface the "install a
   container runtime to enable autonomous dispatch" nudge **in the task panel** rather than a
   dead toast.

5. **Adopt the pending-task panel as a UX precedent for FR-12 HITL — but keep the human
   signpost singular.** The panel is good prior art for surfacing in-flight FR-12 dispatches
   and `needs-review` blocks. It collides with MinSpec's own next-task principle
   ([SPEC-012](../../specs/minspec/SPEC-012-next-task-resolver/requirements.md): a single
   deterministic next *human* task, not a list). Resolution: the pending-task list is the
   **agent queue** (its own pane); the human **signpost stays one task**. The borrowed
   list-UI must not turn the human signpost into a list.

6. **Reinforce R4 — visible feels safe, isn't.** A polished pending-task panel is an
   [R4](../../specs/agent-execute/SPEC-019-execution-substrate/requirements.md) magnet
   ("operator wires the loop treating Layer-1 as sufficient"): it makes unattended,
   credentialed, unsandboxed fan-out *feel* monitored. Record this as a named sub-case of R4;
   the mitigation is unchanged — cron/unattended **requires** an attested sandbox (FR-6),
   however good the task panel looks.

## Alternatives considered

- **Treat the host-IDE task runner as a no-container degrade substrate for autonomy.**
  Rejected: it runs credentialed in the ext host — it keeps autonomy while dropping the
  boundary, the exact inversion FR-10 forbids and the FR-1 category error.
- **Leave FR-9/FR-10 untouched (rely on the implied principle).** Rejected for FR-10: the
  binary prose invites the misread; one clause removes the ambiguity at near-zero cost.
- **Add a new FR-11 fallback reason for "host runner present."** Rejected: it is still
  `no-runtime` for autonomy; a new union member would split one concept into two and
  complicate the gate for no behavioural gain. Richer *affordance*, same *reason*.

## Consequences

- **Positive:** the autonomy-capable-but-boundary-incapable host runner is named; FR-10's
  invariant is stated, not merely implied; the pending-task UX is harvested for the
  agent-queue pane without weakening the singular human signpost or the attestation gate.
- **Cost (owed):** a **one-clause prose edit to SPEC-019 FR-10** (+ a short FR-12 UX note on
  the queue-pane-vs-signpost split). Editing SPEC-019 while `status: specifying` is in place,
  pre-approval — no hash-lock ceremony yet. Both edits fold into the **SPEC-019 → sealbox
  migration cascade** ([DR-044](DR-044.md) follow-up), so they land with the rename rather
  than as a separate churn.
- **No code.** SPEC-019 is unbuilt; this changes a requirements record only (FR-16 / Tier-0
  untouched).
