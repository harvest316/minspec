# DR-039: Goals drive priority (the human dial is epic.order + goals; auto-WSJF later)

## Context

[DR-019](DR-019.md) makes next-task priority a deterministic DAG; the one thing the DAG
cannot derive — relative importance between independent branches — it lifts into the
human-set `epic.order` field. Three gaps surfaced this session:

1. **Is business value computed, and correctly?** Yes, but in the wrong place: a **WSJF**
   scorer exists (`minspec.scoreWsjf`, `backlog.ts`) — human-entered, 4 dimensions × 1–10
   — but it scores **GitHub issues only**, is **not wired to the resolver**, and asks for
   four numbers per issue. That is exactly the ceremony the "just enough human" principle
   says the human won't actually perform.
2. **Do we need a human priority flag?** `epic.order` + a per-artifact `priority:` already
   *are* one — human-edited, version-controlled, deterministic — but there is no UI to set
   them and the resolver does not yet read `priority:`.
3. **Do we need documented core values/goals to map tasks against?** There is **no goals
   artifact**. Strategic intent lives only as prose (DR-001, keyword research). The
   constitution's `## Goals` does not exist.

## Decision

**The human priority dial is `epic.order` (coarse, per-epic) + project Goals (thematic,
per-artifact). WSJF is demoted; an *auto-derived* WSJF is recorded as a future upgrade.**

### 1. Goals are a first-class, human-authored artifact

A `## Goals` section in `.minspec/constitution.md`. Each goal has a **stable ID** (`G-1`,
`G-2`, …) and is **human-ranked** (order = importance). Goals are prose intent authored by
the human — never generated, never LLM-scored.

### 2. Artifacts map to goals by an explicit, human-curated ref

An artifact (spec/DR/epic/issue) declares `goal: G-2` in frontmatter, exactly as it
declares `epic:` (DR-013). The engine **reads** the ref; it never **infers** which goal a
task serves. Mapping is human judgement lifted into data — the same discipline DR-019 §3
applies to `epic.order`.

### 3. Goals + epic.order are deterministic tie-break dimensions

DR-019 §2's within-class order becomes `(epic.order, goal-rank, priority, artifact-id)`.
`goal-rank` is the rank of the artifact's goal in the constitution list. This is a
**tie-break inside a severity class only** — it **never** overrides `gate-violation` or
`blocked-ready`. A high-value blocked task is still blocked; surfacing it ahead of its
blocker would make the signpost lie. `priority:` remains an optional fine override, rarely
hand-set.

### 4. WSJF is demoted, not removed

Manual WSJF (`minspec.scoreWsjf`) **stays** as an optional, standalone backlog tool —
it has discoverability value (developers evaluating SDD tools search "WSJF"). But it is
**not wired to the resolver**, does not seed `priority:`, and is not the priority
mechanism. The signpost works fully without anyone ever opening it.

### 5. Auto-derived WSJF is the recorded future upgrade

The valuable part of WSJF is not the manual scoring — it is the *economic shape*
(`Cost of Delay ÷ size`), the only model here that weighs value **against effort** and
**time-criticality** (so small high-value work isn't starved by a big important epic).
MinSpec can compute all four inputs **deterministically** from signals it already has:

| WSJF input | Auto-source |
|---|---|
| Job **size** | the tier classifier — its *measured* strength ([DR-021](DR-021.md); ~96% vs size labels) |
| Business **value** | goal-rank (Decision §3) |
| **Risk** reduction / opportunity | consequence analyzers (SPEC-023, PR #198 merged) |
| **Time** criticality | milestones / `depends_on` (SPEC-012 FR-3b) |

That makes WSJF a **zero-ceremony, deterministic** value-aware sort within a severity
class — and finally gives the under-used classifier a real home. **Future, not v1**
(#262): v1 ships `epic.order` + goals with SPEC-012; auto-WSJF layers on once goals,
consequence analyzers and milestones are all wired. It stays deterministic, so it never
violates DR-019's no-LLM-ranking rule.

### 6. Surface the dial in the explorer

`epic.order` and `goal-rank` are exposed as **drag-and-drop reorder** in the explorer
panes (#261). Reordering is low-frequency, so mouse/DnD is acceptable under the amended
keyboard rule — this is the per-epic/goal equivalent of WSJF's per-issue scoring, at a
fraction of the input cost.

## Consequences

**Positive:**

- The human supplies *just enough* signal — rank a handful of epics and goals, map a task
  to a goal — instead of scoring four numbers per issue. Maximum direction, minimum
  ceremony.
- Priority stays a pure deterministic function (DR-019): goals/epic.order are read, never
  inferred; auto-WSJF (future) is computed from deterministic signals, not an LLM.
- A documented goals artifact closes the "strategic intent lives only in prose" gap and
  gives every artifact an auditable line to a project goal.
- The classifier gets a real downstream consumer (auto-WSJF size input), turning a
  measured-but-under-used signal into priority value.

**Negative:**

- A forgotten `goal:` ref or stale goal-rank silently mis-ranks (same residual as
  `epic.order` in DR-019) — no deterministic rescue; relies on human review.
- Two value systems coexist transitionally (manual WSJF on issues vs goals/epic.order on
  the DAG) until/unless auto-WSJF unifies them — accepted to keep WSJF's discoverability.
- Goals add one more human-maintained frontmatter dimension; under-use degrades it to
  noise (mitigated: goals are optional; absent `goal:` just drops that tie-break term).

## Alternatives considered

- **Keep manual WSJF as the priority mechanism.** Rejected — four scores per issue is
  ceremony the human won't do; it scores issues, not the DAG; it is unused signal.
- **Remove WSJF entirely.** Rejected — it has discoverability/SEO value and is cheap to
  keep; the signpost simply ignores it.
- **Let an LLM score "how much does this task serve goal G-2".** Rejected outright —
  reintroduces exactly the subjective ranking DR-019 forbids. Mapping is human-curated.
- **Build auto-WSJF now (v1).** Rejected — it depends on goals + consequence analyzers +
  milestones all being wired and on SPEC-012 shipping; recorded as a future upgrade (#262)
  to keep v1 scope tight.
- **A separate `docs/core-values.md`.** Rejected — a second source of truth; the
  constitution already exists as the principles/invariants anchor.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | **Value overrides a gate** — a high goal-rank floats a blocked task above its blocker, signpost lies. | Low · High | Decision §3 — goals/epic.order are tie-breaks *within* a severity class only; `gate-violation`/`blocked-ready` always dominate. T0 test asserts ordering precedence. |
| R2 | **Stale/forgotten `goal:` or goal-rank** silently mis-ranks. | Med · Med | Same class as DR-019's `epic.order` residual; soft surfacing (LLM may *suggest* a `goal:`, human accepts), never a silent engine guess. |
| R3 | **Auto-WSJF inputs drift** (classifier mis-sizes, analyzer noise) skewing the future economic sort. | Med (future) · Med | Deferred to #262; auto-WSJF is a *tie-break*, never overrides gates; classifier size accuracy is the documented dependency (DR-021). |
| R4 | **Goals become box-ticking** — every task tagged `goal: G-1`. | Med · Low | Goals are optional and human-reviewed; a degenerate single-goal mapping just collapses the tie-break term, costing nothing structurally. |

## Costly to Refactor

Moderate, and front-loaded onto data. The engine changes (read `goal:`, add a tie-break
term) are a small pure-function edit — cheap to revert. The cost is the **propagated
frontmatter**: once `goal:` refs and ranked Goals are authored across artifacts, dropping
the model orphans that data, like DR-019's `epic.order`. **ADR-filter: borderline —
trivially reversible before goals are populated; > 1 day once they are.** Auto-WSJF (§5)
is independently reversible (a future, isolated tie-break stage). Recorded now, before any
`goal:` data exists, precisely because that is the cheap moment.
