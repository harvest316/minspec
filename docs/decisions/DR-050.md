# DR-050: Shelling the user's own authenticated `gh`/CLI on explicit consent does not violate the zero-network-in-core invariant

> **Born `proposed` per DR-029** (acceptance is a separate human act). Ratifies the
> Tier-0 boundary pre-committed by #365 (ruleset-advisor `invariants.test.ts`
> allowlist entry). Relates to [DR-004](DR-004.md) (Tiered Network Consent Model).

## Context

[AGENTS.md invariant #1](../../AGENTS.md) states: **"MinSpec makes zero network calls in
its core path."** [DR-004](DR-004.md) replaced the old binary "no network" rule with a
three-tier model: Tier 0 (fully offline, no `http`/`https`/`fetch`/`net` imports in
`packages/minspec` or `packages/shared`), Tier 1 (local tool delegation — `gh`, `claude`
— no network code in the extension itself), and Tier 2 (MinSpec Pro / ScroogeLLM).
DR-004's Tier-0 note already cited `gh` CLI issue-creation as an existing shell-out; it
did not, however, settle whether **a shell-out that causes the user's CLI to make a
network call** itself touches invariant #1.

Issue #356 catalogued three options for the ruleset-advisor feature:

- **(a) Allowed-in-core** — gh-shell is simply permitted in the core path; no
  further ceremony.
- **(b) Push to Tier-1 / SealBox** — move any gh-dependent action outside core
  into the agent-execute extension.
- **(c) Tier-gated opt-in with audit** — allow in core but require explicit
  per-action consent and a zero-network default fallback.

PR #365 (ruleset-advisor implementation) pre-committed an interpretation by adding
`gh` to the `child_process` allowlist in `invariants.test.ts`. That interpretation needs
to be explicitly ratified — or overturned — before the test becomes precedent.

## Decision

**Adopt a hybrid of options (a) and (c).** Shelling the user's OWN installed and
authenticated CLI (`gh`, in the same category as `git` which MinSpec already shells) does
**not** count as a MinSpec core network call, **provided all three conditions hold**:

1. **MinSpec opens no socket itself.** The extension invokes the user's tool via
   `child_process`; the tool owns its own auth and network transport. No `http`,
   `https`, `fetch`, or `net` import may appear in `packages/minspec` or
   `packages/shared` — the existing Tier-0 lint gate (DR-004) is unchanged.

2. **Explicit per-action consent.** Every action that will cause a network call
   (e.g. the ruleset READ and the CREATE in the ruleset-advisor feature) must sit
   behind an explicit user opt-in in the UI before it fires. Nothing may trigger
   autonomously.

3. **Zero-network default path intact.** The always-available fallback (e.g. a
   docs link, a copy-to-clipboard prompt) makes **no** network call. The consented
   gh-shell path is strictly additive — the feature degrades gracefully to zero
   network if the user declines or `gh` is absent.

### What this does NOT permit (boundary)

- MinSpec making direct HTTP/HTTPS calls from extension code in any tier.
- Bundling network libraries (`node-fetch`, `axios`, `got`, etc.) in
  `packages/minspec` or `packages/shared`.
- Background or autonomous network activity — every network-triggering action
  requires an in-context user gesture.
- **Autonomous or background egress of user artifacts, spec content, or telemetry
  initiated by MinSpec itself.** Those remain Tier-2 surfaces per DR-004
  (ScroogeLLM / Pro pack only). This is distinct from the user consciously pushing
  their *own* content to their *own* remote via their *own* consented `gh` — that
  is the user's action, not MinSpec's: it is explicitly permitted and is precisely
  what the ruleset-create feature and the existing parking-lot feature
  (`packages/minspec/src/lib/parking-lot.ts`, `gh issue create` with user-authored
  title/body) do. The load-bearing distinction is **who initiates the egress and
  whose data goes where**: MinSpec-initiated exfiltration / telemetry is forbidden;
  a user-directed push of the user's own content to the user's own repository is
  permitted under conditions 2 and 3.

## Amendment (2026-07-01)

**Refines condition 2.** The original condition 2 required explicit per-action
consent for *every* network call, treating the ruleset READ and the ruleset
CREATE identically. In practice this over-gated: it forced a consent toast in
front of a **read-only config probe** that egresses nothing, producing an
"are you sure you want me to check?" prompt with no user-facing risk and no
decision to make. This amendment splits condition 2 by the *class* of the
network action.

**The load-bearing distinction — read-only-config-probe vs. mutation/egress:**

- **Read-only config PROBE → autonomous, no prior consent toast.** A `gh api`
  **GET** of the repo's OWN settings (the `.../rulesets` list read in the
  ruleset-advisor feature; likewise any capability probe such as `gh --version`
  / `gh auth status`) reads back the repository's own configuration and egresses
  **no** user artifacts, spec content, or telemetry. It is the same class of
  action as MinSpec shelling `git fetch` — a read of remote state the user
  already owns, initiated to *decide whether there is anything to offer*. Under
  this amendment such a probe MAY run **autonomously** on init (once `gh` is
  ready and the repo resolves) **without** a preceding consent toast. It still
  honours condition 1 (no socket in the extension; delegated to the user's `gh`)
  and condition 3 (a zero-network fallback remains).

- **Mutating / egressing action → consent-gated (unchanged in spirit).** Any
  network action that **WRITES** to the user's repository or otherwise egresses
  data — the ruleset CREATE (`gh api -X POST .../rulesets`), and a fortiori any
  push of user content — remains behind an **explicit, in-context user gesture**.
  In the reworked ruleset-advisor, the create toast IS that consent: the user's
  click on **"Create ruleset"** is the per-action opt-in for the mutation. Nothing
  writes autonomously. The global/one-time-toggle prohibition in *Costly to
  Refactor* below still stands — this amendment does not replace per-action
  consent for mutations with a global setting; it only exempts the read-only
  probe from needing a *prior* toast at all.

**Crisp rule:** read-only-config-probe = autonomous OK; mutation/egress =
consent-gated (and the mutation's own confirm toast is that consent). This keeps
the Tier-0 story honest — the extension still opens no socket, and MinSpec still
*autonomously* egresses nothing — while removing a consent prompt that guarded a
zero-egress read.

**Status unchanged** (`proposed`); this amendment refines the decision's
condition 2 rather than reversing the decision. Materialised in
`packages/minspec/src/lib/ruleset-advisor.ts` (autonomous `isGhReady` +
`hasRequiredChecksRuleset`; consent-gated `createRequiredChecksRuleset`) and
`packages/minspec/src/commands/init.ts` (`offerRulesetAdvisory` — probe runs
without a toast; a create-offer toast is shown only when the probe finds no
qualifying ruleset).

## Rationale

The intent of invariant #1 — and of DR-004's Tier-0 rule — is that MinSpec never
*autonomously* phones home or leaks user data through its core path. A consented,
user-initiated action through the user's own credentialed CLI is the **user's** network
call, not MinSpec's: the extension process opens no socket, holds no auth token, and
sends no data the user has not explicitly triggered. This is exactly the same posture as
shelling `git` — nobody argues `git push` violates the offline invariant because MinSpec
called `git`.

The `gh` CLI is a local, user-owned, user-authenticated binary. Treating it differently
from `git` would require a principled distinction that does not exist at the process or
data-egress level. Both are delegations to a user tool; both honour the "no network code
in the extension itself" boundary.

This reading is consistent with the "visibility-via-UI counts as auditable" principle
(stored in project memory): opt-in + settings text + status-bar indication = compliant
with invariant #8 (auditability). Applying the same logic: consent + graceful-fallback +
zero-socket-in-extension = compliant with invariant #1.

## Consequences

**Positive:**

- Ratifies the `invariants.test.ts` allowlist entry added in #365 (ruleset-advisor),
  unblocking that feature without a tier-boundary change or a SealBox dependency.
- Establishes a durable, reviewable precedent for any future `gh`-shell feature: three
  conditions, all mechanical (no-socket, consent-gate, zero-network-default), easy to
  gate in CI.
- Keeps the Tier-0 air-gapped story honest: the extension process itself still makes
  zero outbound connections; the user's CLI accounts for its own activity.

**Negative:**

- Introduces a second class of Tier-0 action (direct-offline vs. consent-gated
  gh-shell) that a contributor can confuse — placing a gh-shell without the required
  consent gate "in the same bucket" as a plain offline call.
- Each future gh-shell feature carries the same UX tax as Tier-1 features (DR-004
  Consequences): probe CLI availability at activation, degrade gracefully, never hard-error.

## Alternatives Considered

- **Option (b) — push all gh-shell to Tier-1 / SealBox.** Rejected: this forces the
  ruleset-advisor (a lightweight init feature) to carry a full SealBox dependency for
  what is a single, user-consented shell-out. Tier-1 / SealBox is the right home for
  *autonomous* agent dispatch; it is disproportionate for a manually-triggered read.
- **Option (a) alone — unconditional allowance.** Rejected: without the consent gate
  (condition 2) and the zero-network-fallback (condition 3), invariant #1's
  intent — no autonomous network — is not preserved. The hybrid adds those two guards
  without adding a tier.
- **Strict interpretation — any gh-shell violates Tier-0.** Rejected: this would require
  MinSpec to remove its existing `gh`-shell parking-lot feature (noted in DR-004 Context)
  and would draw a distinction between `git` (allowed) and `gh` (banned) that has no
  defensible technical basis. Both delegate to a local, user-owned binary; the
  extension process makes no outbound connection in either case.

## Costly to Refactor

Low. The three conditions are each individually reversible: a future DR could tighten
condition 1 (ban all child_process CLIs from core), relax condition 2, or eliminate
the fallback requirement (condition 3). None of those changes
would require a code rewrite beyond the affected feature's activation path. The
`invariants.test.ts` allowlist is the only structural artefact that materialises this DR
in code; it is a <1-day edit. The precedent this DR sets (gh-shell = user delegation,
not MinSpec egress) is load-bearing only if future contributors rely on it — this record
is where they find it.

> **Note — global/one-time consent toggle NOT licensed here.** Relaxing condition 2
> by replacing per-action consent with a global "allow MinSpec network" setting is
> **not** permitted under this DR — it would re-admit the autonomous-feeling egress
> that per-action consent exists to prevent (a user who toggled a setting once, months
> ago, may not recall they consented to today's action). Any move from per-action to
> global consent requires its own DR to evaluate the trade-off explicitly. Per-action
> consent is the load-bearing guard in this decision, not an implementation detail.
