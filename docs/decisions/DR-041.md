# DR-041: Canonical term for review-gate artifacts is "Approvable"

## Context

MinSpec tracks five artifact kinds that all share one property: a human must
read and approve them before work proceeds. The signpost (DR-019) surfaces the
single next human task from this set. The approval gate (DR-012 / DR-034) hashes
and locks them. Nothing in the codebase named the set as a whole.

Two candidate terms were evaluated:

| Term | Assessment |
|---|---|
| **Review item / review surface** | "Surface" already refers to the *pane/view*, not the item. "Item" is generic. Neither teaches vocabulary. |
| **Approvable** | Precise — directly names the defining property (carries an approval gate). Maps to the `type Approvable` union in TypeScript. Doubles as user vocabulary: users can say "which Approvables are pending?" to their Claude Code and be understood. |

## Decision

The canonical collective term is **Approvable** — in UI labels, prose, docs, and
code alike.

Specific applications:

- **UI:** signpost header reads "Approvables" (or "Next Approvable"). Queue label:
  "Pending Approvables."
- **Type system:** `type Approvable = Spec | DR | PR | Epic | Issue` in
  `packages/shared/src/contracts/`.
- **Prose / docs:** use "Approvable" (capitalised as a product noun) when referring
  to the set; lowercase "approvable" as an adjective.
- **User vocabulary:** teach it explicitly in onboarding copy — "Specs, DRs, PRs,
  Epics, and Issues are all Approvables: MinSpec tracks which one needs your eye
  next."

## Rationale

Shared vocabulary between the tool and the user reduces friction in both
directions: the user can describe what they need ("show me pending Approvables")
and the UI can label it without circumlocution. A coined term that matches the
type name is also grep-stable — it won't collide with prose words.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Term feels jargony to new users | Defined on first appearance in onboarding UI; tooltip on hover. |
| Conflicts with existing `approvals.json` semantics | Distinct: `approvals.json` is the *store*; an *Approvable* is an artifact type. Clarified in the constitution glossary. |

## Consequences

- Constitution gains a `## Glossary` section with this entry as the first term.
- `packages/shared/src/contracts/` gains `type Approvable` union (follow-up issue).
- All future UI copy and docs use "Approvable" consistently.

## Amendment (2026-06-30)

DR-047 (independent AI review across every Approvable surface) expands this `Approvable`
union. The original union was `Spec | DR | PR | Epic | Issue` (five types). It is amended
to also include **Plan**, **design.md**, **tasks.md**, and **constitution invariant** —
the remaining doc types MinSpec produces or consumes:

```ts
type Approvable =
  | Spec | Plan | DesignDoc | TasksDoc        // spec + its downstream Plan artifacts
  | DR | ConstitutionInvariant                // decision + constitution
  | Epic | Issue | PR;                         // grouping, intake, code
```

**AI-reviewed vs human-gated — a superset/subset relationship (the reconciliation
DR-047's own AI review flagged).** DR-047 introduces a key distinction this amendment
records:

- **AI-reviewed set = the FULL union above (all doc types, always).** Every Approvable —
  *including* non-human-approvable types like **Issue** — receives an independent AI
  substance review. AI review is the universal floor.
- **Human-approvable set = the critical subset.** Only the high-criticality types (Spec,
  **Plan**, DR, constitution invariant, PR-to-main, Epic) carry a *human* approval gate.
  Plan is human-gated because a bad plan drives a bad implementation — spec-level blast
  radius (DR-047 Decision 5). The low-criticality types (design.md, tasks.md, Issue) MAY
  be configured (dev opt-in) to auto-accept on AI greenlight with no human. Issue is never
  human-gated.

So the `Approvable` type name now denotes the **AI-reviewed** superset; the human gate
applies to a *subset* of it. This reconciles the membership delta DR-047's earlier draft
left unstated (it added Plan + constitution and appeared to drop Issue — neither is
dropped; Issue is AI-reviewed but not human-gated). The `## Glossary` constitution entry
and the `type Approvable` union (Follow-ups below) adopt the amended membership.

`status` stays **accepted** — this is an expansion of an accepted decision, not a reversal;
the canonical term and its rationale are unchanged.
