# DR-012: Hard-to-skip HITL spec gate

## Context

MinSpec's review steps are advisory. A spec can move to `implementing` without a
human ever reading it, and Claude Code in **bypass-permissions** mode will edit
source regardless of spec state. Three weaknesses:

1. **Skippable review.** No enforced gate between "spec written" and "code
   written". Phase status (`plan: done`) is set by tooling/AI, not by a human
   approval act.
2. **No completeness floor.** A T3/T4 spec with a UX surface can reach
   implement with no mockup; an API change with no payload/schema; an
   architectural change with no diagram. Nothing checks.
3. **Low visibility.** Sidebar shows tier + status but no progress/roll-up, so
   drift is easy to miss.

Research (Claude Code hooks): a `PreToolUse` hook returning exit-code 2 /
`permissionDecision: "deny"` **blocks Edit/Write even under bypass-permissions**,
and receives `tool_input.file_path` on stdin. This is the only primitive that
survives bypass mode. `ExitPlanMode`'s prettified approval panel is **model-only**
— not invocable by an extension — so a reusable native review UI is not available.

## Decision

Ship a three-part HITL gate. Ceremony stays proportional (Constitution: ceremony
∝ complexity) — **enforcement applies to T3/T4 only**.

### 1. Content-hash approval state — `.minspec/approvals.json`
Approval is an explicit human act recorded as:
`{ specId: { specHash, approvedAt, tier } }` where `specHash = sha256(spec file bytes)`.
Editing the spec after approval changes the hash → approval auto-invalidates.
`sha256sum` (bash hook) and Node `crypto` agree on raw-bytes hashing → hook and
extension see the same truth. Flat layout (one file/spec) in v1; spec-kit
multi-file hashing is a follow-up.

### 2. Tier-aware completeness validator — `lib/spec-validator.ts` (pure)
`validateSpec(spec, config) → { complete, violations[], detectedAspects, declaredAspects }`.
- Required-phase sections (per `phaseMappings[tier].requiredPhases`) must be non-empty.
- T3/T4 must carry acceptance criteria.
- **Aspect-conditional artifacts.** Aspects come from frontmatter `aspects:` (explicit)
  unioned with keyword detection (fallback):
  - `ux` → mockup required (image link, `## UX`/`## Mockup`, or fenced/ascii block)
  - `api` → schema/payload required (fenced `json`/`ts`/`openapi`, or interface)
  - `data` → schema/ER required (`sql`/`mermaid erDiagram`/table)
  - `architecture` → diagram required (`mermaid`/`.puml`/`.drawio`/`## Diagram`)
  Severity is tier-scaled (error for T3/T4, warning for T2). `complete` = no errors.
Approval **refuses** when `!complete`.

### 3. Src-edit gate — `scripts/hooks/spec-gate.sh` (PreToolUse on Edit|Write|MultiEdit)
Self-contained bash (jq + sha256sum, grep fallback). Logic:
- Always **allow** edits to `specs/**`, `docs/**`, `.minspec/**`, `*.md`,
  `scripts/**`, `.claude/**` (never lock out spec/review editing).
- For code edits: scan `specs/**` for `tier: T3|T4` + `status: implementing`;
  if any such spec is unapproved (missing or hash-mismatch in approvals.json),
  **deny** with a reason naming the spec.
- **Kill-switch:** env `MINSPEC_GATE_OFF=1` disables the gate (escape hatch).

### 4. Progress indicator (#4)
Spec tree gains per-spec phase-progress + task-% and a synthetic roll-up root
("N specs · X% complete"); status bar shows active-spec %.

## Consequences

**Positive:**
- Bypass-permissions mode no longer bypasses review for T3/T4: the
  `scripts/hooks/spec-gate.sh` PreToolUse `deny` (exit-2) is the one primitive
  that survives bypass mode, so `status: implementing` T3/T4 specs cannot be
  coded against until approved in `.minspec/approvals.json`. Friction is
  intentional and tier-gated; T1/T2 untouched (Constitution: ceremony ∝ complexity).
- Approval is content-bound: `specHash = sha256(spec file bytes)` means editing
  the spec after approval auto-invalidates it → re-review is forced after edits.
- A completeness floor now exists: `validateSpec` refuses approval when
  `!complete`, so T3/T4 UX specs can't reach implement with no mockup, API specs
  with no schema/payload, etc. (the gaps named in Context point 2).
- `sha256sum` (bash hook) and Node `crypto` hash identical raw bytes → hook and
  extension agree on approval truth with no drift.

**Negative:**
- New frontmatter convention `aspects:` becomes part of the spec format (optional;
  keyword detection in `lib/spec-validator.ts` is the fallback if absent) —
  another field authors and the validator must understand.
- Productization debt: this DR wires the gate for *this* repo only. End users get
  no protection until `minspec init` installs `spec-gate.sh` and merges the
  PreToolUse entry into their `.claude/settings.json` (parked as
  [#37](https://github.com/harvest316/minspec/issues/37)) — so the gate's value is
  currently asymmetric (maintainers protected, users not).
- The kill-switch `MINSPEC_GATE_OFF=1` is a deliberate escape hatch that also
  fully disables enforcement — an env var in a shell profile silently re-opens the
  bypass hole this DR closes.

## Risks & Mitigations

|------|-----------|------------|
| Validator asymmetry — checks values that exist but not that they *should* exist | This is the exact class that stranded SPEC-004 (`validateSpec` flagged dangling `epic:` refs but never *missing* ones, per DR-003). Residual gap: weak keyword recall — tracked as a validator-hardening concern, not closed here. |
| Gate locks out spec/review editing itself | A naïve PreToolUse block on Edit/Write could deny edits to the very spec that needs approving. | `spec-gate.sh` always-allows `specs/**`, `docs/**`, `.minspec/**`, `*.md`, `scripts/**`, `.claude/**` (Decision §3). |
| Stub code shipped behind a `[x]` task while spec moves toward done | Happened in-tree: `statusCommand` was a hardcoded stub at 113/113 "complete" (Addendum). The spec gate enforces *spec* completeness only, never reads implementation code. | Companion code-completeness gate added in the Addendum (stub-marker scan, spec-traced files only, T3/T4 warn-level) — tracked in SPEC-006. |
| Kill-switch / always-allowed paths abused to route code edits around the gate | `MINSPEC_GATE_OFF=1` disables everything; `scripts/**` is always-allowed, so code placed under `scripts/` is never gated. | Accepted as escape-hatch tradeoff; gate targets *source* edits under bypass mode, not a hostile author. Documented, not enforced. |

## Alternatives Considered

- **Rely on phase-status flags (`plan: done`) as the gate** — rejected: status is
  set by tooling/AI, not a human approval act (Context point 1), so it provides no
  evidence a human read the spec. Content-hash approval in
  `.minspec/approvals.json` makes approval an explicit, content-bound human act.
- **Reuse `ExitPlanMode`'s native prettified approval panel** — rejected as
  technically impossible: it is **model-only**, not invocable by an extension
  (Context). The prettified review webview is therefore deferred to a custom
  webview ([#36](https://github.com/harvest316/minspec/issues/36)).
- **Ordinary permission prompt / advisory review** — rejected: does not survive
  **bypass-permissions** mode, where Edit/Write proceed regardless. Only a
  `PreToolUse` hook returning exit-2 / `permissionDecision: "deny"` blocks under
  bypass (Context, research).
- **Whole-tree stub scan + all-tier + hard PreToolUse block on stubs** (for the
  Addendum code-completeness gate) — rejected as disproportionate (Addendum, "Not
  in scope"); chosen instead: spec-traced files only, T3/T4, warn-level.
- **Semantic "is this actually implemented" judgement** — rejected: needs AI,
  violating Tier 0 / DR-004 (no AI, no network); markers-only scan chosen.

## Costly to Refactor

Mostly reversible, with two sticky points. **Reversible in <1 day:** the gate is
opt-out by env var (`MINSPEC_GATE_OFF=1`) and the hook/validator are isolated
files (`scripts/hooks/spec-gate.sh`, `lib/spec-validator.ts`) — deleting them
restores advisory behavior. **Sticky:** (1) the `.minspec/approvals.json` schema
`{ specId: { specHash, approvedAt, tier } }` and the **raw-bytes** `sha256`
contract are shared between the bash hook and the Node extension — changing the
hash basis (e.g. to multi-file spec-kit hashing, the parked follow-up) silently
invalidates every stored approval and must be migrated, not just edited; (2) once
`minspec init` ships the gate to end users (#37), the PreToolUse entry lives in
*their* `.claude/settings.json`, so any later change to the deny contract becomes a
distributed migration, not a one-repo edit. ADR-filter: undoing the gate in this
repo is <1 day; undoing it after distribution and the `aspects:` frontmatter
convention spread across specs is not.

## Out of scope / follow-ups (parked)

- **#2 Prettified review webview** (text-select + comment pins written back to
  spec). ExitPlanMode not reusable → must be a custom webview. Deferred.
- **Productization:** `minspec init` must install `spec-gate.sh` + merge the
  PreToolUse entry into the user's `.claude/settings.json`, else end users don't
  get the gate. This DR wires only *this* repo. Deferred.
- Spec-kit (multi-file) approval hashing.

## Addendum — 2026-05-30: code-completeness (stub) gate — companion to spec-completeness (SPEC-006)

The gate above enforces *spec* completeness (required sections present). It does
not look at implementation code, so a task can be marked `[x]` and a spec moved
toward done while the code behind it is a stub. This happened in-tree:
`statusCommand` was a hardcoded stub while `tasks.md` showed 113/113 complete —
because (a) `[x]` is a hand assertion nothing verifies, (b) the stubbed
click-handler had no task line of its own, and (c) no MinSpec check scans source
for stub markers. The global "post-completion grep check" is an agent habit, not
a product feature.

### Decision (sibling gate to the spec-completeness gate)

Add a **code-completeness gate** mirroring the tier-aware philosophy here:

1. **Stub detection.** A pure file-system scan for stub markers: `TODO`,
   `FIXME`, `HACK`, `throw new Error('not implemented')` / `notImplemented`,
   empty/placeholder bodies, `test.skip`/`it.skip`. Marker set configurable.
2. **Scope = spec-traced files only.** Scan files mapped to the active spec via
   `.minspec/traceability.json` — not the whole tree. Untraced files are out of
   scope (avoids noise).
3. **Tier-gated T3/T4** (matches this DR). T1/T2 untouched.
4. **Warn, not hard-block.** Surface as editor diagnostics + block marking the
   spec/phase **done** while traced files contain stubs. Does not block edits
   (distinct from the PreToolUse src-edit block above). User override via the
   same settings posture.

### Rationale

- Closes the exact hole that shipped a stub: completeness must mean *code*
  complete, not just *spec* complete.
- Tier-gated + warn-level keeps ceremony ∝ complexity (Constitution).
- Pure file-system (Tier 0 / DR-004) — no AI, no network.

### Not in scope

- Whole-tree scanning, all-tier enforcement, hard PreToolUse blocking on stubs —
  considered and rejected as disproportionate.
- Semantic "is this actually implemented" judgement (needs AI) — markers only.

Implementation tracked in SPEC-006.
