# DR-012: Hard-to-skip HITL spec gate

## Context

MinSpec's review steps are advisory. A spec can move to `implementing` without a
human ever reading it, and Claude Code in **bypass-permissions** mode will edit
source regardless of spec state. Three weaknesses:

1. **Skippable review.** No enforced gate between "spec written" and "code
   written". Phase status (`plan: done`) is set by tooling/AI, not by a human
   approval act.
2. **No completeness floor.** A T3/T4 spec with a UX surface can reach
   implement with no mockup; an API change with no payload/schema; an
   architectural change with no diagram. Nothing checks.
3. **Low visibility.** Sidebar shows tier + status but no progress/roll-up, so
   drift is easy to miss.

Research (Claude Code hooks): a `PreToolUse` hook returning exit-code 2 /
`permissionDecision: "deny"` **blocks Edit/Write even under bypass-permissions**,
and receives `tool_input.file_path` on stdin. This is the only primitive that
survives bypass mode. `ExitPlanMode`'s prettified approval panel is **model-only**
‚Äî not invocable by an extension ‚Äî so a reusable native review UI is not available.

## Decision

Ship a three-part HITL gate. Ceremony stays proportional (Constitution: ceremony
‚àù complexity) ‚Äî **enforcement applies to T3/T4 only**.

### 1. Content-hash approval state ‚Äî `.minspec/approvals.json`
Approval is an explicit human act recorded as:
`{ specId: { specHash, approvedAt, tier } }` where `specHash = sha256(spec file bytes)`.
Editing the spec after approval changes the hash ‚Üí approval auto-invalidates.
`sha256sum` (bash hook) and Node `crypto` agree on raw-bytes hashing ‚Üí hook and
extension see the same truth. Flat layout (one file/spec) in v1; spec-kit
multi-file hashing is a follow-up.

### 2. Tier-aware completeness validator ‚Äî `lib/spec-validator.ts` (pure)
`validateSpec(spec, config) ‚Üí { complete, violations[], detectedAspects, declaredAspects }`.
- Required-phase sections (per `phaseMappings[tier].requiredPhases`) must be non-empty.
- T3/T4 must carry acceptance criteria.
- **Aspect-conditional artifacts.** Aspects come from frontmatter `aspects:` (explicit)
  unioned with keyword detection (fallback):
  - `ux` ‚Üí mockup required (image link, `## UX`/`## Mockup`, or fenced/ascii block)
  - `api` ‚Üí schema/payload required (fenced `json`/`ts`/`openapi`, or interface)
  - `data` ‚Üí schema/ER required (`sql`/`mermaid erDiagram`/table)
  - `architecture` ‚Üí diagram required (`mermaid`/`.puml`/`.drawio`/`## Diagram`)
  Severity is tier-scaled (error for T3/T4, warning for T2). `complete` = no errors.
Approval **refuses** when `!complete`.

### 3. Src-edit gate ‚Äî `scripts/hooks/spec-gate.sh` (PreToolUse on Edit|Write|MultiEdit)
Self-contained bash (jq + sha256sum, grep fallback). Logic:
- Always **allow** edits to `specs/**`, `docs/**`, `.minspec/**`, `*.md`,
  `scripts/**`, `.claude/**` (never lock out spec/review editing).
- For code edits: scan `specs/**` for `tier: T3|T4` + `status: implementing`;
  if any such spec is unapproved (missing or hash-mismatch in approvals.json),
  **deny** with a reason naming the spec.
- **Kill-switch:** env `MINSPEC_GATE_OFF=1` disables the gate (escape hatch).

### 4. Progress indicator (#4)
Spec tree gains per-spec phase-progress + task-% and a synthetic roll-up root
("N specs ¬∑ X% complete"); status bar shows active-spec %.

## Consequences

- Bypass mode no longer bypasses review for T3/T4. Friction is intentional and
  tier-gated; T1/T2 untouched.
- Approval is content-bound ‚Äî re-review is forced after edits.
- New frontmatter convention `aspects:` (optional; keyword fallback if absent).

## Out of scope / follow-ups (parked)

- **#2 Prettified review webview** (text-select + comment pins written back to
  spec). ExitPlanMode not reusable ‚Üí must be a custom webview. Deferred.
- **Productization:** `minspec init` must install `spec-gate.sh` + merge the
  PreToolUse entry into the user's `.claude/settings.json`, else end users don't
  get the gate. This DR wires only *this* repo. Deferred.
- Spec-kit (multi-file) approval hashing.

## Addendum ‚Äî 2026-05-30: code-completeness (stub) gate ‚Äî companion to spec-completeness (SPEC-006)

The gate above enforces *spec* completeness (required sections present). It does
not look at implementation code, so a task can be marked `[x]` and a spec moved
toward done while the code behind it is a stub. This happened in-tree:
`statusCommand` was a hardcoded stub while `tasks.md` showed 113/113 complete ‚Äî
because (a) `[x]` is a hand assertion nothing verifies, (b) the stubbed
click-handler had no task line of its own, and (c) no MinSpec check scans source
for stub markers. The global "post-completion grep check" is an agent habit, not
a product feature.

### Decision (sibling gate to the spec-completeness gate)

Add a **code-completeness gate** mirroring the tier-aware philosophy here:

1. **Stub detection.** A pure file-system scan for stub markers: `TODO`,
   `FIXME`, `HACK`, `throw new Error('not implemented')` / `notImplemented`,
   empty/placeholder bodies, `test.skip`/`it.skip`. Marker set configurable.
2. **Scope = spec-traced files only.** Scan files mapped to the active spec via
   `.minspec/traceability.json` ‚Äî not the whole tree. Untraced files are out of
   scope (avoids noise).
3. **Tier-gated T3/T4** (matches this DR). T1/T2 untouched.
4. **Warn, not hard-block.** Surface as editor diagnostics + block marking the
   spec/phase **done** while traced files contain stubs. Does not block edits
   (distinct from the PreToolUse src-edit block above). User override via the
   same settings posture.

### Rationale

- Closes the exact hole that shipped a stub: completeness must mean *code*
  complete, not just *spec* complete.
- Tier-gated + warn-level keeps ceremony ‚àù complexity (Constitution).
- Pure file-system (Tier 0 / DR-004) ‚Äî no AI, no network.

### Not in scope

- Whole-tree scanning, all-tier enforcement, hard PreToolUse blocking on stubs ‚Äî
  considered and rejected as disproportionate.
- Semantic "is this actually implemented" judgement (needs AI) ‚Äî markers only.

Implementation tracked in SPEC-006.
