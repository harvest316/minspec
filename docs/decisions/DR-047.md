# DR-047: Independent AI review across every Approvable surface

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records the context, decision, and follow-ups; no code ships with this DR.

## Context

**DR-033 §6** built an independent reviewer-agent for **PRs** (code): a fresh-context
opus agent — never the one that authored the branch — critiques each diff before the
human approves or auto-merges it. Its verdict is recorded as an `ai-review:*` status
check and label; the signpost (SPEC-010/SPEC-012, #182) may only surface a PR as a human
task if `ai-review:pass` ∧ mergeable ∧ checks-green.

**DR-041** canonised the collective term for all review-gate artifacts: an **Approvable**
is a Spec, DR, PR, Epic, or Issue that carries a human approval gate. MinSpec already
tracks five Approvable types; DR-033 §6's independent review covers only one (PR/code).

**The four remaining types are worse for rubber-stamping, not better:**

1. **AI-authored by design.** The architect agent, Specify agent, and Propose-Constitution
   agent draft Specs, DRs, and constitution invariants. The human approval that follows is
   the **only check** — there is no independent review before the author hands work to the
   human. Self-attestation (author → human, no third voice) is a weaker signal than an
   adversarial second read.

2. **Existing gates check structure, not substance.** `validateSpec`, the dangling-ref
   checker, and the symmetric-validator class (DR-032, SPEC-021) verify schema conformance
   and reference resolution. None audits whether the *reasoning* is sound, whether an FR
   contradicts a requirement, whether a DR overlooks a risk class, or whether a
   constitution invariant is internally inconsistent.

3. **Friction-reduction lowered scrutiny.** The HITL approval UX (non-modal toast,
   inherit-AI-consent from upstream, low-friction Apply — #214 / DR-041) was designed to
   remove *unnecessary* ceremony; a side effect is that the path of least resistance for a
   fatigued reviewer is an uncritical "approve." The 2026-06-29 rubber-stamp session
   (#344–349: 7 defects waved through, including never-wrong violations and an egress-leak
   hole) demonstrates this is not a theoretical risk.

DR-033 §6's architectural choices — independent agent (fresh context), advisory not
gating, auto-loop on request-changes, always-on — are correct and should be extended, not
redesigned. This DR generalises the pattern.

## Decision

**1 — Generalise the independent reviewer from PR-only to every Approvable type.**

For each Approvable kind (Spec, Plan, DR, constitution invariant, Epic, and PR), an
independent opus agent — fresh context, never the author of the artifact — critiques the
artifact for **substance** before the human approval gate opens:

- Spec: are FRs internally consistent, grounded in the context, scoped correctly for the
  tier, free of unresolved OQs that block implementation?
- Plan: does the approach match the spec's FRs, are T0 invariant tests sequenced first,
  are risks called?
- DR: are the alternatives genuinely considered, is the decision reversible within its
  Costly-to-Refactor claim, are follow-ups materialised (DR-023)?
- Constitution invariant: is the invariant testable, non-contradicting of existing
  invariants, correctly scoped to the tier?
- Epic: are member artifacts consistent with the epic goal, is the goal statement
  measurable?
- PR/code (already in DR-033 §6): substance + security audit; unchanged.

Same shape as DR-033 §6 for each type: verdict + findings block, `request-changes`
auto-loops to the **author agent** (bounded — default 2 cycles, then escalate to the
human with `agent-escalated`), an `approve` verdict ends the cycle and the Approvable
proceeds to its human gate. **ADVISORY** — the independent reviewer advises; the human
**always** holds the final approval keystroke. The reviewer can never approve, merge, or
modify an artifact (never-wrong / HITL).

**2 — Signpost-contract generalisation.**

The next-human-task signpost (SPEC-010/SPEC-012, #182) currently presents a PR as a
human task only if `ai-review:pass` (DR-033 §6). That predicate generalises:

> The signpost MUST present **any** Approvable as a human task only if the
> corresponding AI reviewer has greenlit it for its type.

An un-reviewed or `ai-review:changes` Spec, DR, or constitution invariant MUST NOT
appear in the human queue — surfacing it re-exposes the human to the exact
rubber-stamp surface this stage removes. This extends DR-033 §6's signpost contract
from PR-only to all Approvable types.

**3 — Ordering invariant: doc Approvables reviewed before the PR that implements them.**

When a PR contains or implements an Approvable doc (e.g. a spec, DR, or constitution
change checked in alongside its implementation), **that doc must be AI-reviewed and
greenlit before the PR's code review runs**. Code is judged against its spec; reviewing
code against an un-reviewed approvable is inverted — the spec may still have defects
that would change the code.

Precedence rule: **approvable-doc AI review → code AI review → human gate**. A PR
carrying an un-greenlit doc-Approvable is blocked from entering the code-review stage
until the doc is greenlit (or escalated).

**4 — Recording: extend the `ai-review:*` family per type.**

Reuse the `ai-review:*` status-check + label family (DR-033 §6, #342), scoped per
Approvable type:

| Status check | Label | Meaning |
|---|---|---|
| `ai-review/spec` | `ai-review:spec:pass` / `:changes` / `:pending` / `:escalated` | Spec substance verdict |
| `ai-review/plan` | `ai-review:plan:pass` / `:changes` / … | Plan substance verdict |
| `ai-review/dr` | `ai-review:dr:pass` / `:changes` / … | DR substance verdict |
| `ai-review/constitution` | `ai-review:constitution:pass` / `:changes` / … | Constitution invariant verdict |
| `ai-review/epic` | `ai-review:epic:pass` / `:changes` / … | Epic substance verdict |
| `ai-review` | `ai-review:pass` / `:changes` / … | PR/code verdict (existing, DR-033 §6) |

Named `ai-review:*` not `ai-approved:*` — the AI *advises*; only the human *approves*
(never-wrong / HITL). The distinction **AI-greenlit** (reviewer verdict = pass) vs
**Approvable** (greenlit ∧ prior-stage gates clear ∧ human-gate open) holds for all
types, mirroring DR-033 §6's PR wording.

## Rationale

The rubber-stamp session (#344–349) is a measured, concrete failure: 7 live defects in
merged work, including never-wrong violations, cleared by a fatigued human. The
independent reviewer pattern exists exactly to close this gap — its value is the
separation of "author" from "reviewer." Restricting it to PRs while AI-authored Specs
and DRs receive no independent read is a structural asymmetry. The cost of the
generalisation is proportional (one reviewer-agent per Approvable type, same script
shape), and the ADVISORY / HITL / never-auto-approve constraints are identical.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Review latency slows the human's queue | Reviewer runs async before the Approvable enters the signpost queue; it never blocks the author from continuing work on the next task. |
| Reviewer LLM disagrees with valid design choices (false `request-changes`) | Bounded auto-loop (2 cycles); on persistent disagreement the issue escalates to the human as `agent-escalated` — the human is still the arbiter. |
| Per-type reviewer prompts drift out of sync with artifact schemas | Reviewer prompts are part of the same MinSpec harness; a validator checks that a prompt exists for each registered Approvable type. |
| The ordering invariant (§3) blocks a fast PR merge when a doc is large | The doc reviewer runs in parallel with CI/lint; only the code-review stage is sequenced after the doc greenlight, not the human merge. In practice the doc review completes before CI. |
| Recording per-type labels creates label proliferation | Labels are prefixed `ai-review:<type>:*`; the existing `ai-review:*` set is unchanged. Proliferation is bounded by the fixed set of Approvable types. |

## Costly to Refactor

Low-medium. The reviewer-agent runner (`review-branch.sh` + `review-decide.sh`, DR-033
§6 / #342) is designed to be trigger-agnostic. Adding per-type reviewer prompts and a
dispatch path per Approvable type extends the existing seam rather than redesigning it.
The signpost-predicate generalisation (§2) is a filter change in SPEC-012's resolver;
the ordering invariant (§3) is a gate added to the dispatch chain. No new egress or
credential surface is introduced.

## Consequences

- The human approval gate is no longer the **first** independent read for any Approvable
  type — it becomes the **second** (after AI greenlight). Human attention concentrates on
  resolving genuine disagreements, not scanning for structural defects the AI already
  caught.
- AI-authored Specs and DRs gain the same adversarial second-read that PR diffs already
  have; the rubber-stamp failure class (#344–349) becomes structurally harder to repeat.
- The signpost queue becomes a **pre-filtered, AI-greenlit set** across all types — the
  human's next task is always something an independent agent already found sound.
- The ordering invariant eliminates the category of "code reviewed against a spec that
  still had defects," which is the root of at least two of the #344–349 audit findings.

## Alternatives considered

- **Keep PR-only, add human-prompt nag for non-PR Approvables.** Rejected: a nag targets
  attention and motivation; the rubber-stamp session shows both can fail under fatigue.
  A structural independent review is the only reliable gate.
- **Block approval until a human reads for N seconds (time-gate).** Rejected: a time
  gate is gameable and does not catch substance defects; the independent reviewer catches
  what the human skims over.
- **Run the reviewer as the same agent that authored the artifact.** Rejected: the value
  of the pattern is the separation of author and reviewer (DR-033 §6, "independence is the
  value"). Self-review is worth ~0.
- **Make the reviewer gating (auto-reject, block merge).** Rejected: never-wrong / HITL
  — the human always holds the final gate. The reviewer is advisory.
