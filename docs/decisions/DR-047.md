# DR-047: Independent AI review across every Approvable surface

> **Born `proposed` per DR-029** (acceptance is a separate human act — *MinSpec: Accept
> ADR*). Records the context, decision, and follow-ups; no code ships with this DR.

## Context

**DR-033 §6** built an independent reviewer-agent for **PRs** (code): a fresh-context
opus agent — never the one that authored the branch — critiques each diff before the
human approves or auto-merges it. Its verdict is recorded as an `ai-review:*` status
check and label; the signpost (SPEC-010/SPEC-012, #182) may only surface a PR as a human
task if `ai-review:pass` ∧ mergeable ∧ checks-green.

**DR-041** canonised the collective term for all review-gate artifacts: an **Approvable**
is a Spec, DR, PR, Epic, or Issue that carries a human approval gate. MinSpec already
tracks five Approvable types; DR-033 §6's independent review covers only one (PR/code).
DR-041's union is **amended by this DR** (see DR-041 Amendment 2026-06-30) to include
Plan, design.md, tasks.md, and constitution invariant — the full set of doc types MinSpec
produces or consumes.

**Two sets, decoupled (the key distinction this DR introduces).** AI review and human
approval are not the same set:

- **AI-reviewed set = ALL document types, always.** Every doc MinSpec produces or consumes —
  spec, plan, design.md, tasks.md, DR, constitution invariant, epic, **issue**, PR — gets an
  independent AI substance read, *including types that carry no human approval gate* (e.g. an
  Issue is an intake stub: AI-reviewed, never human-gated). AI review is the universal floor.
- **Human-gated set = the critical subset.** Only the high-criticality Approvables (spec,
  **plan**, DR, constitution invariant, PR-to-main, epic) require a human approval keystroke
  after AI greenlight. Plan is human-gated because a bad plan drives a bad implementation —
  spec-level blast radius. Low-criticality docs (design.md, tasks.md, issue) MAY be
  configured to auto-accept on AI greenlight — no human (Decision 5, dev opt-in).

Whether an AI-reviewed doc needs a *human* gate depends on its criticality and the dev's
coverage setting (Decisions 5–6), not on whether it was AI-reviewed. Every approvable is
AI-reviewed; only the critical subset is human-gated.

**The remaining types are worse for rubber-stamping, not better:**

1. **AI-authored by design.** The architect agent, Specify agent, and Propose-Constitution
   agent draft Specs, DRs, and constitution invariants. The human approval that follows is
   the **only check** — there is no independent review before the author hands work to the
   human. Self-attestation (author → human, no third voice) is a weaker signal than an
   adversarial second read.

2. **Existing gates check structure, not substance.** `validateSpec`, the dangling-ref
   checker, and the symmetric-validator class (DR-032, SPEC-021) verify schema conformance
   and reference resolution. None audits whether the *reasoning* is sound, whether an FR
   contradicts a requirement, whether a DR overlooks a risk class, or whether a
   constitution invariant is internally inconsistent.

3. **Friction-reduction lowered scrutiny.** The HITL approval UX (non-modal toast,
   inherit-AI-consent from upstream, low-friction Apply — #214 / DR-041) was designed to
   remove *unnecessary* ceremony; a side effect is that the path of least resistance for a
   fatigued reviewer is an uncritical "approve." The 2026-06-29 rubber-stamp session
   (#344–349: 7 defects waved through, including never-wrong violations and an egress-leak
   hole) demonstrates this is not a theoretical risk.

DR-033 §6's architectural choices — independent agent (fresh context), advisory not
gating, auto-loop on request-changes, always-on — are correct and should be extended, not
redesigned. This DR generalises the pattern.

## Decision

**1 — AI review runs on EVERY document type MinSpec produces or consumes.**

For each doc type — Spec, Plan, design.md, tasks.md, DR, constitution invariant, Epic,
**Issue**, and PR — an independent opus agent — fresh context, never the author of the
artifact — critiques the artifact for **substance**. This is the **AI-reviewed set**: it
is universal and includes types that have **no** human approval gate (Issue). For
human-gated types the review runs before the human approval gate opens; for non-gated
types (Decision 5) it informs auto-accept.

- Spec: are FRs internally consistent, grounded in the context, scoped correctly for the
  tier, free of unresolved OQs that block implementation?
- Plan: does the approach match the spec's FRs, are T0 invariant tests sequenced first,
  are risks called?
- design.md (downstream Plan artifact): does the design realise the plan's approach, are
  contracts defined before implementation, are the slice boundaries coherent?
- tasks.md (downstream Plan artifact): do the tasks cover the plan, are they ordered with
  T0 invariant tests first, is each task checkable?
- DR: are the alternatives genuinely considered, is the decision reversible within its
  Costly-to-Refactor claim, are follow-ups materialised (DR-023)?
- Constitution invariant: is the invariant testable, non-contradicting of existing
  invariants, correctly scoped to the tier?
- Epic: are member artifacts consistent with the epic goal, is the goal statement
  measurable?
- Issue (**AI-reviewed, never human-gated**): is the report reproducible, is the scope a
  single concern, is the root cause named (RCDD) rather than a bad-state restatement?
- PR/code (already in DR-033 §6): substance + security audit; unchanged.

Same shape as DR-033 §6 for each type: verdict + findings block, `request-changes`
auto-loops to the **author agent** (bounded — default 2 cycles, then escalate to the
human with `agent-escalated`), an `approve` verdict ends the cycle and a human-gated
Approvable proceeds to its human gate (a non-gated doc proceeds per Decision 5).

**Human-authored-approvable fallback.** Some Approvables have **no author agent** to loop
back to — e.g. a human-authored DR or a hand-edited PR (this very PR is one). When a
reviewer returns `request-changes` on a human-authored artifact, there is no agent to
re-dispatch; the auto-loop is skipped and the verdict **escalates straight to the human**
(`agent-escalated`) with the findings attached. The human, not an agent, resolves it.

**ADVISORY** — the independent reviewer advises; the human **always** holds the final
approval keystroke for human-gated types. The reviewer can never approve, merge, or modify
an artifact (never-wrong / HITL).

**2 — Signpost-contract generalisation.**

The next-human-task signpost (SPEC-010/SPEC-012, #182) currently presents a PR as a
human task only if `ai-review:pass` ∧ mergeable ∧ checks-green (DR-033 §6). The full
predicate generalises — greenlight alone is **not** sufficient:

> The signpost MUST present **any** human-gated Approvable as a human task only if the
> full predicate holds: the corresponding AI reviewer has **greenlit** it for its type
> **∧** all prior-stage gates are clear (for a PR: mergeable ∧ checks-green; for a
> doc Approvable: any upstream doc it depends on is itself greenlit, per §3) **∧** its
> human gate is still open (not already approved). Greenlit-for-type is necessary, never
> alone sufficient.

An un-reviewed or `ai-review:changes` Spec, DR, or constitution invariant MUST NOT
appear in the human queue — surfacing it re-exposes the human to the exact
rubber-stamp surface this stage removes. Non-gated docs (Issue; auto-accepted design.md /
tasks.md per Decision 5) never enter the human queue at all. This extends DR-033 §6's
signpost contract from PR-only to all human-gated Approvable types.

**3 — Ordering invariant: doc Approvables reviewed before the code that IMPLEMENTS them
— unrelated code is NOT blocked.**

When a PR contains a doc Approvable (e.g. a spec, DR, or constitution change) **and the
code that implements that doc**, the doc must be AI-reviewed and greenlit before the
**implementing** code's review runs. Code is judged against its spec; reviewing
implementing code against an un-reviewed approvable is inverted — the spec may still have
defects that would change the code.

**The gate is scoped to implementing code only.** In a mixed-content PR, code that does
**not** implement the reviewed doc (an unrelated fix, a sibling feature, a refactor with
its own spec) is **not blocked** by that doc's review — it proceeds through its own review
stage independently. The doc-before-code sequence applies pairwise between a doc and the
code that realises *that* doc, never PR-wide. Over-sequencing all code in a PR behind one
doc's review (the earlier reading) is rejected: it blocks unrelated work for no
correctness gain.

Precedence rule (per doc↔implementing-code pair): **approvable-doc AI review →
implementing-code AI review → human gate**. A PR is held from merge only for the
doc↔code pairs whose doc is still un-greenlit; unrelated code in the same PR advances on
its own gates.

**4 — Recording: extend the `ai-review:*` family per type.**

Reuse the `ai-review:*` status-check + label family (DR-033 §6, #342), scoped per
Approvable type:

| Status check | Label | Meaning |
|---|---|---|
| `ai-review/spec` | `ai-review:spec:pass` / `:changes` / `:pending` / `:escalated` | Spec substance verdict |
| `ai-review/plan` | `ai-review:plan:pass` / `:changes` / … | Plan substance verdict |
| `ai-review/design` | `ai-review:design:pass` / `:changes` / … | design.md substance verdict |
| `ai-review/tasks` | `ai-review:tasks:pass` / `:changes` / … | tasks.md substance verdict |
| `ai-review/dr` | `ai-review:dr:pass` / `:changes` / … | DR substance verdict |
| `ai-review/constitution` | `ai-review:constitution:pass` / `:changes` / … | Constitution invariant verdict |
| `ai-review/epic` | `ai-review:epic:pass` / `:changes` / … | Epic substance verdict |
| `ai-review/issue` | `ai-review:issue:pass` / `:changes` / … | Issue substance verdict (AI-reviewed, **never** human-gated) |
| `ai-review` | `ai-review:pass` / `:changes` / … | PR/code verdict (existing, DR-033 §6) |

Named `ai-review:*` not `ai-approved:*` — the AI *advises*; only the human *approves*
(never-wrong / HITL). The distinction **AI-greenlit** (reviewer verdict = pass) vs
**Approvable** (greenlit ∧ prior-stage gates clear ∧ human-gate open) holds for all
types, mirroring DR-033 §6's PR wording. AI-greenlit is the universal floor (every doc
type); only the critical subset then needs the human gate (Decision 5).

**5 — Auto-approve for low-criticality Approvables (dev opt-in) — DR-033 consequence-hybrid
applied to docs.**

DR-033 §3's consequence-hybrid moves the *human* gate by blast radius — low-blast changes
auto-merge once their signals are green; high-blast hold for a human read. The same
principle applies to doc Approvables: **gate by criticality.**

- **Low-criticality docs — `design.md`, `tasks.md`, Issue — MAY auto-accept on AI
  greenlight, no human.** When a dev opts in, an `ai-review:<type>:pass` verdict on these
  types advances the artifact with **no human keystroke**. Rationale: these are
  downstream/derivative (design.md and tasks.md derive from an already-human-gated spec)
  or non-binding (an Issue is an intake stub). The AI greenlight is a sufficient floor for
  them; a human gate adds friction without proportional safety. Issue is **never**
  human-gated regardless of the setting (it has no gate to move).
- **High-criticality docs — Spec, Plan, DR, constitution invariant, PR-to-main, Epic — keep
  the human gate.** AI greenlight is necessary but **not** sufficient; the human approval
  keystroke remains required. Plan joins this tier because it drives the implementation —
  a flawed plan has spec-level blast radius. These are the irreversible / wide-blast
  surfaces where the rubber-stamp failure (#344–349) did its damage.
- **Dev opt-in, off by default.** Auto-approve is a per-dev setting (Decision 6's coverage
  config), **opt-in** — the conservative default keeps every gated type human-gated. A dev
  who trusts the AI floor for derivative docs turns it on; the choice is theirs, surfaced
  and auditable (HITL / never-wrong: the dev consents once, explicitly, to the auto-accept
  policy — the tool never silently drops a gate).

This reconciles AI-review (all docs, always) with human-approval (the critical subset):
the *gate placement* is what criticality + the dev's setting decide, never whether the
doc was AI-reviewed.

**6 — Per-dev AI-review coverage = a sliding scale.**

Each dev chooses **how much** AI review to apply: a per-dev **coverage** setting — a
sliding scale over *which doc types are reviewed* and *at what depth*. It composes with
Decision 5: coverage decides what gets reviewed; auto-approve decides whether a reviewed
low-criticality doc still needs a human.

- **The scale** runs from minimal (review only the highest-criticality types — Spec, DR,
  constitution — at standard depth) to **max** (review every doc type, including Issue and
  the downstream design.md/tasks.md, at full opus depth), with intermediate steps between.
- **Default for the MinSpecPro dogfood projects = MAX coverage** — every doc type reviewed
  at full depth, **accepting the token-window waits** that entails. We eat the latency to
  dogfood the strongest configuration; the rubber-stamp session is fresh evidence that the
  floor must be high for this team.
- **Per-dev, surfaced and overridable** — a dev on a latency-sensitive machine can dial
  coverage down; the setting is theirs, auditable, never silently changed (HITL /
  never-wrong). It is config, not a hardcoded policy, mirroring DR-033's per-dev gate
  placement (#183).

## Rationale

The rubber-stamp session (#344–349) is a measured, concrete failure: 7 live defects in
merged work, including never-wrong violations, cleared by a fatigued human. The
independent reviewer pattern exists exactly to close this gap — its value is the
separation of "author" from "reviewer." Restricting it to PRs while AI-authored Specs
and DRs receive no independent read is a structural asymmetry. The cost of the
generalisation is proportional (one reviewer-agent per Approvable type, same script
shape), and the ADVISORY / HITL / never-auto-approve constraints are identical.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Review latency slows the human's queue | Reviewer runs async before the Approvable enters the signpost queue; it never blocks the author from continuing work on the next task. |
| Reviewer LLM disagrees with valid design choices (false `request-changes`) | Bounded auto-loop (2 cycles); on persistent disagreement the issue escalates to the human as `agent-escalated` — the human is still the arbiter. |
| Per-type reviewer prompts drift out of sync with artifact schemas | Reviewer prompts are part of the same MinSpec harness; a validator checks that a prompt exists for each registered Approvable type. |
| The ordering invariant (§3) blocks a fast PR merge when a doc is large | The doc reviewer runs in parallel with CI/lint; only the *implementing* code-review stage is sequenced after the doc greenlight, not the human merge, and not unrelated code in the same PR (§3). In practice the doc review completes before CI. |
| Auto-approve (Decision 5) silently drops a human gate the dev wanted | Auto-approve is **opt-in, off by default**, scoped to low-criticality types only; the dev consents once, explicitly, and the policy is auditable. High-criticality types can never auto-approve. |
| Max-coverage default (Decision 6) imposes token-window latency on the dogfood projects | Accepted trade-off, made explicit in the setting; coverage is per-dev and dial-able down for latency-sensitive machines. |
| Recording per-type labels creates label proliferation | Labels are prefixed `ai-review:<type>:*`; the existing `ai-review:*` set is unchanged. Proliferation is bounded by the fixed set of Approvable types. |

## Costly to Refactor

Low-medium. The reviewer-agent runner (`review-branch.sh` + `review-decide.sh`, DR-033
§6 / #342) is designed to be trigger-agnostic. Adding per-type reviewer prompts and a
dispatch path per Approvable type extends the existing seam rather than redesigning it.
The signpost-predicate generalisation (§2) is a filter change in SPEC-012's resolver;
the ordering invariant (§3) is a gate added to the dispatch chain. No new egress or
credential surface is introduced.

## Consequences

- The human approval gate is no longer the **first** independent read for any Approvable
  type — it becomes the **second** (after AI greenlight). Human attention concentrates on
  resolving genuine disagreements, not scanning for structural defects the AI already
  caught.
- AI-authored Specs and DRs gain the same adversarial second-read that PR diffs already
  have; the rubber-stamp failure class (#344–349) becomes structurally harder to repeat.
- The signpost queue becomes a **pre-filtered, AI-greenlit set** across all human-gated
  types — the human's next task is always something an independent agent already found
  sound. Non-gated docs (Issue; auto-accepted design.md/tasks.md) never enter the queue.
- The ordering invariant eliminates the category of "code reviewed against a spec that
  still had defects," which is the root of at least two of the #344–349 audit findings —
  while leaving unrelated code in a mixed PR free to advance (§3).
- **AI review and human approval are decoupled** (Decision 5/6): every doc is AI-reviewed
  (universal floor); only the critical subset is human-gated, and a dev may opt low-
  criticality docs into auto-accept. The human's attention is concentrated on the
  irreversible / wide-blast surfaces, not on derivative docs an AI floor already cleared.
- **DR-041's Approvable union is amended** (see DR-041 Amendment 2026-06-30) to add Plan,
  design.md, tasks.md, and constitution invariant — reconciling the membership delta the
  AI review of this PR flagged.

## Alternatives considered

- **Keep PR-only, add human-prompt nag for non-PR Approvables.** Rejected: a nag targets
  attention and motivation; the rubber-stamp session shows both can fail under fatigue.
  A structural independent review is the only reliable gate.
- **Block approval until a human reads for N seconds (time-gate).** Rejected: a time
  gate is gameable and does not catch substance defects; the independent reviewer catches
  what the human skims over.
- **Run the reviewer as the same agent that authored the artifact.** Rejected: the value
  of the pattern is the separation of author and reviewer (DR-033 §6, "independence is the
  value"). Self-review is worth ~0.
- **Make the reviewer gating (auto-reject, block merge).** Rejected: never-wrong / HITL
  — the human always holds the final gate. The reviewer is advisory.
