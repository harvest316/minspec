# DR-025: Canonical spec frontmatter schema + field-order gate

## Context

Spec frontmatter field ordering has drifted across **three generations**, visible
chronologically by SPEC id:

| Gen | Order | Specs |
|---|---|---|
| G1 epic-first | `epic, id, type, [tier], status, product` | scroogellm 100/101/102, minspec 001/002/003, SPEC-005, SPEC-006 |
| G2 id-first, epic-last | `id, type, [tier], status, product, epic` | SPEC-004, 007/008/009, 010, 011 |
| G3 id-first, tier-after-status, +refs | `id, type, status, tier, product, epic, depends_on/aspects/relates_to` | SPEC-012, 013, 014, 015 |

The instinct is to read this as harmless **backfill** — old specs written under an
older hand-schema. That is the *symptom*, not the cause. Three mechanisms keep the
drift alive:

1. **No code owns the order.** Specs are authored by the `specify` skill markdown
   template, which is regenerated per session by an LLM and drifts freely.
2. **The one authoritative emitter has rotted off the live schema.**
   `SpecFrontmatter` (`packages/minspec/src/lib/spec.ts:25`) and
   `serializeFrontmatter` (`:227`) emit `id, title, tier, status, created, epic,
   phases` — an order that matches **zero** specs on disk. It emits `title`,
   `created`, `phases` (no disk spec has them) and **omits** `type`, `product`,
   `depends_on`, `aspects`, `relates_to` (every disk spec carries `type`/`product`).
   It is bypassed entirely; `writeSpec` is not the path that authored these files.
3. **No gate.** `spec-validator.ts` has no field-set or field-order assertion.
   Nothing makes a drifted ordering un-committable.

Per DR-003: a pure data edit (reorder the old specs) to a state that "shouldn't be
possible" is the tell that a **gate is missing**. Backfill alone fixes the bad state
and leaves the mechanism intact — the next generation drifts again.

## Decision

**One canonical frontmatter schema owns field set and field order, expressed in
code, enforced by the validator, and emitted by the spec writer. The `specify`
skill template and any backfill conform to it — they do not define it.**

1. **Reconcile `SpecFrontmatter`** to the live schema: add `type`, `product`, and
   optional `depends_on`, `aspects`, `relates_to`; drop `title`/`created` if
   genuinely unused (confirm no reader depends on them first). Field declaration
   order in the interface = canonical emit order.
2. **`serializeFrontmatter` becomes the single emitter** and is actually used —
   reconciled to match (2) so round-tripping a spec is order-stable.
3. **Validator gate (soft).** `spec-validator` asserts the canonical field order
   and required-field presence. Soft-warn, advisory — only DR-012 approval blocks
   (mirrors the soft `epic:` rule, DR-013 §4). **Asymmetry check (DR-003 Phase 4):**
   assert each required field *exists*, not merely that present values are valid —
   the exact gap that stranded SPEC-004 (validator flagged dangling refs, never
   missing ones).
4. **Backfill G1/G2 specs** to the canonical order — *after* the gate exists, so the
   line holds once swept.

## Consequences

- **`SpecFrontmatter` / `serializeFrontmatter`** change — touches every reader of the
  interface. Type-level; tsc surfaces breakage.
- **Validator** gains a field-order + presence rule (rides on the SPEC-013 validator
  work, same as DR-023's soft rule).
- **DR-012 approval interaction (important).** Approval binds a sha256 of the whole
  spec file. Reordering frontmatter during backfill **voids every approval** on the
  swept specs (→ stale). The backfill must re-run "MinSpec: Approve Spec" on each, or
  be sequenced before those specs are approved. Backfill is therefore not free churn.
- **`specify` skill template** is edited once to emit canonical order; future specs
  start compliant.
- One source of truth replaces three implicit ones (skill, dead serializer, habit).

## Alternatives considered

- **Backfill only — reorder the old specs, no gate.** Rejected — data-only fix to a
  shouldn't-happen state; mechanism survives, drift returns (DR-003 corollary).
- **Hard-block on field-order mismatch.** Rejected — only DR-012 blocks; every other
  MinSpec check is advisory. A reorder nit must not wedge a commit.
- **Sort frontmatter alphabetically.** Rejected — order is semantic/ergonomic
  (`id, type, status` reads top-down); alpha scatters related fields.
- **Leave it — cosmetic.** Rejected — it is not only cosmetic: the dead serializer
  proves the schema and the code have silently forked, which is a latent correctness
  bug the moment anything calls `writeSpec`.

## Risks & Mitigations

| # | Risk | Likelihood · Impact | Mitigation |
|---|---|---|---|
| R1 | Dropping `title`/`created` breaks a hidden reader. | Med · Med | Grep all `.title`/`.created` accesses before removal; keep if any live reader; tsc backstop. |
| R2 | Backfill silently voids approvals, specs look approved but are stale. | Med · Med | Stale *is* the correct signal — DR-012 marks them; sweep + re-approve as one tracked step, not piecemeal. |
| R3 | Soft warning ignored → drift continues anyway. | Low · Med | Same residual as every soft rule (DR-013, DR-023); the gate at least makes drift *visible* at author time. |
