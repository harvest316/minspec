# DR-035 — Normalize checkbox state before hashing approved spec files

## Context

Approval system (DR-012) binds a spec to its sha256 hash at approval time. Any byte change → `stale`. Intended: editing spec content forces re-review.

During investigation of checkbox-ticking during implement phase, a structural mismatch surfaced:

- 13 of 14 minspec specs have **only** `requirements.md` (rank-0 canonical, always hashed).
- The checkboxes in `requirements.md` live under `## Acceptance Criteria` — e.g. `- [ ] (FR-1) resolver returns same output on repeated runs`.
- Ticking one to record "this criterion is now verified" = 1 byte change (`[ ]` → `[x]`) = approval revoked.
- `tasks.md` is body-only (no frontmatter) → skipped by `spec-gate.py` → safe for Claude to tick. But most specs have no `tasks.md`.

Two semantic types of checkbox exist in the spec kit:

| Type | File | Meaning | Should tick void approval? |
|---|---|---|---|
| Acceptance Criteria | `requirements.md` (hashed) | "criterion X must be true to ship" | **No** — ticking = verified, not spec change |
| Task progress | `tasks.md` (unhashed) | "step Y is done" | Already safe — not hashed |

## Decision

**Normalize checkbox state before hashing** — collapse `- [x]` and `- [X]` to `- [ ]` before computing sha256, in both gate implementations.

Normalization: single regex substitution on each line: `^(\s*-\s*)\[[xX]\]` → `$1[ ]`

Applied in:
1. `packages/minspec/src/lib/approval.ts` — new `normalizeForHash(content: string): string` helper, called by `hashContent()` before digest
2. `scripts/hooks/spec-gate.py` — same normalization in `sha(sp)` before hash computation

## What this protects

| Action | Hash | Approval |
|---|---|---|
| Tick checkbox (AC verified or task done) | **unchanged** | holds ✓ |
| Edit criterion TEXT | changes | revoked ✓ (contract protected) |
| Add or remove a checkbox LINE | changes | revoked ✓ (structure protected) |
| Edit any non-checkbox content | changes | revoked ✓ |

## Risk — gate divergence

The never-wrong invariant depends on TS and Python producing **identical** hashes for the same file. Introducing normalization in two languages risks drift.

Mitigation required alongside implementation:
- A shared fixture: one canonical string in, one expected hash out, reproduced by both implementations identically.
- CI step or cross-impl test that verifies both agree on the fixture hash.
- Normalization must be a pure, deterministic, stateless step applied before `crypto.createHash` / `hashlib.sha256`.

## What this does NOT change

- `hashContent()` signature remains `Buffer | string` — normalization converts Buffer to utf-8 string first.
- Approval state machine (`approved` / `stale` / `unapproved`) unchanged.
- `spec-gate.py` skip logic (files lacking `---` frontmatter) unchanged — `tasks.md` remains unscanned.
- Existing stored hashes become stale on deploy (normalization changes the hash function output). All specs need one re-approval after this ships. Acceptable cost.

## Alternatives rejected

**Option B — ban checkboxes from hashed files, force to `tasks.md`:** Acceptance Criteria ARE spec contract content. Exiling them to an unhashed file drops them from approval coverage — a weakened criterion would silently pass the gate. Category error.

**Option C — separate "progress" file alongside `requirements.md`:** Adds new kit complexity for a problem Option A solves with one regex. Over-engineered.
